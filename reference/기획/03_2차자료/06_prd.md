## 1. 개요

| 항목 | 내용 |
|---|---|
| 프로젝트명 | MOA |
| 한 문장 정의 | 충동 소비를 반복하지만 무엇을 줄여야 할지 모르는 20~30대 직장인을 위해, AI가 이번 달 줄여볼 소비 카테고리를 제안하고 사용자가 고른 절약 강도만큼의 지킬 돈을 실제 소비로부터 지켜내도록 돕는 서비스 |
| 해결하려는 문제 | 사용자는 소비 후 후회하지만 무엇을 줄여야 할지 판단하지 못한다. 기존 금융 앱은 이미 발생한 소비만 분석하며 행동 변화까지 이어지지 않는다. |
| 타깃 사용자 | 20~30대 직장인. 월 2회 이상 충동 소비 후 후회한 경험이 있으며 금융 앱을 사용하지만 행동 변화로 이어지지 않는 사용자 |

### 1.1 프로젝트 성격과 구현 경계

MOA는 **포트폴리오용 금융 서비스 프로토타입**이다.

- 실제 개인신용정보와 실제 자금을 취급하지 않는다.
- 마이데이터 표준을 모사한 더미 서버로 소비 분석·거래 갱신 경험을 재현한다.
- 카드 승인내역과 계좌 거래내역을 함께 수집한다.
- 1차 구현은 `소비 분석 → 카테고리·강도 선택 → 지킴 추적 → 월말 확정`까지다.
- 2차 구현은 별도 가상 금융 서버로 지킴 계좌 개설·가상 이체·누적 잔액을 재현한다.
- 실제 마이데이터 사업자 연동과 실제 계좌 개설·이체는 범위 밖이다.

세 명의 팀원은 소비 분석(①), 지킴·성장(②), 취향·추천(③)을 각각 Agent로 구현한다. 금액 계산과 상태 판정은 규칙 엔진이 수행하며 LLM이 계산 결과를 결정하거나 변경하지 않는다.

## 2. 제품 목표

MOA는 모든 소비를 막는 서비스가 아니라, 사용자가 직접 선택한 1~2개 카테고리에서 스스로 정한 만큼의 돈을 이번 달 동안 지켜가도록 돕는 서비스이다.

사용자가

- 이번 달 줄여볼 소비 카테고리를 스스로 선택하고
- 각 카테고리를 얼마나 줄일지 직접 정하고
- 소비가 현재 지키는 금액에 어떤 영향을 주는지 이해하고
- 월말까지, 그리고 다음 달에도 절약 행동을 이어가도록 돕는다.

### 제품 원칙

제품 원칙은 `05_솔루션컨셉.md`의 "반드시 지킬 제품 원칙"을 정본으로 한다.

## 3. 솔루션 가설

AI가 과거 소비 데이터를 분석해 이번 달 줄여볼 소비 카테고리와 근거를 먼저 제안하면 사용자는 무엇을 줄여야 할지 쉽게 결정할 수 있다.

한도를 넘긴 만큼 지킨 돈이 깎이는 모습을 보여주면 사용자는 월말 이전에도 자신의 소비 상태를 인지하고 행동을 조정할 가능성이 높아진다.

## 4. 핵심 기능

1. **소비 분석 및 절약 후보 추천** — 최근 3개월 거래 분석, 반복 소비 탐색, 기준 소비 산출, 이번 달 줄여볼 카테고리와 근거·강도 추천
2. **절약 강도와 지킬 돈 설정** — 사용자가 카테고리 선택 → 카테고리별 절약 강도 선택 → 지킬 돈 확정. 사용 한도는 내부 계산
3. **지킴 현황 관리** — 거래 수신 시 카테고리 판단, 누적 소비 반영, 한도 초과분만큼 지킨 돈 차감, 초과 시점부터 알림
4. **캐릭터 여정과 성장** — 월간 챕터, 월말 확정 지킨 돈의 누적 성장, 레벨·외형·성장 기록
5. **주간·월말 리포트** — 주간 소비 흐름 요약, 월말 지킨 돈 확정과 캐릭터 성장 연출

## 5. 지킬 돈 계산 모델

### 5.1 정의

> **사용 한도 = 기준 소비 × (1 − 절약 강도)** — 이번 달 써도 되는 돈
> **지킬 돈 = 기준 소비 × 절약 강도** — 이번 달 지키기로 한 돈

절약 강도 100%를 고르면 사용 한도가 0원이 되어 카테고리 전체 절감과 완전히 같아진다. **전체 절감은 부분 절감의 특수한 경우다.**

### 5.2 용어

| 용어 | 정의 |
|---|---|
| 기준 소비 | 해당 카테고리의 평소 소비 수준. 최근 3개월 월평균 **(잠정)** |
| 선택 카테고리 | 사용자가 이번 달 줄이기로 확정한 카테고리 |
| 절약 강도 | 기준 소비 중 지킬 비율. 사용자가 카테고리별로 선택 |
| 사용 한도 | `기준 소비 × (1 − 절약 강도)`. 차감 시작점을 정하는 내부 계산값. 사용자 화면에 노출하지 않음 |
| 지킬 돈 | `기준 소비 × 절약 강도`. 월초에 확정되는 이번 달 최대 지킴 금액 |
| 누적 소비 | 이번 달 선택 카테고리에서 발생한 실제 지출 합계 |
| 남은 한도 | `max(사용 한도 − 누적 소비, 0)`. 내부 상태값. 사용자 화면에 노출하지 않음 |
| 지킨 돈 | `max(지킬 돈 − max(누적 소비 − 사용 한도, 0), 0)`. Home의 주 지표 |
| 초과 지출 | 누적 소비가 기준 소비를 넘은 금액 |

> 기준 소비 산정은 잠정 확정이다. 이전 달 실제 소비 기준안도 검토했으나, 이전 달이 이상치일 때 목표가 흔들리고 "평소와 크게 다른 경우"의 임계값을 새로 정의해야 하므로 MVP에서는 3개월 평균 단일 기준을 사용한다.

### 5.3 카테고리와 강도 선택

- AI는 `이번 달 줄여볼 카테고리`와 추천 근거, 권장 강도를 제안한다.
- 사용자는 추천을 수락·거절하고 다른 카테고리를 직접 추가할 수 있다.
- 강도는 카테고리별로 따로 정한다. AI 권장값은 초기값일 뿐 사용자가 바꾼다.
- 1차 구현에서는 무리한 목표를 줄이기 위해 1~2개 선택을 권장한다. 시스템상 최대 개수는 프로토타입 테스트 후 확정한다.
- 사용자가 선택한 카테고리만 계산 대상이 된다.
- 선택 카테고리 안에서 일부 거래만 면제하지 않는다. 카테고리가 틀린 경우에만 오분류 수정으로 복구한다.
- 강도는 월초에 확정하며 월중에는 변경하지 않는다. 월중 변경은 Later다.

### 5.4 계산 규칙

| 항목 | 규칙 |
|---|---|
| 기준 소비 | 최근 3개월 카테고리별 월평균 (잠정) |
| 사용 한도 | `기준 소비 × (1 − 절약 강도)` |
| 지킬 돈 | `기준 소비 × 절약 강도` |
| 지킨 돈 | `max(지킬 돈 − max(누적 소비 − 사용 한도, 0), 0)` |
| 남은 한도 | `max(사용 한도 − 누적 소비, 0)` |
| 차감 단위 | 해당 카테고리 거래 **전액**을 누적 소비에 가산 |
| 차감 시점 | 누적 소비가 사용 한도를 넘어선 이후의 금액만 지킨 돈에서 차감 |
| 지킨 돈 하한 | 0원에서 멈춘다 |
| 초과 지출 | 누적 소비가 기준 소비를 넘으면 초과분을 별도 누적·표시 |
| 미선택 카테고리 | 계산 대상 아님. 금액 변화만 로그로 기록 |
| 취소·환불 | 해당 카테고리 누적 소비에서 복구 |
| 합산 단위 | 지킨 돈은 전체 합계 기준으로 계산한다. 카테고리별 한도 상태는 내부 기록하며 합계 누적 소비가 합계 사용 한도를 넘을 때 차감·알림한다 |

**차감 규칙은 하나뿐이다. 한도를 넘긴 만큼만 깎인다.**

기준 소비·사용 한도·남은 한도는 규칙 엔진이 계산하는 내부 값이다. 사용자에게는 현재 지키는 금액만 노출한다. 사용 한도 이내 거래는 내부 원장에만 반영하고 알림을 보내지 않는다.

식을 구간으로 나누면 v6 모델과의 관계가 드러난다.

```text
누적 소비 ≤ 사용 한도   →   지킨 돈 = 지킬 돈            변하지 않는다
누적 소비 > 사용 한도   →   지킨 돈 = max(기준 소비 − 누적 소비, 0)
```

두 구간은 사용 한도 지점에서 끊김 없이 이어진다.

### 5.5 달성 상태

| 상태 | 조건 | 의미 |
|---|---|---|
| 완전 달성 | 누적 소비 ≤ 사용 한도 | 한도를 지켜 지킬 돈을 전부 지켰다 |
| 부분 달성 | 사용 한도 < 누적 소비 < 기준 소비 | 한도는 넘겼지만 평소보다 덜 써서 일부를 지켰다 |
| 미달성 | 누적 소비 ≥ 기준 소비 | 평소 수준 이상 사용해 지킨 돈이 0원이다 |

월중에는 결과가 확정되지 않으므로 `지키는 중` 또는 `지킬 돈 소진`으로만 표시한다. 완전 달성·부분 달성·미달성은 월말에 판정한다.

### 5.6 예시

```text
기준 소비 (최근 3개월 월평균)
택시 80,000원 · 배달 150,000원

이번 달 줄일 카테고리와 강도 선택
택시 80,000원 · 강도 50% → 사용 한도 40,000원 · 지킬 돈 40,000원
배달 150,000원 · 강도 50% → 사용 한도 75,000원 · 지킬 돈 75,000원

이번 달 사용 한도 115,000원 · 지킬 돈 115,000원

--- 8일차, 택시 18,500원 결제 ---
누적 소비 18,500원 (한도 이내)
지킨 돈 115,000원 · 변화 없음
사용자 알림 없음

--- 20일차, 누적 소비 130,000원 ---
사용 한도를 15,000원 초과
지킨 돈 115,000 − 15,000 = 100,000원
초과액과 차감 결과를 사용자에게 알림

--- 월말 ---
택시 42,000원 · 배달 98,000원 · 누적 소비 140,000원
초과 25,000원
지킨 돈 115,000 − 25,000 = 90,000원 → 부분 달성
```

### 5.7 판정 기준 (미정 · 별도 담당)

- 기준 소비 산출식 (단순 평균 / 중앙값 / 이상치 제외)
- 절약 강도 단계와 비율 (20 / 50 / 100% 잠정)
- 3개월 미만 데이터일 때의 대체 산정
- 선택 카테고리 권장·허용 개수
- 월 경계 처리 (결제일 / 승인일)
- 할부·정기결제의 월 귀속
- 부분 취소·부분 환불의 복구 계산
- 월말 마감 이후 지연 취소·정정 처리

## 6. 거래 판정 및 사용자 확정 플로우

### 6.1 기본 정책

> 모든 신규 거래는 AI가 자동으로 판정하여 누적 소비에 반영한다.
> 사용 한도 이내에서는 사용자에게 알리지 않고, 한도를 넘긴 순간부터 초과액과 차감 결과를 알린다.
> 사용자가 오분류 수정을 수행한 경우에만 금액이 복구된다.

```text
신규 거래 수신 → AI 1차 판단 → 선택 카테고리면 누적 소비 가산
                                  ├─ 합계 사용 한도 이내 → 침묵
                                  └─ 합계 사용 한도 초과 → 초과액 차감 + 알림
                              → 미선택 카테고리면 침묵
       ↓
사용자가 맥락 태그 선택 (선택) 또는 오분류 수정
```

분류 신뢰도가 낮은 거래(PG사 결제, 미분류 가맹점, 사업자등록번호 누락)만 카테고리 선택을 직접 요청한다.

### 6.2 알림 구성

차감 알림은 합계 사용 한도를 넘겨 지킨 돈이 실제로 줄어든 순간부터 보낸다. **깎인 금액과 한도를 넘긴 금액은 언제나 같다.**

```text
🚕 택시 18,500원
정해둔 범위를 12,000원 넘었어요
지키는 금액 112,000원 → 100,000원

[🎯 계획했던 소비]  [😅 줄이려던 소비]  [🚨 불가피한 소비]
[분류가 틀렸어요]
```

한도 이내 거래에는 거래 알림·소진 임박 알림·사전 경고를 보내지 않는다. 거래 자체는 원장과 거래 내역에 기록한다.

### 6.3 금액에 영향을 주는 동작

| 구분 | 사용자의 의미 | 누적 소비 | 현재 지키는 금액 |
|---|---|---|---|
| 기본 반영 | — | 선택 카테고리면 가산 | 합계 한도 초과분만큼 감소 |
| 오분류 수정 | "카테고리 자체가 틀렸다" | 복구 또는 재가산 | 원장 재계산 |
| 거래 취소·환불 | — | 복구 | 원장 재계산 |

금액을 되돌리는 경로는 오분류 수정과 거래 취소·환불뿐이다.

### 6.4 소비 맥락 태그

| 태그 | 의미 |
|---|---|
| 🎯 계획했던 소비 | 줄이기로 했지만 미리 예정돼 있던 소비 |
| 😅 줄이려던 소비 | 줄이려 했는데 못 줄인 소비 |
| 🚨 불가피한 소비 | 통제 불가능했던 소비 |

**태그는 금액에 영향을 주지 않는다.**

- 선택 카테고리의 모든 거래가 누적 소비에 반영된다. 계획했거나 불가피했다는 이유로 면제하지 않는다.
- 금액과 무관하므로 유리한 태그를 고를 유인이 없고, 데이터가 정직해진다.
- 태그 입력은 선택이다.

### 6.5 태그 데이터 활용

- 다음 달 카테고리 추천과 권장 강도의 근거로 사용한다.
- 불가피한 소비가 반복되면 해당 카테고리를 다음 달 후보에서 제외하거나 강도를 낮춰 제안한다.

```text
지난달 택시 불가피 3건 · 32,000원
이번 달에는 택시 강도를 50%에서 20%로 낮춰볼까요?
```

## 7. 캐릭터 여정과 성장

캐릭터·성장·챕터의 **상태와 계산(게이미피케이션 진행)은 ② 지킴·성장 Agent가 소유**한다. 그 상태를 **리포트로 서술·연출하는 것은 ③ 취향·추천 Agent**가 담당한다(§11). 상세는 `10_여정과_성장.md`를 따른다.

### 7.1 지킴이 캐릭터

지킴이는 사용자의 돈을 함께 지키며 여정을 이어가는 캐릭터이자, 사용자에게 말을 거는 **화자**다. 실시간 거래 알림 문구는 ②가, 주간·월말 리포트의 서술은 ③이 같은 캐릭터의 목소리로 만든다.

| 항목 | 내용 |
|---|---|
| 역할 | 알림·회고 문장의 화자 · 유일한 감정 이입 대상 |
| 등장 지점 | 캐릭터 생성 · Home 여정 영역 · 거래 알림 · 월말 리포트 · 성장 기록 |
| 이름 | 사용자 입력, 최대 10자 |
| 톤 | 비난하지 않는다. 사실을 말하고 다음 행동을 제안한다 |
| 반응 | 월중에는 지키는 상태를 표현하고, 월말 성장 확정 때 기뻐한다. 지출에 부정 반응 금지 |
| 하지 않는 것 | 금액 계산·상태 판정·카테고리나 강도 변경 · 소비 목표나 보상 제안 |

문장은 LLM이 생성하고 그 재료가 되는 숫자는 규칙 엔진이 계산한다 (`07` R2).

```text
"정해둔 범위를 15,000원 넘었어요. 지금 100,000원 지키는 중이에요."
"금액이 줄었지만 아직 100,000원이 남아 있어요. 남은 열흘이 더 중요해요."
```

### 7.2 성장 원장

월중 지킨 돈은 거래 취소·환불·오분류 수정으로 바뀔 수 있으므로 성장 원장에 즉시 반영하지 않는다. ②가 월말 값을 확정하고 성장 원장에 한 번만 누적한다.

```text
monthly_growth_amount = finalized_kept_amount
cumulative_kept_amount = previous_cumulative_kept_amount + monthly_growth_amount
level = threshold(cumulative_kept_amount)
```

| 규칙 | 내용 |
|---|---|
| 반영 시점 | 월말 마감 후 1회 |
| 성장 연료 | 월말 확정 지킨 돈 |
| 감소 | 없음 |
| 월 전환 리셋 | 없음 |
| 0원인 달 | 성장 정지, 레벨·외형 유지 |
| 중복 반영 방지 | `user_id + period` 멱등키 |

레벨 구간과 외형 단계 수는 미정이다. 레벨·성장은 ②의 규칙 엔진이 계산하며, 그 결과를 설명하는 문장은 ③이 리포트에서 만든다. LLM은 레벨을 계산하지 않는다.

### 7.3 월간 챕터

매월은 종착점 없는 여정의 새 구간이다.

```text
월초   새 챕터 시작 · 이번 달 카테고리와 강도 확정
월중   캐릭터가 현재 지킬 돈을 지키는 상태
월말   최종 지킨 돈 확정 · 누적 성장 · 챕터 기록 저장
다음 달 새 챕터 시작 · 누적 레벨·외형 유지
```

챕터는 `완주/실패`로 사용자에게 낙인찍지 않는다. 내부 분석용 `full/partial/missed`는 유지하지만 리포트 전면에는 `이번 달 {금액}원을 지켰어요`를 표시한다.

### 7.4 표시 언어

| 시점 | 쓰지 않는다 | 쓴다 |
|---|---|---|
| 월중 | 이번 달 115,000원 모았어요 | 현재 115,000원 지키는 중이에요 |
| 여정 | 오사카까지 27% · 584,000원 남음 | 7월의 길을 지키는 중이에요 |
| 월말 | 목표 실패 · 목적지에 못 갔어요 | 이번 달 90,000원을 지켰어요 |
| 성장 | 15,000원 잃어서 레벨이 내려갔어요 | 누적 420,000원과 함께 Lv.3이 되었어요 |

2차 가상 지킴 계좌가 연결되면 이체가 완료된 금액만 `지킴 계좌에 담았다`고 표현한다.

## 8. 소비 목표·보상 비개입 · 금융상품 정보성 추천

### 8.1 소비 목표·보상 비개입 (유지)

MOA는 절약을 새로운 소비 동기로 바꾸지 않는다.

- 여행·물건·구매 목표를 추천하거나 입력받지 않는다.
- 목적지, 목표 금액, 도착률, 남은 거리, 도착 예상일을 표시하지 않는다.
- 지킨 돈으로 살 소비형 보상·상점 아이템을 추천하지 않는다.
- 캐릭터 성장을 결제·과금으로 구매할 수 없다.
- 지킨 돈의 실제 사용처는 사용자의 자유이며 서비스가 묻거나 평가하지 않는다.
- 분석 Agent가 찾은 가치 소비 신호는 `줄이지 않을 소비`를 존중하는 데만 사용한다.

### 8.2 금융상품 정보성 추천 (③ 담당)

소비가 아니라 **아낀 돈의 보관처(저축)** 를 다루는 정보성 추천은 범위에 포함한다. 새 소비 동기를 심지 않으므로 8.1의 비개입 원칙과 충돌하지 않는다.

- ③은 실제 금융상품 정보(실 금리 등)를 사용자의 소비 패턴·취향에 맞춰 **개인화된 정보성 비교**로 제공한다.
- **판매·중개·가입 대행을 하지 않는다.** 무판매목적·무제휴이며, 가입은 각 사에서 하도록 안내한다(`가입은 각 사로`).
- 목표 금액이나 달성 페이스(`이 상품을 채우려면 매달 얼마`)로 제시하지 않는다. 그 자체가 8.1에서 제거한 목표·페이스의 재도입이므로 금지한다.
- 노출 위치는 월말·누적 맥락과 2차 가상 지킴 계좌다(§9.9 · §12).
- 상품군 범위, 실제 정보의 출처, 취향 분석 방식, 매칭·정렬 기준은 **미정**이며 구현 단계에서 정한다.

## 9. 마이데이터 연결 절차

최초 1회만 수행한다. 월간 루프와 분리된 층이다.

### 9.1 화면 구성

| # | 화면 | 실제 구현 |
|---|---|---|
| 1 | 마이데이터 안내 | 화면만 |
| 2 | 전송요구서 · 동의 | 동의 항목 처리 |
| 3 | 기관 선택 (카드사 · 은행) | 선택 결과를 조회에 반영 |
| 4 | 통합인증 | 화면만. 버튼 통과 |
| 5 | 조회 진행 | 실제 API 호출 |

### 9.2 전송요구서

| 항목 | 값 |
|---|---|
| 목적 | 소비 분석 및 절약 목표 제안 |
| 항목 | 카드 국내 승인내역 · 수시입출금 계좌 거래내역 |
| 기간 | 최근 3개월 |
| 보유기간 | 서비스 이용 종료 시까지 |
| 종료시점 | 전송요구일로부터 1년 |
| 전송주기 | 주 1회 |

### 9.3 가맹점명 동의는 필수 처리한다

마이데이터 표준상 `merchant_name`·`merchant_regno`는 **선택 동의 항목**이다. 전송요구 시 동의하지 않으면 응답에 아예 포함되지 않는다.

**MOA는 이 정보 없이는 카테고리 분류가 불가능하다.** 따라서 동의하지 않으면 진행을 차단하고 사유를 안내한다.

```text
가맹점 정보 없이는 소비 분석을 만들 수 없어요. 동의가 필요해요.
```

더미 서버도 이 플래그를 실제로 반영한다. false면 가맹점명을 제외하고 응답한다.

### 9.4 사용 API

MOA는 **카드와 은행 업권**을 요청한다. 보험·증권은 수집하지 않는다.

| 업권 | 용도 | 엔드포인트 |
|---|---|---|
| 공통 | 접근토큰 | `POST /oauth/2.0/token` |
| 공통 | 전송요구 내역 | `GET /v2/consents` |
| 카드 | 카드 목록 | `GET /v2/card/cards` |
| 카드 | 국내 승인내역 | `GET /v2/card/cards/{card_id}/approval-domestic` |
| 은행 | 계좌 목록 | `GET /v2/bank/accounts` |
| 은행 | 수시입출금 거래내역 | `GET /v2/bank/accounts/{account_id}/deposit/transactions` |

카드 승인내역 주요 응답 필드

```
approved_num, approved_dtime, status, pay_type, trans_dtime,
merchant_name*, merchant_regno*, approved_amt, modified_amt, total_install_cnt

* 가맹점명 동의 시에만 회신
```

계좌 거래내역 주요 응답 필드

```
trans_dtime, trans_no, trans_type, trans_class,
print_content, trans_amt, balance_amt
```

- `status` — 01 승인 / 02 승인취소 / 03 정정 / 04 무승인매입. 취소·정정 처리가 §5.4의 "취소·환불 복구"에 대응한다.
- 계좌 거래내역에는 **가맹점명이 없다.** 적요(`print_content`)만 제공된다. 분류 처리는 §10.4를 따른다.
- 페이지네이션(`next_page`, `limit` 최대 500)을 실제로 구현한다. 3개월치 조회에서 실제로 걸린다.

### 9.5 조회 정책

```text
최초   from_date = 3개월 전, to_date = 오늘 → DB 적재
이후   from_date = 마지막 조회일, to_date = 오늘 (증분)
       Home 진입 시 또는 당겨서 새로고침
```

분석·차감·리포트는 전부 DB에서 수행한다. 마이데이터 서버는 갱신 시에만 호출한다.

부분 성공을 전제로 설계한다. 기관별 진행 상태를 표시하고, 성공한 기관은 먼저 반영하며 실패한 기관만 재시도한다. 카드사와 은행이 섞이므로 업권별 진행 상태를 구분해 표시한다.

### 9.6 실연동 시 제약 (중요)

마이데이터 2.0의 정기적 전송 주기는 기본 주 1회이며, 이용자가 1주 간격으로 최대 한 달까지 선택한다. 비정기적 전송은 이용자 요청 시 최대 1개월 내 정보를 받는다. 표준 API는 정보제공자가 push하는 방식이 아니라 마이데이터사업자가 호출해 가져오는 pull 방식이다.

> **결제 직후 푸시 알림은 마이데이터만으로는 불가능하다.**

- MVP는 더미 데이터를 사용하므로 실시간 시뮬레이션으로 동작한다. 푸시 알림도 Later이므로 MVP 범위에는 영향이 없다.
- 실연동 단계에서는 §9.5의 "앱 진입 시 증분 조회" 모델이 그대로 성립한다.
- 문서·포트폴리오에서 "실시간 결제 알림"으로 표현하지 않는다.

### 9.7 제외

가상(더미) CI를 생성해 인증 화면(§9.1 4단계)을 시뮬레이션하고, 그때 쓰는 더미 이름·전화번호는 CI 반환 즉시 파기한다. **실제** CI 생성·검증, 실제 본인인증, 인증서 검증, 종합포털 연동, 기관코드 발급, 그리고 **실제 개인정보 수집·저장**은 하지 않는다. 전부 화면 흉내까지만 한다.

### 9.8 더미 서버

- 표준 스펙 엔드포인트와 별도로 `/demo/*` 네임스페이스에 시연용 엔드포인트를 둔다: 시계 이동, 거래 주입, 초기화
- 실연동 전환 시 `/demo/*`만 제거한다
- 거래 생성은 **패턴 규칙 + 표면 랜덤**으로 한다. 균등 랜덤이면 월평균이 흔들려 기준 소비가 무의미해진다
- 시드를 고정해 시연 재현성을 확보한다
- 계좌 거래는 카드 결제 대금 출금·급여 입금·구독 자동이체를 포함해 생성한다

### 9.9 2차 구현 — 가상 지킴 계좌

가상 계좌 개설·잔액·이체는 마이데이터 조회 API와 섞지 않고 별도 가상 금융 서버가 담당한다.

```text
더미 마이데이터 서버 → 카드 거래 · 계좌 거래 제공
MOA Agent 시스템      → 지킬 돈 추적·월말 지킨 돈 확정
가상 금융 서버        → 출금 계좌·지킴 계좌·가상 이체·누적 잔액
```

2차 구현의 최소 API는 다음과 같다.

| 용도 | 엔드포인트 |
|---|---|
| 가상 계좌 개설 | `POST /demo/banking/accounts` |
| 계좌·잔액 조회 | `GET /demo/banking/accounts/{account_id}` |
| 거래내역 조회 | `GET /demo/banking/accounts/{account_id}/transactions` |
| 가상 이체 | `POST /demo/banking/transfers` |

- 월말 `지킨 돈`이 0원보다 크면 완전 달성 여부와 관계없이 지킴 계좌로 옮긴다.
- 이체 요청은 월·사용자별 멱등키를 사용해 중복 이체를 막는다.
- 가상 지킴 계좌는 누적 잔액만 보여주며 목표 진척률이나 사용처를 연결하지 않는다.
- 1차 구현에서는 이체를 실행하지 않고 월말 지킨 돈 확정까지만 수행한다.

## 10. 카테고리 분류 파이프라인

### 10.1 MCC를 사용하지 않는다

마이데이터 카드 승인내역 응답에는 **MCC 필드가 없다.** 제공되는 것은 가맹점명과 사업자등록번호뿐이다. v4까지의 "MCC 1차 분류" 전제를 폐기한다.

### 10.2 3단 구조

| 단계 | 처리 | 대상 |
|---|---|---|
| 1차 | 규칙 — 가맹점명 키워드 사전 | 사전에 등재된 가맹점 |
| 2차 | ML — 가맹점명 텍스트 분류 모델 | 1차 미분류 거래 |
| 3차 | LLM 분류 + 불확실 표시 | 2차 신뢰도 임계값 미만 |

- 1차에서 확정되면 하위 단계를 호출하지 않는다.
- 3차 결과는 불확실 표시와 함께 사용자 확인을 요청한다.
- 사용자의 오분류 수정 결과는 키워드 사전 또는 학습 데이터에 반영한다.
- 사업자등록번호 기반 업종 역조회는 별도 데이터 소스가 필요하므로 Later로 둔다.

> MCC가 없어지면서 ML의 역할이 커졌다. 가맹점명 텍스트 분류는 규칙만으로 처리할 수 없는 영역이며, 이것이 이 프로젝트에서 ML을 사용하는 근거다.

### 10.3 학습 데이터 (확인 중)

자체 더미 데이터만으로 학습하면 모델이 더미 생성 규칙을 재학습하는 순환 구조가 된다. 외부 데이터(공개 상권 데이터, 가맹점명 데이터셋) 확보 가능 여부를 확인한 뒤 확정한다.

더미 가맹점명에는 세 유형을 섞는다. 전부 명확한 이름이면 1차 규칙에서 100% 분류돼 ML·LLM 단계를 시연할 수 없다.

```
명확     스타벅스 강남2호점 · 카카오T
PG사     (주)우아한형제들 · 네이버페이
애매     (주)에이치디케이 · 제일상사
```

### 10.4 계좌 거래는 누적 소비에 반영하지 않는다

계좌 거래내역은 카드와 **다른 목적**으로 수집한다.

> **누적 소비는 카드 승인내역만으로 계산한다.**

카드 결제 대금은 나중에 계좌에서 빠져나간다. 계좌 출금을 소비로 세면 같은 지출을 두 번 세게 된다. 이 이중 계상을 막기 위해 계좌 거래는 금액 계산에서 제외한다.

계좌 거래의 용도는 다음과 같다.

| 용도 | 내용 |
|---|---|
| 고정지출 탐지 | 월세·통신비·보험료 등 조정 불가능한 지출을 식별해 절약 후보에서 제외 |
| 구독 탐지 | 같은 금액이 같은 날 반복되는 자동이체 |
| 가치 소비 보호 신호 | 반복되는 취미·가족·건강 소비를 절약 후보에서 제외하거나 추천 우선순위를 낮춤 |
| 절약 여력 산출 | 급여 입금과 고정지출의 차이 |

계좌 거래에는 가맹점명이 없고 적요(`print_content`)만 있다. 적요는 형식이 자유롭고 축약이 많아 §10.2의 가맹점명 분류 모델을 그대로 쓸 수 없다. **계좌 거래는 별도의 규칙 기반 분류를 사용한다.**

```text
적요 예시
"우리카드결제"     → 카드 대금. 무시
"(주)케이티"       → 고정지출 · 통신
"넷플릭스"         → 구독
"급여"             → 수입
```

적요 분류 규칙 사전과 미분류 적요 처리 정책은 미정이다.

## 11. Agent 정의

MOA는 3개 Agent로 구성한다. 연결부 상세는 `07_에이전트_연결부.md`를 정본으로 한다. 아래는 각 Agent의 책임 경계이며, 구체 알고리즘·데이터 소스·필드 설계는 구현 단계에서 정한다.

### 11.1 소비 분석 Agent (상현)

거래 원자료를 소비의 구조로 분석한다. 취향 분석은 하지 않는다(③ 담당).

| 항목 | 내용 |
|---|---|
| 트리거 | 온보딩 시 · 신규 거래 발생 시 · 월말 |
| 입력 | 마이데이터 카드·계좌 거래 리스트 · (G) 전월 지킴 실적 |
| 출력 | 카테고리별 월평균 · 반복 결제 목록 · 요일·시간대 패턴 · 줄일 수 있는 소비 후보와 예상 절약액 · 권장 강도 · 추천 근거 · 고정지출·구독 · 가치 소비 보호 신호 · 소비 성향 프로필(충동 구매 지수 포함) |
| LLM | 애매한 가맹점 재분류 · 추천 근거 문장 · 성향 프로필 문장 |
| 규칙 | 집계 · 반복 결제 탐지 · 가맹점명 키워드 1차 분류 · 적요 규칙 분류 · 기준 소비 산출 |

### 11.2 지킴·성장 Agent (윤정)

이번 달 돈을 지키는 금액 엔진이자, 캐릭터·성장 등 게이미피케이션의 진행(상태·계산)을 담당한다.

| 항목 | 내용 |
|---|---|
| 트리거 | 신규 거래 수신 시 · 월말 · 새 달 시작 · 캐릭터 생성 |
| 입력 | 신규 거래 · (A) 분석 결과 · (F) 재분류 · 사용자 선택 카테고리·강도·캐릭터 이름 |
| 출력 | 사용 한도 · 지킬 돈 · 누적 소비 · 남은 한도 · 지킨 돈 · 월중 상태 · 실시간 차감 알림 · 캐릭터 이름·레벨·외형·월간 챕터·누적 지킨 돈(성장 상태) · (C) 지킴·성장 상태 · (G) 월말 실적 |
| LLM | 실시간 거래 알림 문구(지킴이 화자, 비난 없는 톤). 금액·레벨 계산과 판정은 하지 않는다 |
| 규칙 | 선택 카테고리 매칭 · 누적 소비 가산 · 한도 초과분 차감 · 월중 상태 판정 · 월말 지킨 돈 확정 · 월말 확정 지킨 돈의 성장 누적(멱등) · 성장 구간 판정 · 챕터 상태 전이 |

성장은 월말 확정 지킨 돈만 누적하며(§7.2) 월중 변동은 성장에 반영하지 않는다. 성장·누적 값은 감소하거나 리셋되지 않는다. 리포트와 성장 서술은 ③이 담당한다.

### 11.3 취향·추천 Agent (가원)

마이데이터로 소비자의 취향을 분석하고, ①의 소비 분석과 ②의 지킴·성장 성과를 받아 리포트와 맞춤 금융상품 정보성 추천을 만든다.

| 항목 | 내용 |
|---|---|
| 트리거 | 주간 · 월말 · 사용자가 리포트·추천 화면 진입 시 |
| 입력 | 마이데이터(취향 분석용) · (B) ①의 소비 분석 요약(카테고리 월평균·반복 결제·요일/시간대 패턴·절약 후보와 예상액·소비 성향 프로필과 충동 구매 지수) · (C) ②의 지킴·성장 상태 · (D) 실제 금융상품 정보 |
| 출력 | 취향·성향 분석 · 주간 리포트 · 월말 리포트 · 맞춤 금융상품 정보성 비교 추천 |
| LLM | 리포트 서술 문장 · 성장·회고 서술 · 추천 사유 문장 |
| 규칙 | 리포트 집계 · 취향 특징 추출 · 상품 매칭·정렬 (구체 방식 미정) |

③은 사용자의 카테고리·강도, ②의 금액·성장 값을 직접 바꾸지 않는다(`07` R1). 금융상품은 **정보성 비교만** 제공한다 — 판매·중개·가입 대행 없이, 실 금리 등 실제 정보를 표시하되 `가입은 각 사로` 안내한다(§8.2). 여행·물건 목표·목적지·보상은 만들지 않는다.

### 11.4 Agent 간 연결

기호는 `07_에이전트_연결부.md`와 일치시킨다. 전달 데이터의 상세 스키마는 구현 단계에서 확정한다.

| 연결 | 방향 | 전달 데이터 | 주기 |
|---|---|---|---|
| A | ① → ② | 기준 소비 · 절약 후보 · 권장 강도 · 추천 근거 | 월초 1회 · 거래 시 증분 |
| B | ① → ③ | 소비 분석 요약(카테고리 월평균·반복 결제·패턴·절약 후보와 예상액·성향 프로필·충동 구매 지수) | 월초·주간·월말 |
| C | ② → ③ | 지킨 돈·차감·월중 상태·월말 확정 지킨 돈·누적 지킨 돈·성장 레벨 | 상태 변경 시·월말 |
| D | 외부 → ③ | 실제 금융상품 정보(실 금리 등, 정보성) | 갱신 시 |
| F | ① → ② | 오분류 재분류 결과 | 오분류 수정 시 |
| G | ② → ① | 월말 지킴 실적 · 맥락 태그 집계 · 오분류율 | 월말 1회 |

돈에서 성장으로 이어지는 처리는 ② 내부에서 끝난다. 별도 ②→③ 성장 반영 연결이 필요 없어 이전 (E)는 폐기하고, (D)는 외부 금융상품 정보 연결로 재정의한다. ①은 ②의 월말 실적(G)을 받은 뒤 다음 달 분석을 시작한다. ③의 리포트·추천은 사용자에게 노출되는 산출물이며 ①·②의 상태를 바꾸지 않는다.

## 12. User Flow

```text
[최초 1회]
마이데이터 연결 (안내 → 전송요구 → 기관 선택 → 인증 → 조회)
        ↓
캐릭터 이름 설정
        ↓
[월초 · 매달 반복]
지난달 결과 확인 → 소비 분석 → 줄일 카테고리 선택 → 강도 선택 → 지킬 돈 확정
        ↓
[월중 · 상시]
Home ↔ 거래 발생 → 알림 → 맥락 태그 / 오분류 수정
        ↓
[월말]
지킨 돈 확정 → 캐릭터 성장·챕터 기록 → 월말 리포트·맞춤 추천 → 다음 달 지킬 것 정하기 ┐
        ↑                                                │
        └────────────────────────────────────────────────┘
```

월말 리포트의 `다음 달 지킬 것 정하기`가 월초 화면으로 되돌아간다. 별도 진입점을 만들지 않는다.

**누적 성장만 월간 루프 밖에 있다.** 월이 바뀌어도 리셋되지 않고 계속 누적된다.

1회차와 N회차는 **같은 화면**을 쓴다. 차이는 월초 화면 상단에 지난달 결과 블록이 붙는지 여부뿐이다. 첫 달은 빈 상태로 처리한다.

### 온보딩 가이드

- 선택 카테고리가 많을수록, 강도가 높을수록 완전 달성이 어려워진다는 점을 확정 전에 안내한다.
- 1차 구현에서는 1~2개 카테고리 선택을 권장한다.
- 지킬 돈 총액을 성과처럼 강조하지 않는다.

## 13. Home 화면 정의

Home는 사용자가 현재 절약 상태를 가장 빠르게 이해할 수 있도록 설계된 핵심 화면이다.

### 한 숫자 구조

Home의 금액 주 지표는 `현재 지키는 금액` 하나다.

| 구분 | 처리 |
|---|---|
| 사용자 노출 | 현재 지키는 금액 |
| 내부 계산 | 기준 소비 · 사용 한도 · 누적 소비 · 남은 한도 · 초과액 |

**기준 소비·사용 한도·남은 한도·소진률·목표선은 Home과 카테고리 상세에 표시하지 않는다.** 사용 한도 안에서는 금액이 그대로 유지되고 서비스가 침묵한다. 지키는 금액이 깎이기 시작하는 순간 자체가 한도 초과 신호다.

### 표시 정보

- 현재 지키는 금액 (주 지표)
- 선택 카테고리와 절약 강도 (금액 한도는 표시하지 않음)
- 월중 상태 (`지키는 중` / `지킬 돈 소진`)
- 지킴이 말풍선 (차감 설명 · 회복 안내)
- 여정 영역 (캐릭터 · 이번 달 챕터 · 성장 레벨). 누적 지킨 돈은 성장 상세에서 확인
- 최근 거래와 맥락 태그
- 초과 지출 (지킬 돈 소진 후)
- 미설정 시 승계 배너 ("지난달 카테고리와 강도를 그대로 지키는 중")

### 지키는 금액 동작

```text
 1일   내부 누적 소비      0원 (한도 이내)      115,000원 지키는 중
 8일   내부 누적 소비 18,500원 (한도 이내)      115,000원 지키는 중 · 알림 없음
20일   내부 누적 소비 130,000원 (15,000 초과)   100,000원 지키는 중 · 차감 알림
월말   내부 누적 소비 140,000원 (25,000 초과)    90,000원 지킴
```

시작값은 이번 달 지킬 돈과 같다. 사용 한도 안에서는 변하지 않고, 한도를 넘어선 뒤부터 초과분만큼 줄어들며 0원에서 멈춘다. 카테고리 상세에는 선택한 강도·관련 거래·차감 기여액만 표시하고 내부 한도 금액은 노출하지 않는다.

### 목표

앱을 실행한 뒤 3초 안에 "이번 달 내가 얼마나 돈을 지키고 있는지"를 이해할 수 있어야 한다.

> **리포트는 주간과 월간을 모두 제공한다.** 주간은 선택 카테고리 소비 흐름과 현재 지키는 금액을 보여주는 요약이며 성장이나 달성 여부를 확정하지 않는다. 월간은 최종 지킨 돈을 마감하고 캐릭터 성장과 챕터 기록을 확정한다. 내부 결과 판정은 분석에 사용하되 사용자 메시지는 지킨 금액을 중심으로 구성한다. 리포트 생성과 서술은 ③이 담당하고, 금액·성장 확정은 ②가 한다.

## 15. AI Responsibility

### Agent 시스템이 담당

- 거래 분석 · 반복 소비 탐색 · 분류 (①)
- 절약 후보 추천 · 기준 소비 산출 · 권장 강도 제안 · 추천 근거 생성 (①)
- 소비 성향 프로필·충동 구매 지수 산출 (①)
- 선택 카테고리 거래 반영과 지킴 상태 갱신 (②)
- 실시간 거래 알림 문구 생성 (②)
- 캐릭터·성장·월간 챕터 상태 진행 (②)
- 취향·성향 분석 (③)
- 주간 리포트 · 월말 리포트 생성과 성장·회고 서술 (③)
- 맞춤 금융상품 정보성 비교 추천과 추천 사유 생성 (③)
- 맥락 태그 집계 기반 다음 달 카테고리·강도 제안 (①)

### 규칙 엔진이 담당

- 사용 한도·지킬 돈·누적 소비·지킨 돈 계산
- 완전 달성·부분 달성·미달성 판정
- 월말 성장 반영액·누적 지킨 돈·레벨 산출
- 성장 반영 멱등성·챕터 상태 전이
- 2차 구현의 월말 이체액 확정과 중복 이체 방지

### 사용자가 담당

- 가치 소비 선택
- 이번 달 줄일 카테고리 선택·확정
- 카테고리별 절약 강도 선택
- 캐릭터 이름 설정
- 소비 맥락 태그 선택
- 오분류 수정

> **AI는 소비 기준을 대신 결정하지 않는다.**
> AI는 분석과 추천을 담당하고, 최종 카테고리·강도는 사용자가 결정한다. 외부 소비 목표는 설정하지 않으며, 금융상품은 정보성 비교만 제공한다(§8.2). LLM은 금액·레벨을 계산하거나 변경하지 않는다.

| 거래 자동 반영 성공률 | 신규 거래 발생 시 정상 판정·반영된 비율 |

| 월말 성장 확인률 | 월말 확정 사용자가 캐릭터 성장 결과까지 확인한 비율 |
| 다음 달 챕터 시작률 | 월말 확정 후 다음 달 카테고리·강도를 다시 설정한 비율 |
| 누적 성장 유지율 | 2개월 이상 연속으로 성장 기록이 생성된 사용자 비율 |
| 금융상품 정보성 추천 노출·유용성 | 월말 추천 노출과 유용성 응답 (지표 정의 미정) |

> 리텐션은 핵심 지표로 유지하되, 더미 데이터 기반 MVP에서는 월말 성장 확인률과 다음 달 챕터 시작률을 대리 지표로 검토한다. 측정 방식은 아직 `TBD`이며 Q4를 삭제하지 않는다.

### UX KPI

| 지표 | 목표 |
|---|---:|
| `지킬 돈`이 이번 달 지켜야 할 금액임을 이해 | 80% 이상 |
| AI 추천 신뢰도 | 80% 이상 |
| 차감 이유 설명 가능 응답 | 80% 이상 |

### 시스템 검증 기준

- 가맹점명 동의를 거부하면 연결이 차단된다.
- 기관 일부 조회 실패 시 성공분만 반영되고 실패분만 재시도된다.
- 계좌 거래는 누적 소비에 반영되지 않는다.
- 신규 거래 발생 시 누적 소비와 내부 사용 한도 상태가 자동 갱신된다.
- 누적 소비가 사용 한도 이내이면 지킨 돈이 변하지 않는다.
- 누적 소비가 사용 한도 이내이면 사용자 알림이 발생하지 않는다.
- 누적 소비가 사용 한도를 넘으면 초과분만큼만 지킨 돈이 줄어든다.
- 누적 소비가 사용 한도를 넘은 순간부터 차감 알림이 발생한다.
- 선택 카테고리 거래만 계산에 반영된다.
- 맥락 태그는 금액에 영향을 주지 않는다.
- 오분류 수정 시 금액이 복구되고 재분류 결과가 적재된다.
- 지킨 돈은 0원에서 멈추고 초과 지출이 별도 누적된다.
- 절약 강도 100%를 선택하면 사용 한도가 0원이 되고 첫 거래부터 지킨 돈이 줄어든다.
- 거래 취소·환불 시 누적 소비가 복구된다.
- 월중 지킨 돈의 변화는 누적 성장에 즉시 반영되지 않는다.
- 월말 확정 지킨 돈이 동일 월에 한 번만 성장 원장에 반영된다.
- 누적 지킨 돈과 캐릭터 레벨은 어떤 경우에도 감소하지 않는다.
- 월이 바뀌면 새 챕터가 시작되고 누적 성장·레벨·외형은 유지된다.
- 0원을 지킨 달에도 캐릭터가 퇴화하거나 비난하는 문구가 나오지 않는다.
- 목적지·남은 거리·소비형 보상은 어떤 화면에도 나타나지 않는다. 금융상품은 정보성 비교만 제공하며 판매·중개·가입 대행 화면은 없다.
- 주간 리포트에 완전 달성·부분 달성·미달성 판정이 나타나지 않는다.
- 월말 누적 소비가 사용 한도 이하이면 완전 달성, 기준 소비 미만이면 부분 달성, 기준 소비 이상이면 미달성으로 판정된다.
- 월말에 ②의 지킴 실적(G)이 도착한 뒤 다음 달 분석이 시작된다.

## 17. Out of Scope

| 기능 | 제외 이유 |
|---|---|
| 월중 강도 변경 | 진행 중인 기준을 소급 변경하면 판정이 흔들림. 월초 확정으로 고정 |
| 성역 카테고리 명시 지정 | **선택하지 않은 카테고리가 이미 성역이다.** 미선택 카테고리는 계산 대상이 아니고(§5.3) 알림도 침묵한다(§6.1). 별도 지정 화면은 사용자 요구가 확인된 뒤 설계 |
| 여행·물건 목표 및 목적지 | 절약 서비스가 새로운 소비 욕구와 사용처를 제안하는 모순 |
| 소비형 보상·상점 추천 | 지킨 돈으로 소비를 유도하지 않음. 사용처는 사용자 자유 |
| 금융상품 판매·중개·가입 대행 | **정보성 비교는 범위 내(§8.2).** 판매·중개·가입 편의는 금소법 부담으로 제외 |
| 참기 세션·자기 보고·사후 추가 기록 | 자동 거래 분석·차감 원장과 중복되고 기록 부담·과장 입력 가능성이 생김 |
| 지킨 날 원장·보호 쿠폰 | 성장 연료를 월말 확정 지킨 돈 하나로 통일 |
| 소비 행동 단위 절약 | 맥락 태그 데이터 축적 이후 가능 |
| 주간 정산의 사후 추가 기록 | 검증 없는 금액이 성장 원장에 유입되는 경로 |
| 계좌 거래의 누적 소비 반영 | 카드 대금 출금과 이중 계상됨 (§10.4) |
| 보험·증권 업권 | 소비 분석에 불필요 |
| 실제 본인인증(CI) 검증 · 실제 개인정보 수집·저장 | 더미 CI로 인증 화면만 시뮬레이션하므로 실제 검증·수집은 불필요 |
| 실제 마이데이터 사업자 연동 | 라이선스·보안 부담, 6주 일정 |
| 실제 계좌 개설·이체 | 실제 금융기관과 자금을 다루지 않는 포트폴리오 프로젝트. 2차에서 가상 금융 서버로 경험만 재현 |
| OS 위젯 | 코어 가설 검증과 무관하고 네이티브 구현 부담이 큼 |
| 푸시 알림 | 마이데이터 표준상 결제 즉시 알림이 불가능하고, MVP 검증에 필수 아님 |
| 미선택 카테고리 풍선효과 분석 | 로그만 적재하고 분석은 후속 단계 |
| 커뮤니티 | 개인 소비 관리 경험에 집중 |
