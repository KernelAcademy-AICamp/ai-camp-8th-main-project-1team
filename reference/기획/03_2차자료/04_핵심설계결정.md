# 04. 핵심 설계 결정

> 상태: 부분 절감 모델 확정 · 2026-07-24
> 이 문서는 프로젝트 방향이 바뀐 과정과 현재 결정을 기록한다.

## 결정해야 했던 질문

**앱은 사용자가 소비를 조절했다는 사실을 어떻게 알 수 있는가?**

돈을 쓰면 카드 내역이 남는다. 하지만 사지 않은 것, 줄인 소비는 어디에도 기록되지 않는다.

MOA는 이 신호를 만드는 방법을 검토하며 여러 차례 방향을 조정했다.

---

## 1차 검토 — 사용자가 참은 순간을 직접 기록한다

처음에는 사용자가 참은 순간을 직접 기록하는 방식을 검토했다.

> 오늘 배달 22,000원 참았어요

### 장점

- 사용자가 실제로 참은 순간을 스스로 판단할 수 있음
- 기록 행위 자체가 성취감을 강화함
- 간단한 입력 UI만으로 구현 가능

### 기각 이유

- 매번 기록하는 일이 또 하나의 숙제가 됨
- 앱을 열지 않으면 데이터가 쌓이지 않음
- 참은 금액을 사용자가 과장하거나 임의로 입력할 수 있음
- 실제 소비가 목표를 얼마나 위협하는지 보여주지 못함

직접 기록은 성취감은 만들 수 있지만 `뒤늦은 소비 인지`와 `무엇을 줄일지 모름`이라는 핵심 문제를 충분히 해결하지 못했다.

---

## 2차 결정 — 지킬 돈을 먼저 정하고 실제 소비를 자동 차감한다

사용자가 참은 순간을 기록하는 대신 과거 소비로 이번 달 지킬 돈을 만들고, 선택한 소비가 실제로 발생할 때마다 자동으로 차감하는 방향을 채택했다.

```text
과거 마이데이터 분석
→ AI가 줄일 가능성이 있는 소비 후보 제안
→ 사용자가 후보 검수·선택
→ 이번 달 지킬 돈 확정
→ 신규 거래 수신
→ 선택 카테고리 거래를 자동 반영
→ 지킨 돈 갱신
→ 월말 지킨 돈 확정
```

사용자는 더 이상 `참았어요`를 직접 기록하지 않는다. 거래가 들어오면 시스템이 자동으로 금액을 갱신하고, 사용자는 분류가 틀렸을 때만 수정한다.

---

## 3차 검토 — 부분 절감

카테고리를 완전히 끊는 부담을 줄이기 위해 카테고리별 절약 강도를 두는 부분 절감 모델을 설계했다.

```text
기준 소비 230,000원
사용 한도 115,000원
지킬 돈   115,000원
```

이 모델은 현실적인 소비 허용치를 줄 수 있다는 장점이 있었다. 그러나 당시 설계에서 다음 문제가 보였다.

- `사용 한도`와 `지킬 돈`을 동시에 설명해야 함
- 사용자가 115,000원을 쓰는 돈인지 지키는 돈인지 혼동할 수 있음
- Home에 기준 소비·사용 한도·누적 소비·남은 한도가 함께 등장함
- 감소하는 게이지와 목표선을 동시에 해석해야 함
- 절약 강도별 비율과 사용자 직접 수정 규칙이 추가됨

이 시점에는 부담이 크다고 판단해 1차 구현에서 보류했다.

---

## 4차 검토 — 카테고리 전체 절감으로 단순화

지킬 돈을 선택 카테고리 기준 소비 전액으로 두고, 목표 소비를 0원으로 고정했다.

```text
지킬 돈     = Σ(선택 카테고리 기준 소비)
지키는 금액 = max(지킬 돈 − 누적 소비, 0)
```

숫자가 하나로 줄어 설명은 쉬워졌다. 그러나 화면 시안을 그려 검토하면서 두 가지 문제가 드러났다.

**첫째, 절감 강도를 고를 수 없어 목표가 전부 아니면 전무가 된다.** 택시를 끊기로 하면 이번 달 택시를 한 번도 타지 않아야 완전 달성이다. 출근길에 한 번 늦잠을 자면 그 달의 완전 달성은 사라진다.

**둘째, 게이지가 사용자에게 여유를 준다.** 기준 소비 230,000원에서 시작해 거래마다 깎이는 화면은 `아직 211,500원이나 남았다`로 읽힌다. 절약을 유도하려던 화면이 소비를 허락하는 화면이 됐다.

3차에서 지적했던 복잡도 문제는 실재한다. 그러나 그것은 **금액 계산이 아니라 화면 설계의 문제**였다. 계산식을 다시 세우면 숫자를 늘리지 않고도 부분 절감을 구현할 수 있다는 것이 4차 검토의 결론이다.

---

## 현재 결정 — 부분 절감 (전체 절감을 포함하는 모델)

사용자는 카테고리마다 절약 강도를 고른다. 강도는 그 카테고리의 기준 소비 중 **얼마를 지킬 것인가**를 뜻한다.

```text
사용 한도 = 기준 소비 × (1 − 절약 강도)   이번 달 써도 되는 돈
지킬 돈   = 기준 소비 × 절약 강도          이번 달 지키기로 한 돈
```

절약 강도 100%를 고르면 사용 한도가 0원이 되어 4차 검토의 카테고리 전체 절감과 완전히 같아진다. **전체 절감은 부분 절감의 특수한 경우다.** 두 모델 중 하나를 고른 것이 아니라, 하나가 다른 하나를 품는다.

### 계산 구조

```text
기준 소비 = 카테고리별 평소 소비 수준 (최근 3개월 월평균 · 잠정)
절약 강도 = 사용자가 카테고리별로 선택
사용 한도 = 기준 소비 × (1 − 절약 강도)   내부 차감 기준
지킬 돈   = 기준 소비 × 절약 강도          사용자 노출 금액
누적 소비 = 이번 달 선택 카테고리 실제 지출 합계

지킨 돈   = max(지킬 돈 − max(누적 소비 − 사용 한도, 0), 0)
```

**한도를 넘긴 만큼만 깎인다.** 이것이 유일한 차감 규칙이다.

사용 한도는 차감을 시작하는 내부 기준이며 사용자 화면에 금액으로 노출하지 않는다. 사용자는 자신이 지키기로 한 `지킬 돈`만 본다. 사용 한도 안에서는 지킨 돈이 그대로 유지되고 알림도 보내지 않는다. 한도를 넘긴 거래가 발생한 순간부터 초과액과 지킨 돈의 변화를 알린다.

식을 구간으로 나눠 보면 두 모델의 관계가 드러난다.

```text
누적 소비 ≤ 사용 한도   →   지킨 돈 = 지킬 돈           변하지 않는다
누적 소비 > 사용 한도   →   지킨 돈 = max(기준 소비 − 누적 소비, 0)
```

한도를 넘어선 뒤에는 4차 검토의 공식과 정확히 같아진다. 두 구간은 사용 한도 지점에서 끊김 없이 이어진다.

### 예시

```text
택시 기준 소비  80,000원 · 강도 50% → 사용 한도 40,000원 · 지킬 돈 40,000원
배달 기준 소비 150,000원 · 강도 50% → 사용 한도 75,000원 · 지킬 돈 75,000원

이번 달 기준 소비 230,000원 · 사용 한도 115,000원 · 지킬 돈 115,000원

--- 8일차 · 택시 18,500원 결제 ---
누적 소비 18,500원 (사용 한도 이내)
지킨 돈 115,000원 · 변화 없음
사용자 알림 없음

--- 20일차 · 누적 소비 130,000원 ---
사용 한도를 15,000원 초과
지킨 돈 115,000 − 15,000 = 100,000원
사용자에게 초과액과 차감 결과 알림

--- 월말 · 택시 42,000원 + 배달 98,000원 ---
누적 소비 140,000원 · 초과 25,000원
지킨 돈 115,000 − 25,000 = 90,000원 → 부분 달성
```

지킨 돈은 0원에서 멈춘다. 누적 소비가 기준 소비를 넘으면 그 초과분은 별도로 표시한다.

### 달성 상태

| 상태 | 조건 | 의미 |
|---|---|---|
| 완전 달성 | 누적 소비 ≤ 사용 한도 | 한도를 지켜 지킬 돈을 전부 지킴 |
| 부분 달성 | 사용 한도 < 누적 소비 < 기준 소비 | 한도는 넘겼지만 평소보다 덜 써서 일부를 지킴 |
| 미달성 | 누적 소비 ≥ 기준 소비 | 평소 수준 이상 사용해 지킨 돈이 0원 |

월중에는 결과가 확정되지 않으므로 `지키는 중` 또는 `지킬 돈 소진`으로만 표현한다.

---

## 화면에는 지킬 돈 하나만 보여준다

3차 검토가 우려한 것은 기준 소비·사용 한도·누적 소비·남은 한도가 한 화면에 함께 등장하는 상황이었다. 현재 모델은 계산과 노출을 분리해 이를 피한다.

| 구분 | 값 | 역할 |
|---|---|---|
| 내부 계산 | 기준 소비 · 사용 한도 · 누적 소비 · 초과액 | 차감 시점과 금액을 결정 |
| 사용자 노출 | 현재 지키는 금액 | 이번 달 지켜야 할 돈 |

예를 들어 기준 소비가 100,000원이고 강도 50%를 고르면 내부 사용 한도는 50,000원이지만, 사용자 화면에는 `이번 달 50,000원 지키는 중`만 표시한다.

Home에는 기준 소비·사용 한도·남은 한도·소진률·목표선을 표시하지 않는다. 사용 한도 안에서는 금액이 유지되고 서비스가 침묵한다. 한도를 넘긴 순간부터 초과액만큼 지키는 금액이 줄어들며, 그 변화가 곧 초과 신호가 된다.

---

## 5차 검토 — 소비 목표와 보상이 서비스 가치와 충돌한다

처음에는 지킨 돈을 여행·물건 같은 목표와 연결하고, 마일스톤마다 보상이나 금융상품을 추천하려 했다.

```text
기각한 구조
절약 → 목표 금액 달성 → 여행·물건 구매 또는 보상 추천
```

그러나 MOA의 가치는 `아낄 수 있는 돈을 아끼는 것`이다. 서비스가 새 소비 목적지를 정하게 하거나 보상을 추천하면, 절약을 돕는 동시에 새로운 소비 욕구를 만드는 모순이 생긴다. 목표가 외부 대상에 있으면 사용자의 관심도 절약 행위보다 `무엇을 살 것인가`로 이동한다.

따라서 여행·물건 목표, 도착지, 남은 거리, 보상 추천, 금융상품 추천을 코어 루프에서 제거한다. 지킨 돈의 사용처는 사용자의 자유이며 MOA가 관여하지 않는다.

## 현재 결정 — 자동 지킴과 캐릭터 여정·성장의 2층 구조

기존의 자동 분석·부분 차감 구조는 유지한다. 바꾸는 것은 동기 부여 층뿐이다.

```text
월간 금액 층   지킬 돈          한도 초과분만큼 감소   매월 새로 시작   ②
누적 성장 층   캐릭터 여정·성장  월말 확정 지킨 돈 누적  감소·리셋 없음   ③
```

캐릭터는 목적지에 도착하기 위해 걷는 존재가 아니다. 사용자가 지키기로 한 돈을 함께 지키며 매달 새로운 구간을 이어가는 **유일한 감정 이입 대상**이다. 종착점·남은 거리·소비 목표는 표시하지 않는다.

### 성장 연료 — 월말에 확정된 지킨 돈

월중 `지킨 돈`은 거래 취소·환불과 오분류 수정으로 바뀔 수 있다. 이를 즉시 성장에 반영하면 나중에 캐릭터를 되돌려야 하므로 `성장은 감소하지 않는다`는 원칙과 충돌한다.

따라서 성장 원장에는 월말에 마감된 값만 한 번 반영한다.

```text
이번 달 성장 반영액 = 월말 확정 지킨 돈
누적 지킨 돈       = 이전 누적 지킨 돈 + 이번 달 성장 반영액
성장 레벨          = 누적 지킨 돈이 속한 성장 구간
```

- 월중: 캐릭터가 이번 달 돈을 지키며 여정을 이어가는 모습을 보여준다.
- 월말: 최종 지킨 돈을 확정하고 누적 지킨 돈에 한 번만 더한다.
- 월 전환: 새 챕터를 시작하지만 누적 지킨 돈·레벨·외형은 유지한다.
- 0원을 지킨 달: 성장은 멈추지만 감소하거나 비난하지 않는다.
- 월말 이후 지연 취소·정정: 이미 확정된 성장 원장을 소급 차감하지 않고 별도 정책으로 처리한다.

### 함께 폐기하는 장치

금액 외에 별도 성장 연료를 만들지 않는다. 사용자의 선언·자기 보고·참기 세션·지킨 날 원장·보호 쿠폰은 자동 거래 원장과 중복되거나 사용자의 기록 부담을 만든다. 위젯은 코어 가설에 필요하지 않아 MVP에서 제외한다.

여정·성장 상세는 `10_여정과_성장.md`를 따른다.

## 소비 맥락은 금액이 아니라 데이터로 회수한다

거래마다 소비 맥락 태그를 선택할 수 있다.

```text
🎯 계획했던 소비 · 😅 줄이려던 소비 · 🚨 불가피한 소비
```

- 세 태그 모두 누적 소비에 그대로 반영된다.
- 맥락 태그는 금액을 면제하지 않는다.
- 불가피한 소비가 반복된 카테고리는 다음 달 후보에서 제외하거나 강도를 낮춰 제안한다.
- 금액을 되돌리는 경로는 오분류 수정과 거래 취소·환불뿐이다.

---

## 왜 이 모델을 1차 구현으로 선택했는가

- **차감이 완전히 설명 가능**: 깎인 금액과 한도를 넘긴 금액이 1:1로 같다.
- **화면 단순화**: Home에는 현재 지키는 금액 하나만 둔다. 기준 소비와 사용 한도는 내부 계산값이며 목표선도 없다.
- **알림 최소화**: 사용 한도 안에서는 침묵하고, 한도를 넘겨 지킬 돈이 실제로 줄어드는 순간부터 알린다.
- **실패 지점이 완만함**: 강도를 고를 수 있으므로 한 번의 소비로 그 달이 끝나지 않는다.
- **손실 회피가 절약을 유도**: 한도를 넘기는 순간부터 눈앞에서 금액이 줄어든다.
- **자동 반영**: 사용자가 기록하지 않아도 거래로 상태가 갱신된다.
- **통제감 유지**: AI는 후보와 근거만 제안하고 사용자가 카테고리와 강도를 확정한다.
- **확장이 아니라 포함**: 강도 100%가 곧 전체 절감이므로 두 모델을 따로 만들지 않는다.

카테고리 수가 많을수록 완전 달성이 어려워지므로 1차 구현에서는 1~2개 선택을 권장한다.

## 지킴 계좌와의 관계

1차 구현의 월중 지킨 돈은 아직 별도 계좌에 들어간 돈이 아니다.

```text
1차 구현
소비 분석 → 카테고리·강도 선택 → 지킴 추적 → 월말 지킨 돈 확정

2차 구현
월말 지킨 돈 확정 → 가상 지킴 계좌로 이체 → 누적 잔액 보관
```

2차 구현에서는 완전 달성 여부와 관계없이 월말 지킨 돈이 0원보다 크면 가상 지킴 계좌로 옮긴다. 실제 계좌와 실제 자금은 사용하지 않는다.

## 지킨 것과 미룬 것

### 지킨 것

- AI는 정답이 아니라 후보와 근거를 제안한다.
- 가치 소비의 기준은 사용자에게 있다.
- 사용자가 직접 선택한 소비만 반영한다.
- 성공을 지적이 아닌 보호의 언어로 보여준다.
- 분류 오류를 사용자가 쉽게 수정하고 다음 분석에 반영한다.
- 실패보다 회복을 지원한다. 캐릭터 성장은 뒤로 가지 않는다.
- 감정 이입 대상을 하나로 유지한다.
- 지킨 돈의 사용처를 제안하거나 평가하지 않는다.

### 폐기한 것

- 여행·물건 목표와 목적지 — 절약이 새 소비를 유도하는 모순
- 보상·금융상품 추천 — 서비스가 지킨 돈의 사용처에 개입
- 참기 세션과 자기 보고 — 자동 분석·차감 원장과 중복
- 지킨 날 원장(`daily_ledger`)과 보호 쿠폰 — 별도 성장 연료를 두지 않음
- 도착률·남은 거리·예상 도착일 — 종착점 없는 여정 원칙과 충돌

### Later로 미룬 것

- 카테고리별 강도를 월중에 변경하는 기능
- 성역 카테고리 명시 지정 — 선택하지 않은 카테고리가 이미 성역이므로 구조적으로는 불필요하다
- 소비 행동 단위 절약 (야식 배달만 / 늦잠 택시만)
- OS 위젯

## 프로토타입에서 확인할 질문

- 사용자는 `사용 한도`와 `지킬 돈`을 구분해서 이해하는가?
- 한도 내에서 지킨 돈이 변하지 않는 것을 안정으로 느끼는가, 무감각으로 느끼는가?
- 한도를 넘긴 순간의 차감이 행동 조정으로 이어지는가?
- 절약 강도 선택이 부담을 줄이는가, 결정을 늘리는가?
- 1~2개 카테고리 추천이 현실적인가?
- 부분 달성도 성과로 받아들이는가?
- 외부 소비 목표 없이도 캐릭터의 여정과 성장이 동기로 작동하는가?
- 월중에는 성장하지 않고 월말에 확정 성장하는 구조를 자연스럽게 이해하는가?
- 이번 달 지키는 금액과 누적 지킨 돈을 서로 다른 숫자로 이해하는가?
- 종착점 없는 월간 챕터가 다음 달 재방문 동기가 되는가?
