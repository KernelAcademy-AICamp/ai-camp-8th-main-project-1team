# 07. Agent 연결부 계약

> 프로젝트명: MOA
> 문서 목적: 3개 Agent가 주고받는 데이터와 영향 관계 정의
> 상태: Draft v5 · 소비 분석(①) + 지킴·성장(②) + 취향·추천(③) 3-Agent 모델
> 최종 업데이트: 2026-07-24
> 담당: ① 소비 분석(상현) · ② 지킴·성장(윤정) · ③ 취향·추천(가원)

이 문서는 세 사람이 병렬로 개발하기 위한 계약서다. 각 Agent는 상대 Agent를 이 스키마대로 모킹하고 독립적으로 작업한다.

---

## 0. 용어 확정

| 용어 | 소유 | 뜻 |
|---|---|---|
| **기준 소비** | ① | 카테고리별 평소 소비 수준. 최근 3개월 월평균 잠정 |
| **절약 후보** | ① | AI가 이번 달 줄여볼 대상으로 제안한 카테고리 |
| **권장 강도** | ① | AI가 제안하는 절약 강도 초기값 |
| **가치 소비 보호 신호** | ① | 반복되는 취미·가족·건강 소비 등을 절약 후보에서 보호하기 위한 신호 |
| **선택 카테고리** | ② | 사용자가 이번 달 줄이기로 확정한 카테고리 |
| **절약 강도** | ② | 사용자가 카테고리별로 확정한 지킬 비율 |
| **사용 한도** | ② | `기준 소비 × (1 − 절약 강도)`. 차감 시작점을 정하는 내부값 |
| **지킬 돈** | ② | `기준 소비 × 절약 강도`. 월초 확정값 |
| **누적 소비** | ② | 이번 달 선택 카테고리에서 발생한 카드 승인 금액 합계 |
| **남은 한도** | ② | `max(사용 한도 − 누적 소비, 0)`. 사용자에게 노출하지 않는 내부값 |
| **지킨 돈** | ② | `max(지킬 돈 − max(누적 소비 − 사용 한도, 0), 0)`. 월중 변동값 |
| **월말 확정 지킨 돈** | ② | 해당 월 원장을 마감한 뒤 확정한 지킨 돈 |
| **월간 챕터** | ② | 한 달 동안 캐릭터가 지킬 돈을 지키는 여정 구간(상태) |
| **누적 지킨 돈** | ② | 월말 확정 지킨 돈의 합계. 감소하거나 리셋되지 않음 |
| **성장 레벨** | ② | 누적 지킨 돈 구간으로 계산한 캐릭터 성장 단계 |
| **소비 성향 프로필** | ① | 소비 성향 요약과 충동 구매 지수 |
| **취향 분석** | ③ | 마이데이터로 도출한 소비자 취향·선호 특징 |
| **금융상품 정보성 추천** | ③ | 실제 상품 정보를 소비 패턴·취향에 맞춰 개인화한 정보성 비교. 판매·중개 없음 |
| **주간·월말 리포트** | ③ | ①의 분석과 ②의 지킴·성장 성과를 받아 서술한 리포트 |

절약 강도 100%를 고르면 사용 한도가 0원이 되어 카테고리 전체 절감과 같아진다.

> ②의 `지킨 돈`은 월중 거래에 따라 바뀌는 현재값이고 `누적 지킨 돈`은 월말 확정값만 쌓인 장기 성장값이다. 둘 다 ②가 소유하며, ③은 이를 읽어 리포트로 서술한다.

## 1. 전체 관계

```text
                     (A) 분석 결과
┌──────────────────┐ ───────────────▶ ┌──────────────────┐
│ ① 소비 분석      │ ── (F) 재분류 ──▶ │ ② 지킴·성장      │
│    (상현)         │ ◀── (G) 월말 실적 │    (윤정)         │
└──────────────────┘                  └──────────────────┘
        │                                     │
        │ (B) 소비 분석 요약                    │ (C) 지킴·성장 상태
        ▼                                     ▼
┌───────────────────────────────────────────────────────┐
│ ③ 취향·추천 (가원)                                      │
│   ◀── (D) 외부 실제 금융상품 정보                        │
│   ──▶ 사용자: 주간·월말 리포트 + 맞춤 금융상품 정보성 추천 │
└───────────────────────────────────────────────────────┘
```

| 연결 | 방향 | 성격 | 주기 |
|---|---|---|---|
| A | ① → ② | 배치·이벤트 | 월초 1회 · 신규 거래 분류 시 |
| B | ① → ③ | 배치 | 월초·주간·월말 |
| C | ② → ③ | 상태 이벤트 | 지킴·성장 상태 변경 시 · 월말 |
| D | 외부 → ③ | 데이터 | 갱신 시 |
| F | ① → ② | 이벤트 | 오분류 수정 시 |
| G | ② → ① | 배치 | 월말 1회 |

돈에서 성장으로 이어지는 처리는 ② 내부에서 끝난다. 이전 (E) ③→① 성장 요약과 (D) ②→③ 성장 반영은 폐기하고, (D)를 외부 금융상품 정보 연결로 재정의했다. ③에서 ①·②로 상태를 바꾸는 역방향 연결은 없다.

## 2. 영향 규칙

### R1. ③은 ②의 금액·성장과 ①의 선택을 바꾸지 않는다

③은 ①의 분석과 ②의 지킴·성장 상태를 읽어 리포트·추천으로 표현할 뿐이다.

```text
허용  "이번 달 100,000원을 지키는 중이에요."
허용  "아낀 돈을 둘 곳으로 이런 예·적금이 있어요. 가입은 각 사로."

금지  강도·지킬 돈·지킨 돈·성장 레벨을 임의 변경
금지  특정 상품 가입을 위해 목표 금액·페이스를 만들어 유도
```

### R2. 계산은 규칙 엔진, 표현은 LLM

| 규칙 엔진 | LLM |
|---|---|
| 기준 소비·사용 한도·지킬 돈·지킨 돈 계산 (②) | 실시간 차감 설명 (②) |
| 월말 확정·중복 반영 방지 (②) | 추천 근거·성향 문장 (①) |
| 누적 지킨 돈·성장 레벨 계산 (②) | 주간·월말 리포트·성장·회고 서술 (③) |
| 리포트 집계·상품 매칭·정렬 (③, 방식 미정) | 추천 사유 문장 (③) |

LLM은 금액·레벨·판정을 생성하거나 수정하지 않는다.

### R3. ②는 사실을 만들고 ③은 소통한다

| 시점 | ② 지킴·성장 | ③ 취향·추천 |
|---|---|---|
| 월초 | 지킬 돈·내부 사용 한도 확정 · 새 챕터 시작 | — |
| 월중 | 거래 자동 반영·초과분 차감·실시간 알림 | — |
| 주간 | 지킴 상태 제공 (C) | 판정 없는 주간 리포트 |
| 월말 | 최종 지킨 돈 확정·성장 누적·챕터 기록 | 월말 리포트·성장 서술·맞춤 금융상품 정보성 추천 |

### R4. 금액 복구 경로는 재분류와 취소·환불뿐이다

맥락 태그, 캐릭터 상태, 챕터 연출은 ②의 금액에 영향을 주지 않는다.

### R5. 월중 기준은 소급 변경하지 않는다

진행 중인 기준 소비·선택 카테고리·절약 강도는 월말까지 고정한다. 다음 달 제안은 새 달 선택 화면에서 사용자가 확정한다.

### R6. 다음 달 분석은 월말 마감을 기다린다

```text
1. ② 월말 지킨 돈 확정·성장 누적 → G 발행
2. ①이 G를 수신 → 다음 달 분석 → A와 B 발행
```

③의 리포트·추천은 사용자 산출물이며 다음 달 분석을 막지 않는다.

### R7. 계좌 거래는 ①에서만 소비한다

②의 누적 소비는 카드 승인내역만으로 계산한다. 계좌 거래는 고정지출·구독 탐지와 절약 후보 보호에만 사용한다. 카드 대금 출금의 이중 계상을 막기 위해서다.

### R8. 성장은 월말 확정값만 먹고 뒤로 가지 않는다 (②)

성장 계산·누적은 ② 내부에서 처리한다.

```text
이번 달 성장 반영액 = 월말 확정 지킨 돈
누적 지킨 돈 += 이번 달 성장 반영액
성장 레벨 = 누적 지킨 돈 구간
```

- 월중 지킨 돈 변동은 성장에 반영하지 않는다.
- 동일한 `idempotency_key`는 한 번만 반영한다.
- 지킨 돈이 0원인 달에는 성장값이 유지된다.
- 누적 지킨 돈·레벨·외형을 차감하거나 리셋하지 않는다.
- 월말 이후 늦게 들어온 취소·정정은 별도 보정 정책으로 처리하되 이미 확정된 성장값을 소급 차감하지 않는다.

### R9. 소비 목표·보상은 배제, 금융상품은 정보성만 (D)

목적지·목표 금액·도착률·예상 페이스·소비형 보상·상점 데이터는 어느 Agent 스키마에도 두지 않는다. 지킨 돈의 사용처는 사용자의 자유다.

금융상품은 예외다. 실제 상품 정보는 외부 연결 (D)로 ③에만 들어오며, ③은 개인화된 **정보성 비교**로만 사용한다. 판매·중개·가입 대행 데이터, 상품에 맞춘 목표 금액·달성 페이스는 두지 않는다.

## 3. 연결부 스키마

### (A) ① → ② · 분석 결과

```json
{
  "period": "2026-07",
  "generated_at": "20260701093000",
  "categories": [
    {
      "id": "taxi",
      "name": "택시",
      "baseline": 80000,
      "baseline_method": "avg_3m",
      "recommended_to_cut": true,
      "recommended_intensity": 0.5,
      "reason": "최근 3개월 택시 이용의 68%가 평일 밤 10시 이후였어요.",
      "confidence": 0.82
    },
    {
      "id": "delivery",
      "name": "배달",
      "baseline": 150000,
      "baseline_method": "avg_3m",
      "recommended_to_cut": true,
      "recommended_intensity": 0.5,
      "reason": "금요일 밤 주문이 반복되고 있어요.",
      "confidence": 0.79
    }
  ],
  "fixed_expenses": [
    { "label": "통신비", "amount": 55000, "day_of_month": 25, "source": "bank" }
  ]
}
```

- `recommended_intensity`는 추천일 뿐이며 최종 확정은 사용자 입력이다.
- `reason`은 필수다.
- `confidence`가 임계값 미만이면 불확실 표시를 붙인다.
- `fixed_expenses`는 절약 후보에서 제외한다.

### (B) ① → ③ · 소비 분석 요약

③의 리포트·취향 분석·상품 매칭에 쓰는 소비 분석 요약이다. 필드는 예시이며 상세 스키마는 구현 단계에서 확정한다.

```json
{
  "period": "2026-07",
  "category_monthly_avg": { "택시": 80000, "배달": 150000 },
  "recurring_payments": [ { "label": "구독A", "amount": 9900, "cycle": "monthly" } ],
  "time_patterns": ["평일 밤 10시 이후 택시", "금요일 밤 배달"],
  "cut_candidates": [ { "category": "택시", "expected_saving": 40000 } ],
  "profile_summary": "야근한 날 저녁에 소비가 몰리는 편이에요.",
  "impulse_index": 0.62,
  "protected_value_signals": ["건강", "가족"]
}
```

- 원본 거래 전체가 아니라 분석 요약을 전달한다. ③이 필요 시 마이데이터로 취향을 별도 분석한다.
- 보호 신호를 소비형 목표·보상 추천으로 사용하지 않는다.

### (C) ② → ③ · 지킴·성장 상태

②가 소유한 금액·성장 상태를 ③이 리포트로 서술하기 위해 읽는다. 필드는 예시이며 상세는 구현 단계에서 확정한다.

```json
{
  "period": "2026-07",
  "as_of": "20260715213000",
  "protect_amount": 115000,
  "kept_amount": 100000,
  "status": "keeping",
  "days_left": 16,
  "last_change": { "type": "over_limit_deduction", "deducted_amount": 15000 },
  "finalized": false,
  "finalized_kept_amount": null,
  "cumulative_kept_amount": 330000,
  "level": 2,
  "chapter_id": "chapter-2026-07",
  "weekly_summary": {
    "from": "20260708", "to": "20260714",
    "selected_category_spend": 42000, "deducted_amount": 15000,
    "recovered_amount": 0, "transaction_count": 4
  }
}
```

- `status`: `keeping` | `depleted`
- 월말 마감 시 `finalized=true`가 되고 `finalized_kept_amount`·`cumulative_kept_amount`·`level`이 확정값으로 채워진다.
- `weekly_summary`는 주간 마감 이벤트에만 포함한다.
- ③은 C를 표시·서술용으로만 쓰고 금액·성장 값을 바꾸지 않는다.
- `spend_limit`, `remaining_limit`, `baseline_total`은 사용자 표시 계약에 포함하지 않는다.

### (D) 외부 → ③ · 실제 금융상품 정보

아낀 돈의 보관처(저축)를 정보성으로 비교하기 위한 실제 상품 정보다. 데이터 출처와 상품군 범위는 **미정**이며, 필드는 예시다.

```json
{
  "fetched_at": "20260731",
  "products": [
    { "provider": "○○은행", "name": "○○적금", "type": "적금", "rate": 3.8, "terms": "12개월", "source_url": "..." }
  ]
}
```

- `rate` 등은 실제 값이며 `가입은 각 사로` 안내와 함께 정보성으로만 표시한다.
- 판매·중개·가입 편의 필드(제휴 링크, 신청 폼 등)는 두지 않는다.
- ③은 이 정보를 소비 패턴·취향에 맞춰 정렬·비교한다(매칭 방식 미정).

### (E) ③ → 사용자 · 리포트·추천 산출물

폐기된 옛 (E) ③→① 성장 요약을 대체한다. ③의 산출물은 Agent 간 상태를 바꾸지 않는 **사용자 노출물**이다. 필드는 예시다.

```json
{
  "period": "2026-07",
  "weekly_report": { "kept_amount": 100000, "narrative": "이번 주도 잘 지켰어요." },
  "monthly_report": { "finalized_kept_amount": 90000, "growth_narrative": "누적 420,000원과 함께 Lv.3이 되었어요." },
  "taste_summary": "야식·구독 절약에 반응이 좋은 편",
  "product_suggestions": [ { "provider": "○○은행", "name": "○○적금", "rate": 3.8, "reason": "매달 지키는 금액 규모에 맞아요", "note": "가입은 각 사로" } ]
}
```

- 다음 달 분석은 ②의 (G)만으로 시작한다. ③ 산출물은 게이팅에 쓰지 않는다.

### (F) ① → ② · 재분류

```json
{
  "approved_num": "0021458823001",
  "from_category": "taxi",
  "to_category": "transport_fixed",
  "amount": 23500,
  "action": "revert_and_reassign"
}
```

②는 원 카테고리 누적 소비를 복구한다. 새 카테고리가 선택 대상이면 다시 가산하고, 아니면 추적 대상에서 제외한다.

### (G) ② → ① · 월말 지킴 실적

```json
{
  "period": "2026-07",
  "baseline_total": 230000,
  "spend_limit": 115000,
  "protect_amount": 115000,
  "accumulated_spend": 140000,
  "kept_amount": 90000,
  "result": "partial",
  "context_tags": {
    "taxi": { "planned": 1, "tried": 1, "unavoidable": 3 },
    "delivery": { "planned": 2, "tried": 1, "unavoidable": 0 }
  },
  "misclassify_rate": {
    "taxi": 0.0,
    "delivery": 0.08
  }
}
```

- `result`: `full` | `partial` | `missed`
- `full`: `accumulated_spend ≤ spend_limit`
- `partial`: `spend_limit < accumulated_spend < baseline_total`
- `missed`: `accumulated_spend ≥ baseline_total`
- `result`는 다음 달 분석용 내부 값이다.

검산:

```text
kept_amount
= max(protect_amount − max(accumulated_spend − spend_limit, 0), 0)
= 115,000 − 25,000
= 90,000
```

## 6. 사용자 표현 원칙

| 시점 | 사용자 표현 |
|---|---|
| 월중 · 한도 이내 | 알림 없음 · Home `115,000원 지키는 중` |
| 월중 · 한도 초과 | `정해둔 범위를 15,000원 넘었어요. 100,000원 지키는 중이에요.` |
| 주간 | `이번 주 소비 흐름을 확인해봤어요. 지금 100,000원 지키는 중이에요.` |
| 월말 | `이번 달 90,000원을 지켰어요.` |
| 레벨 상승 | `누적 420,000원과 함께 모아가 Lv.3으로 성장했어요.` |
| 2차 가상 이체 후 | `지킴 계좌에 90,000원을 담았어요.` |
| 상품 추천 | `아낀 돈을 둘 곳으로 이런 예·적금이 있어요. 가입은 각 사로.` |

다음 표현은 금지한다.

- `오사카까지 27%`, `584,000원 남았어요`
- `목표 실패`, `보상을 골라보세요`
- `이 상품에 지금 가입하세요`, `여기서 바로 신청`(판매·중개 유도)
- `이 적금 채우려면 매달 얼마`(목표·페이스 재도입)
- `캐릭터가 약해졌어요`, `레벨이 내려갔어요`

## 7. 2차 구현 — 가상 금융 서버

②는 월말 확정 후 지킨 돈을 변경하지 않고 가상 금융 서버에 전달한다.

```json
{
  "idempotency_key": "user-001:2026-07:kept-amount",
  "from_account_id": "virtual-spending-001",
  "to_account_id": "virtual-protect-001",
  "amount": 90000,
  "reason": "monthly_kept_amount"
}
```

- 이체액은 ②의 월말 확정 지킨 돈과 같아야 한다.
- 0원은 이체하지 않는다.
- 같은 월의 멱등키로 중복 이체를 막는다.
- 계좌에는 누적 잔액만 표시하며 소비 목표 진척률을 계산하지 않는다.
