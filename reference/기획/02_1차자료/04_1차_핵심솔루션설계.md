# 04. 핵심 설계 결정

## 결정해야 했던 질문

**앱은 사용자가 소비를 참았다는 사실을 어떻게 알 수 있는가?**

돈을 쓰면 카드 내역이 남는다.

하지만 사지 않은 것과 참은 행동은 기록되지 않는다.

이 신호를 만드는 방법은 두 가지다.

---

## 선택지 1. 사용자가 직접 기록한다

예시:

> 오늘 배달 22,000원 참았어요

### 장점

- 사용자가 실제로 참은 순간을 직접 판단할 수 있음
- 기록 행위가 성취감을 강화함
- 간단한 입력 UI만으로 MVP 구현 가능
- 금융 데이터 자동 수집이 필요하지 않음

### 단점

- 매번 기록하기 귀찮음
- 앱을 열지 않으면 데이터가 쌓이지 않음
- 금액을 과장하거나 잘못 입력할 수 있음

---

## 선택지 2. 앱이 자동으로 추정한다

예시:

> 이번 달 배달비가 평소보다 80,000원 줄었어요

### 장점

- 사용자의 입력 부담이 거의 없음
- 장기 사용과 리텐션에 유리할 수 있음
- 반복 소비 패턴을 자동으로 요약할 수 있음

### 단점

- 정말 참은 것인지, 단순히 살 일이 없었던 것인지 구분하기 어려움
- 실시간 금융 데이터와 기준선 분석이 필요함
- 사용자가 해냈다는 주체감이 약해질 수 있음
- 기존 가계부 분석과 차별점이 흐려질 수 있음

---

## 우리의 결정

### MVP는 직접 기록으로 시작한다

우리가 가장 먼저 검증해야 하는 질문은 이것이다.

> 사용자는 참은 순간을 성취로 남기고 싶어 하는가?

처음부터 자동으로 처리하면 이 욕구를 확인할 수 없다.

따라서 MVP에서는 직접 기록으로 시작한다.

---

## 직접 기록의 마찰을 줄이는 방법

### 원터치 기록

배달, 택시, 쇼핑처럼 자주 참는 항목을 버튼으로 제공한다.

금액은 사용자가 빠르게 선택하거나 평균값을 수정하게 한다.

### 알림에 답장하기

사용자가 앱을 찾아오게 하지 않는다.

적절한 순간에 질문을 보내고,

사용자는 예·아니오로만 답한다.

단, 알림은 드물게 보내야 한다.

### 다음날 회고

실시간 기록을 놓쳤다면 다음날 한 번에 돌아보게 한다.

기록을 놓쳐도 불이익은 없다.

### 위시리스트에서 ‘안 삼’으로 전환

사고 싶은 물건을 먼저 저장하고,

사지 않기로 결정했을 때 참은 행동으로 전환한다.

이 방식은 실제 욕구가 있었음을 보여주기 때문에 신뢰도가 높다.

---

## 장기 방향

직접 기록과 자동 추정은 둘 중 하나를 영원히 선택하는 문제가 아니다.

시간에 따라 발전시킨다.

### Stage 1. 직접 기록

사용자가 직접 선택한다.

우리는 어떤 순간과 카테고리에서 참는지 학습한다.

### Stage 2. 감지 후 확인

앱이 가능성이 높은 순간을 먼저 제안한다.

사용자가 한 번 확인한다.

### Stage 3. 자동 처리 비중 확대

데이터가 충분히 쌓이면 자동 처리를 늘린다.

다만 마지막 확인 선택은 남긴다.

---

## 끝까지 남겨야 할 한 번의 선택

> 참은 걸로 기록할까요?  
> 예 / 아니오

이 한 번의 선택은 두 가지를 지킨다.

1. 사용자가 해냈다는 성취감
2. 자동 추정 오류를 사용자가 직접 고칠 수 있는 통제감

완전 자동화가 이 선택을 없애면,

서비스는 다시 평범한 소비 분석 앱에 가까워진다.
