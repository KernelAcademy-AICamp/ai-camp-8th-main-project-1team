## 1. 상호명 = "실존하는 가게 이름" (핵심 원칙)

카드 명세서에 실제로 찍히는 상호를 쓴다. **실존하지 않는 합성 라벨 금지**(교정 전 오류: "코레일 KTX",
"한국도로공사 하이패스", "SK주유소" 등 — 실제로 그렇게 결제·표기되지 않음).

### 소스 A — 서울 일반음식점 인허가(공개데이터, 영업중 12만)
실 상호·주소·동·좌표. 소상공인 개인식당·카페·주점의 **주 샘플링 풀**. 업태→category2 매핑으로
한식·분식·중식·일식·양식·고기구이·치킨·주점·횟집·뷔페·아시안 등에 편입(최종 **17,055 기본상호**).
예: `무등산`·`스시 스이신`·`로티보이`·`찐심맥주`.

**★ 위치 중립화(중요).** 서울 실상호에는 `로티보이 종로5가점`·`이자카야 우규 역삼점`처럼 **서울
지역명+점** 접미사가 박혀 있다. 합성 페르소나는 타 지역(가상 동)에 살 수 있으므로 이 접미사를 그대로
두면 모순이다("부산 사는데 종로5가점"). 그래서 **지점 접미사를 제거해 기본상호만 저장**한다(`재니생삼집
건대본점`→`재니생삼집`, `지티에스 버거 문래점`→`지티에스 버거`). 브랜드명과 일치하면 드롭(브랜드 경로가
처리), 무공백 잔여 지점도 드롭 → **잔여 "지역명+점" 0**. **생성기(Stage C)가 가상 사용자의 동에 맞춰**
프랜차이즈에는 `{가상동}점`을 붙이고 소상공인은 상호만 둔다(위치는 생성 시점에 결정).

### 소스 B — 국회의원 정치자금 지출내역(KAPF, 14.6만 건, 23,478 고유 사용처)
카드 명세서 `사용처`가 **실제 표기 포맷 그대로**다. 여기서 실 명세서 포맷을 학습·편입:
- **주유** = `여명주유소`·`버드나룻길주유소`·`중앙주유소`(동네명+주유소)·`현대오일뱅크(주)여의도현대` — "SK에너지" 같은 정유사 일반명이 아니라 **실 주유소 상호**.
- **철도** = `한국철도공사`·`코레일`·`KTX(한국철도공사)` (❌ "코레일 KTX").
- **택시** = `티머니택시`·`개인택시`·`이비카드`. **항공** = `대한항공`·`아시아나항공`·`에어부산`.
- **통신** = `(주)케이티`·`SK텔레콤`·`LG유플러스`(법인명 변형 다수). **식당** = 실 상호(`남도마루`·`소호정`).

### 소스 C — 실 브랜드 + 지점 합성 + 웹 큐레이션
편의점·카페·패스트푸드·치킨·피자·마트·영화·놀이공원 등 **실 브랜드 232종**(`merchants_brand.json`).
프랜차이즈는 **브랜드 + 동 + "점"** 합성(예: `스타벅스 역삼점`) — KAPF/서울 실데이터에 동일 패턴
관측(`카페드롭탑서여의도점`·`뚜레쥬르국회`·`교촌치킨 암사1호점`)으로 실증. 온라인 이커머스는 부가통신
파일이 통신사업자라 커버 못 하므로 **웹 큐레이션으로 별도**(쿠팡·배민·간편결제·구독).

### 실 명세서 표기 변형(forms) — ML 견고성
실 명세서는 표기가 지저분하다: `파리바게뜨`↔`파리바케뜨`, `CU`↔`씨유`, 법인명(`(주)스타벅스코리아`),
영문(`NETFLIX.COM`·`GOOGLE*YOUTUBEPREMIUM`), 배민=`우아한형제들`. 각 브랜드에 `forms`로 변형을 두어
생성기가 섞어 쓴다 → ML이 **정확한 상호 문자열에 의존하지 않도록**(과적합 방지).

## 1-B. 전국 위치 데이터 (regions.json) — 지점명·동선의 실좌표 기반

프랜차이즈 지점을 **가상 사용자의 거주지에 맞춰** 생성하려면(부산 사용자→`스타벅스 해운대점`) 전국 행정구역이
필요하다. 서울 동 300개만으로는 타 지역 사용자를 못 만든다. 그래서 **전국 3,495개 행정동 + 실 중심좌표 +
인구 가중**을 확보했다(웹 공개 데이터 자동 수집).

- **행정구역·좌표**: [raqoon886/Local_HangJeongDong](https://github.com/raqoon886/Local_HangJeongDong)의 17개 시도 행정동 경계(geojson)에서
  **각 행정동 실 중심좌표(WGS84)를 산출**(경계 정점 평균). 시도/시군구 명칭 동봉. 보조 출처(시군구·시도 중심점·읍면동 매핑):
  [datainworld/administrative_district](https://github.com/datainworld/administrative_district) — `reference/geo/*.csv`로 보존.
- **인구 가중(`weight`)**: 시도 실인구(2026 근사)를 시도 내 행정동에 배분(합=1). **수도권(서울·경기·인천) 비중 0.508** —
  실제 인구 집중과 정합. 생성기는 `weight`로 사용자 거주 동을 뽑는다(대도시 밀집).
- **필요 없는 것**: 건물 단위 도로명주소 DB(수백만 건). 지점은 **동 중심좌표 + 지터**로 배치하면 충분하다.
- **생성기 사용(Stage C)**: `weight`로 거주 동 추출 → `(lat,lon)` 앵커에서 지터로 하루활동·POI 배치 → 프랜차이즈는
  그 `dong`으로 `{동}점` 합성, 소상공인은 상호만. 실좌표 상점 풀은 서울뿐이라 타 지역은 상호(위치중립) 재사용 + 지역 좌표 배치.

## 4-B. 취미 성향 신호 (hobbies.json) — 마이데이터로 성향 식별

일상 지출(식당·편의점·교통)만으로는 사람의 **성향**이 안 드러난다. 그래서 **취미성 지출**을 전용 카테고리로
두고, **가끔이지만 명백히** 나타나게 해 마이데이터에서 취향이 읽히게 한다(공연 티켓이 반복되면 문화형,
사료·동물병원이 보이면 반려인).

- **취미 전용 category2 9종 추가**: 공연전시·여행숙박·반려동물·디지털가전·스포츠레저·아웃도어캠핑·서점문구·
  키즈육아·취미공예(실 브랜드·실 상품·2026 가격). 방문빈도는 **낮게**(0.08~0.20) 둬 '가끔'을 표현.
- **취미 taxonomy 12종**(`hobbies.json`): 각 취미 → 그것이 드러나는 `signatureCategories`. 예:
  `여행 → [여행숙박·항공·철도·렌터카·고속버스]`, `디지털게임 → [디지털가전·간편결제·스트리밍]`,
  `반려동물 → [반려동물]`, `문화공연 → [공연전시]`.
- **생성 조건**(`mydata.generation.hobby`): 사용자당 취미 `1~3`개 배정, 취미별 월 지출 `0.5~3.0`건(가끔이지만
  뚜렷). 생성기는 그 취미의 signatureCategories에서 지출을 주입 → 일상과 구분되는 성향 신호.
- 이는 낭비/필수 라벨(취미=대개 재량)과 개인화(통념상 낭비지만 본인 취미면 보호)에도 직접 쓰인다.

## 5. 재량성 점수 — 재량 ≠ 낭비 (무대 판정용, 낭비성 아님)

각 품목 `재량성(0..1)` = **"생존필수인가?"** 척도(식료품·약·공과금·교통↓, 주류·취미·놀이공원↑). **★ 낭비성향이
아니다** — 재량성은 **무대(필수/재량) 판정·금액 분포에만** 쓰고, **낭비 확률에 직접 넣지 않는다**(사용자 확정 2026-07-22,
재량≠낭비). 낭비는 재량 무대 '안에서' **충동·과다·후회성**(심야·계획안됨·평소대비과다·빈도급증·구독누수·배달과다·치팅)으로만
발생하며, **본인 취미의 비과다 지출은 보호**(비낭비). 따라서 취미 상품의 높은 재량성(공연·캠핑 등)은 낭비로 직결되지 않는다.
라벨러: 필수 무대 `p_waste≈0.02` / 재량 무대 `p=clamp(impulse × 페르소나 충동성 × 곡선(t))`, 취미면 `impulse×hobbyProtection`.
조건은 `mydata.generation.label`(생존필수 목록·충동성 요인·취미 보호). 단일 규칙 금지(순환 방지, W8·launch_plan W1-0 ③).

## 6. 다양성(ML 실패 방지)

각 사용자가 "항상 같은 곳에서 같은 것만" 사지 않도록: 소상공인 category2당 **최대 1,600 실상호** +
동별 좌표 앵커 + 브랜드 지점 합성 → 조합 다양성 폭발. 상품도 category2당 수십 종. 하루활동 모델이
빈도·근접·페르소나로 가중 추출 → 개연성 있는 변주.

## 7. 재구축 이력 (2026-07-22)

**교정 전 문제**(사용자 지적): ① 실존하지 않는 결제명칭(합성 라벨) ② 목록이 너무 적음(가맹점 27맥락×소수·
품목 149) → "항상 같은 곳/같은 것" → ML 실패 위험 ③ 있어야 할 category 누락 ④ 부실한 품목·가격.
**교정**: 국회 지출(KAPF) 실 사용처 + 서울 실상호를 대량 편입(상호 120→17,055), 실 명세서 표기 forms 도입,
상품 149→484(43 category2), 가짜 합성 라벨 제거, fares 표기 포맷 메모 추가.

## 8. 데이터 계보·규제
