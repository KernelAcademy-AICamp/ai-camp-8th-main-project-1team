# Part I — 구현 방향성 & 확정 사항

## 1. 프로젝트 한 줄 정의

**"결심을 도와주는 개인 금융 비서"**
사용자가 이미 알고 있지만 실천하지 못하는 소비 습관을, 소비 데이터 분석 + AI 코멘트로
"인지 → 결심 → 실행 확인 → 보상(축하)"까지 연결하는 서비스.

- 결제/송금 등 라이선스가 필요한 기능은 완전히 배제
- 핵심은 **분석 로직(코드)** 이고, AI는 그 결과를 **사람이 이해하기 쉬운 문장으로 표현**하는 역할만 담당

## 2. 스코프 확정

### 포함 (In Scope)

| 우선순위 | 기능 | 비고 |
|---|---|---|
| 주인공 | 이상거래 탐지(FDS) | 유일한 차별화 지점. 개발 순서도 선행 (→ D-07) |
| 필수 | 소비 패턴 분석 → 금융상품 Top3 추천 | 레드오션이나 RFP 필수 |
| 필수 | 초개인화 절약 리포트 자동 생성 | 레드오션이나 RFP 필수 |
| 부가 | 소비건전성지수, 대안신용평가 개념 반영 | 건전성지수로 갈음 |
| 부가 | 목표기간 설정 → 저축 목표금액 산출 → 축하 알림 | |

> **순위가 뒤집힌 이유**: 최초 기획서는 추천을 1순위, FDS를 부록으로 뒀다. 시장 리서치가 이를 반증했다 —
> 조회·추천은 레드오션이고 그 시장에서 회사들이 실제로 죽었다 (브로콜리 종료, 뱅크샐러드 MAU 40% 급감).
> 반면 FDS+감사로그는 ISMS-P·전자금융거래법이 직접 뒷받침한다. 상세 → **Part III-C, III-D**.

### 제외 (Out of Scope) — 규제 리스크 차단

- 실제 결제/송금 기능
- 실제 인증(본인인증, 계좌연동), 마이데이터 API 직접연동 — **개인에게 경로가 아예 없다** (→ Part III-B §4~5)
- **실재하는 금융상품 데이터** — `FinancialProduct`는 **전부 더미**로 간다

> ★ **규제 방어 논거 (발표에서 이 문장을 쓴다)**
> 이 프로젝트가 규제 대상이 아닌 이유는 "정보제공 수준이라서"가 **아니다.**
> **실재하지 않는 더미 상품을 쓰고, 실제 계약이 성립하지 않으며, 어떤 수수료도 발생하지 않아 '영업'이 아니기 때문**이다.
> 동일 로직을 실제 상품에 적용하는 순간 금소법상 중개업 등록 대상이 된다 — 이 경계를 아는 것 자체가 기술 역량의 증거다.
>
> "정보제공이라 괜찮습니다"라고 말하면 아는 심사자에게 정확히 반박당한다 (→ Part II D-04, Part III-B §2).

### 개인정보 수집 — 포함 (2026-07-20 확정)

> **확정**: 더미 데이터만으로는 사용자 테스트와 소비건전성지수의 "실제 지출 확인"이 성립하지 않으므로,
> **사용자가 자발적으로 입력하는 소비내역은 실제로 수집한다.**

단, **D-01의 2단 일정 구조에 맞춰 배치한다** — 8/7 시연을 인질로 잡지 않기 위해서다:

| 구간 | 데이터 |
|---|---|
| **~8/7 (D+18)** | `DUMMY_SEED`만으로 시연. 수집 기능 **미노출** |
| **8/8~8/25 (D+36)** | `USER_INPUT` 수집 화면 + 동의 플로우 투입 |

수집 설계 상세는 **§5-3**. 인증이 필요한 정보(실명, 계좌번호, 카드번호, 주민번호)는 **입력 필드 자체를 두지 않는다.**

> **RFP `D26`과의 관계 (2026-07-20 확정)**
> RFP 3순위 기능의 필수 수행 규칙은 *"실제 인증이나 개인정보를 요구하지 않고, 가상의 시나리오를 통해 위험을 경고하는 수준으로 기획함"*이다.
> **이 조항의 취지는 FDS를 위해 실제 본인인증·계좌연동을 구축하는 것이 고난이도이므로 요구하지 않는다는 것**이며,
> 본 프로젝트는 애초에 **실제 인증 방식을 쓰지 않고 우회 설계(익명 계정 + 사용자 자발 입력 + 더미 시드)로 구현**하므로 이미 충족된다.
> 수집 항목도 실명·계좌·카드·주민번호를 전부 배제한 4개 필드로 한정했다 (§5-3).
> FDS는 규칙대로 **가상 시나리오 기반 경고 수준**을 유지하며, 실제 거래 차단·인증 개입은 하지 않는다.

## 4. 핵심 설계 원칙 (변경 불가 원칙)

### 원칙 1. 판단은 코드가, 표현은 AI가 한다 — **설명가능성 규제 대응**

- `AnalysisEngine`이 통계적으로 계산한 숫자는 AI가 절대 변경하지 않는다.
- AI(Gemini)는 이 숫자를 사람이 이해하기 쉬운 문장으로 "번역"하는 역할만 한다.

> ⚠️ **프레이밍 주의 (→ D-06)**: 이것을 **독창적 설계로 주장하면 안 된다.**
> 뱅크샐러드의 카드 추천이 이미 룰 기반 결정론 엔진이고, 그렇게 하는 **이유가 규제에 있다** —
> 알고리즘 사전 검증 의무 때문에 블랙박스 ML보다 설명 가능한 룰 엔진이 유리하다.
>
> - ❌ "저희는 AI 할루시네이션을 막는 독창적 설계를 했습니다"
> - ✅ **"금융권이 설명가능성 규제 때문에 룰 엔진을 쓰는 이유를 이해하고 같은 구조를 택했습니다. 판단은 재현 가능해야 감사받을 수 있기 때문입니다"**
>
> 업계 표준을 따랐다는 것은 약점이 아니라 도메인 이해의 증거다.

### 원칙 2. 하나의 분석 결과를 세 갈래로 재사용한다

```
AnalysisEngine.analyze(userId)
  → AnalysisResult {
       카테고리별 평균/중앙값, 성향점수,
       overspendingCategories,       // 카테고리별 지출비중 > 30% → 과소비 카테고리 목록
       shortTermDeviationScore,      // Modified Z-score (단기·FDS 전용)
       longTermVolatilityIndex,      // 전체 기간 지출 변동성 CV (추천·건전성지수용, 장기)
       dataSourceMode,               // ESTIMATED | CONFIRMED
       userInputCount
    }
     ├─ AlertService     : shortTermDeviationScore + 룰 AND 결합 → 이상소비 경고   ← 주인공
     ├─ RecommendService : 성향점수 + overspendingCategories + longTermVolatilityIndex → 상품 Top3
     └─ ReportService    : overspendingCategories → 잘한소비/과한소비 분류
```

- 세 서비스가 각자 계산하지 않고 동일한 분석 엔진 결과를 재사용 → 로직 일관성 보장, 중복 계산 방지.
- **이 원칙이 일정에도 적용된다 (→ D-02)**: 공유 엔진을 먼저 100% 완성하면 나머지 셋은 각각 하루치 작업이다.
  엔진이 안 되면 셋 다 안 된다. **"세 기능 얇게"가 성립하는 조건은 엔진이 두껍다는 것.**

**지표 분리 원칙**: "표준편차 이탈"이라는 이름 하나로 뭉뚱그리지 않는다.
`shortTermDeviationScore`(이번 달 이상거래 탐지)와 `longTermVolatilityIndex`(평소 소비 안정성)는
**계산 윈도우와 목적이 다르므로 같은 변수로 취급하지 않는다.**

**계산 윈도우 확정 (2026-07-20)**

| 지표 | 평가 대상 | 기준 분포 | 최소 표본 |
|---|---|---|---|
| `shortTermDeviationScore` | **최근 1개월**의 개별 거래 | **직전 3개월** 동일 카테고리의 `median`·`MAD` | 해당 카테고리 **10건** |
| `longTermVolatilityIndex` | — | **보유 전체 기간(최대 6개월)** 월별 총지출의 CV | **3개월** |

- 두 지표는 **겹치는 구간을 공유하지 않는다** — 단기는 "최근 1개월 vs 직전 3개월"로 평가 대상과 기준 분포를 분리하고,
  장기는 전체 기간을 한 덩어리로 본다. 최근 1개월이 기준 분포에 섞이면 이상치가 자기 기준을 끌어올려 탐지가 무뎌진다.
- **페르소나 더미는 전부 6개월치로 통일한다.** 3~6개월로 들쭉날쭉하면 장기 지표가 페르소나마다 다른 윈도우로 계산되어 비교가 깨진다.
- 최소 표본 미달 시 해당 지표는 산출하지 않고 `dataSourceMode = ESTIMATED`로 응답한다 (§5-2 ③).

### 원칙 3. 재현성(reproducibility) 검증 필수

- 같은 입력 → 항상 같은 출력이 나오는지 단위테스트로 확인한다.
- 선택 보강 지표 중 하나다 (→ D-10). 사용자 수가 0명이어도 측정되므로, 여건이 되면 우선 넣을 값어치가 있다.

| Report | 생성된 리포트 저장 (재조회 시 재계산 방지) |
| Alert | 이상소비 감지 이력 |
| **AuditLog** | **prev_hash, entry_hash, payload, created_at** (→ §5-4) |
| **AuditBatch** | **batch_root, prev_batch_root, tsa_response(.tsr), anchored_at** (→ §5-4) |

> `Consumption.isPlanned`는 소비건전성지수의 "계획소비 비율" 계산 근거다.
> 최초 설계에는 "계획된 소비인지 충동소비인지" 구분 필드가 없었다.

### 계산 로직 확정 (수식 → 코드 순서, 절대 역순 금지)

#### ① 이상소비 판단 기준 (= `shortTermDeviationScore`, 단기·FDS 전용)

```
shortTermDeviationScore
  = 0.6745 × (log(amount) − median(log(amount))) / MAD
  임계 3.5 초과 → 후보 플래그

최종 경고 = z-score 플래그 AND 룰 1개 이상 일치
  룰 ① 심야 고액        ② 신규 카테고리 급증        ③ 빈도 이탈

캘리브레이션: 더미 100건 중 경고가 3~5건만 뜨도록 역산
             (Phase 0에서 임시 시드로 잠정 확정 → 페르소나 확정 후 재캘리브레이션, D-09)
```

> **왜 단순 2σ가 아닌가 (→ D-11, Part III-D §7)**
> - **2σ로 잡으면 정규분포 가정 하에 정상 소비의 약 5%가 경고로 뜬다.** 소비 변동이 큰 세그먼트라면 더 뜬다.
>   → 데모에서 경고가 도배되면 *"이거 그냥 다 잡는 거 아니냐"*는 질문을 받고, **발표 주인공 기능이 그 자리에서 무너진다.**
> - **금액은 로그정규 분포에 가깝다** → raw z-score는 부적절. `log(amount)`에 건다.
> - 평균·표준편차 대신 **median·MAD**를 쓰는 이유: 이상치 자체에 통계량이 오염되지 않는다(강건성).
> - **단일 조건보다 AND 결합** — 금감원 공개 시나리오가 4개 조건을 AND로 묶는 구조다 (Part III-D §3-1).
> - 임계치는 "최적화"가 아니라 **"제약 충족"**으로 정한다 — 더미 100건 중 3~5건 역산.
>
> 오탐(false positive) 비용이 미탐 비용보다 크다는 것이 업계 통계의 일관된 방향이다.
> 단, 인용되는 구체적 배수(예: "오거절이 사기의 9배")는 **대부분 벤더 자료**이므로 방향성만 취하고 수치는 인용하지 않는다.

#### ② 과소비 기준 (= `overspendingCategories`, ReportService/RecommendService 공용)

```
카테고리별 지출 ÷ 전체 지출 > 30%  → 과소비 표시   (2026-07 확정)
```
> 한 카테고리가 전체 지출의 30%를 넘으면 "쏠림"으로 보는 실무적 기준선. Phase 0에서 조정 가능.
> 이 결과는 `ReportService`(잘한소비/과한소비 분류)와 `RecommendService`(타겟카테고리 매칭)가
> **동일하게 재사용**한다. 두 서비스가 각자 다시 계산하지 않는다.

#### ③ 소비건전성지수 (A/B/C/D 등급, 0~100점)

```
소비건전성점수 = 100 × (0.4 × 저축진행률 + 0.3 × 안정성 + 0.3 × 계획소비비율)

저축진행률   = min(1, 누적저축액 / (목표금액 × 경과기간/목표기간))
안정성       = clamp(1 − CV / 0.6, 0, 1)     # CV = longTermVolatilityIndex = 표준편차/평균
계획소비비율 = 계획소비 금액 / 전체 지출 금액

등급 컷오프: A 85점 이상 / B 70~84 / C 50~69 / D 49점 이하
```

> **가중치 근거**: 저축진행률에 최대 가중치(0.4)를 두는 이유는 서비스 정의(§1)가 "결심의 실행 확인"이므로
> 목표 달성도가 1순위이기 때문이다. 안정성·계획소비비율은 동일 비중(0.3)으로 시작한다.
>
> **범위 검증**: 세 요소가 모두 0~1이므로 점수는 항상 0~100. A~D 전 구간 도달 가능함을 확인했다
> (예: a=1.0, b=0.8, c=0.8 → 88점 = A / a=0.3, b=0.3, c=0.5 → 36점 = D).
>
> 이 산식은 **개선안에서 `f()`인 채로 누락되어 있던 항목**이다. 컷오프만 있고 산식이 없으면 무의미하므로 확정했다.

> **확정 원칙**: 위 수식들은 코드 작성 전 **Phase 0에서 손으로 먼저 확정**한다.
> 나중에 왔다갔다 하지 않도록 계수/임계치까지 숫자로 못박아 놓는다.

### 5-1. 추천 매칭 로직 (사용자 조건 × 상품 속성) — RFP 필수 산출물

> RFP 요구: "사용자의 소비 패턴, 예산, 선호도를 조건으로 추출한 뒤, 금융 상품의 속성과 매칭하여 Top3 상품을 비교 추천"

#### ① 사용자 조건 (AnalysisResult에서 그대로 재사용 — 별도 재계산 금지)

| 조건 | 산출 방식 | AnalysisResult 필드 |
|---|---|---|
| 소비 패턴(고지출 카테고리) | 과소비 기준(§5 ②)과 동일 로직의 결과를 그대로 사용 | `overspendingCategories` |
| 예산(가용 여유자금) | **월소득 − 월평균지출**(음수면 0). *(초기 서술 "목표 저축액 대비 절약 가능분"을 구현과 정합하도록 정정, 2026-07-21)* | (User.monthlyIncome + AnalysisResult 조합, `RecommendService.availableFunds`) |
| 선호도 - 리스크 성향 | 장기 소비 변동성 지표로 대체 산출 (낮음→안정형, 높음→공격형) | `longTermVolatilityIndex` |
| 선호도 - 목표기간 | User.목표기간 | — |

> ⚠️ **모델링 가정(프록시)의 한계**: "리스크 성향"을 실제 투자성향 설문 없이 `longTermVolatilityIndex`(소비의 규칙성)로
> 대체하는 것은 편의상의 프록시이며, **"지출이 불규칙하다"가 "투자 리스크를 감수한다"를 의미하지는 않는다.**
> 더미 데이터 기반 MVP 단계이므로 채택하되, README/리스크 평가 기준 명세서에 이 한계를 명시적으로 기록한다.
> 이 한계를 스스로 밝히는 것이 심사자에게는 오히려 가점 요인이다.

#### ② 상품 속성 (FinancialProduct 더미 데이터, 최소 5개)

| 속성 | 예시 값 |
|---|---|
| 상품유형 | 예금 / 적금 / 펀드 / 카드(캐시백형) |
| 최소가입금액 / 최소기간 | 예: 10만원 이상, 6개월 이상 |
| 예상 수익률/혜택률 | 예: 연 3.5%, 카페 카테고리 캐시백 5% |
| 리스크 등급 | 안정형 / 중립형 / 공격형 |
| 타겟 카테고리 | 특정 소비 카테고리 특화 여부 (예: 카페·쇼핑 특화 카드), 범용이면 null |

> 더미 설계 근거: **마이데이터 표준 API 규격 문서는 로그인 없이 공개 열람 가능**하다.
> "표준 API 규격을 읽고 그 스펙 모양으로 더미를 설계했다"고 쓰는 편이,
> 접근 불가능한 것을 연동하겠다고 쓰는 것보다 훨씬 강하다 (→ Part III-B §6-1).

#### ③ 매칭 점수 산식 (확정)

```
[1단계 — 게이팅 필터] 다음 중 하나라도 걸리면 후보에서 제외
  · 최소가입금액 > 사용자 가용 여유자금
  · 사용자 목표기간 < 상품 최소기간          ← 가입 자체가 불가능

[2단계 — 매칭점수] (게이팅 통과 상품만)
매칭점수 = 0.3 × 목표기간 부합도
         + 0.3 × 리스크등급 부합도
         + 0.4 × 타겟카테고리 부합도

→ 상위 3개를 Top3로 추천
```

**정규화 방식 (확정)**

```
목표기간 부합도   = 상품 최소기간 / 사용자 목표기간          (게이팅 통과 시 0~1]

리스크등급 부합도 = 등급 일치 1.0 / 인접 등급 0.5 / 정반대(안정형-공격형) 0

타겟카테고리 부합도
  상품 타겟카테고리 ∈ overspendingCategories        → 1.0
  타겟카테고리 없는 범용상품(예금/적금)              → 0.5   (중립)
  사용자 지출 상위 3위 이내이나 30% 미만             → 0.5
  사용자 소비와 무관한 타겟카테고리                   → 0.0
```

> **개선안 대비 수정 3건 — 그대로 두면 산식이 작동하지 않는다**
>
> **(1) 목표기간 부합도**: 원안 `1 − |사용자목표 − 상품최소| / max(두 값)`은 **"최소"기간의 의미를 무시한다.**
> 실측하면 24개월 목표 사용자가 6개월 최소 상품에서 0.25점(가입 가능한데 최하점)인 반면,
> 6개월 목표 사용자가 12개월 최소 상품에서 0.50점(**가입 불가인데 더 높은 점수**)이 나온다.
> → 불가 케이스를 게이팅으로 먼저 제거한 뒤 비율식을 쓴다.
>
> **(2) 타겟카테고리 부합도**: 원안의 0/1 이진이면 타겟카테고리가 없는 **예금·적금이 항상 0점**이 되어
> 배점의 40%가 통째로 죽고 Top3가 캐시백 카드로만 채워진다. `overspendingCategories`가 빈 사용자도 전 상품 0점.
> → 범용상품에 중립값 0.5를 부여해 구조적 불리를 제거한다.
>
> **(3) 게이팅 폴백**: 여유자금이 적으면 후보가 3개 미만이라 Top3를 못 채운다.
> ```
> 게이팅 통과 < 3개 → 최소가입금액 오름차순으로 부족분 보충
>                     응답에 gatingRelaxed=true + "여유자금 대비 최소가입금액이 높습니다" 근거 노출
> ESTIMATED 모드   → 게이팅 미적용 (가용자금이 추정치라 오작동). 최소가입금액을 근거필드로만 노출
> ```

> **재사용 준수 확인**: w2·w3의 입력값(`longTermVolatilityIndex`, `overspendingCategories`)은
> `RecommendService`가 별도로 재계산하지 않고 `AnalysisResult`에서 그대로 받는다 — **원칙 2의 준수 여부를 좌우한다.**
> 위 가중치·정규화 방식은 Phase 0에서 더미 데이터로 결과를 눈으로 확인한 뒤 조정 가능한 **출발값**이다.
>
> 이 표가 곧 RFP 필수 산출물인 **"추천 매칭 로직표"**의 원본이 된다.

### 5-2. 더미 시드 vs 사용자 입력 데이터 처리 원칙

> 더미 데이터는 "영구히 쓰는 가짜 데이터"가 아니라 **1단계 엔진 검증용 시드**이고,
> 2단계부터는 사용자의 수동 입력(`USER_INPUT`)이 쌓이며 분석을 보완·개선한다.

#### ① 단계별 흐름

```
1단계 (~8/7, 엔진 개발/검증/시연)
  → 페르소나 기반 DUMMY_SEED만 사용. 재현성 검증(원칙 3)도 이 데이터로 수행
  → 실사용자 계정과 분리된 테스트 계정에만 귀속 (실서비스 화면에 노출되지 않음)

2단계 (8/8~8/25, 수집 개시)
  → 신규 가입 사용자는 USER_INPUT만 누적
  → 데이터가 충분해지기 전까지 "추정 모드", 충분해지면 "확정 모드"로 전환
```

#### ② `AnalysisResult`에 데이터 출처 정보 반영

```
AnalysisResult {
  ...
  dataSourceMode,   // ESTIMATED(데이터 부족, 참고용) | CONFIRMED(임계치 충족, 실제 지표)
  userInputCount    // USER_INPUT 소비 건수 — 임계치 판단 및 화면 표시용
}
```
- `Report`, `Alert`, 추천 결과 화면 **모두** `dataSourceMode`를 함께 내려준다.
  "아직 참고용 추정치입니다" vs "실제 소비 데이터 기반입니다"를 구분해 보여준다.
- 소비건전성지수의 "저축진행률" 요소는 `USER_INPUT` 누적 전에는 목표금액/기간 기반 **추정치**로,
  누적 이후에는 실제 지출 기록 기반 **확정치**로 전환된다 — 이것이 "실제 지출 확인을 지수에 반영"하는 지점.

#### ③ 데이터 부족 시 예외처리

- `USER_INPUT` 건수가 임계치 미만이면 `AnalysisEngine`은 평균/MAD 계산을 시도하지 않고
  `dataSourceMode = ESTIMATED`로 응답 → 화면에는 에러가 아니라
  *"분석에 필요한 데이터가 아직 부족합니다. N건 더 기록하면 정확한 리포트를 받아보실 수 있어요"* 형태로 안내한다.
- 이 기간에도 화면이 비어 보이지 않도록, `User` 입력값 기반의 단순 추정 리포트
  (예: "목표 달성까지 필요한 월 저축액")를 `ESTIMATED`로 표시해 제공한다.
**`ESTIMATED` → `CONFIRMED` 전환 임계치 (2026-07-20 확정)**

```
CONFIRMED 조건 = (USER_INPUT 총 30건 이상)  AND  (최초 기록일로부터 14일 경과)

카테고리별 지표는 해당 카테고리 10건 이상일 때만 개별 산출
  → 미달 카테고리는 "데이터 부족"으로 표시하고 전체 지표에서 제외
```
> **근거**: ① MAD 기반 통계는 표본이 극소수면 중앙값이 한두 건에 좌우된다 — 카테고리당 10건이 하한선이다.
> ② 건수만 보면 하루에 30건을 몰아 입력해 요일·주기 패턴이 전혀 없는 데이터로도 `CONFIRMED`가 되므로 **기간 조건(14일)을 AND로 건다.**
> ③ 14일은 주간 소비 주기를 최소 2회 관측하는 길이다.
> ④ FDS 캘리브레이션 기준이 "더미 100건 중 3~5건"이므로, 30건은 경고가 1건 내외로 뜨는 규모다 — 화면이 비지도 도배되지도 않는다.
>
> 이 값은 Phase 0에서 더미 데이터로 확인한 뒤 조정 가능한 출발값이다.

#### ④ API 영향

- `POST /api/consumption`은 실사용자 입력 전용(`source=USER_INPUT` 고정).
- 더미 시드 삽입은 **개발/관리자 전용 경로로 분리** — 일반 API로는 `DUMMY_SEED`가 생성되지 않는다.

### 5-4. 감사로그 3계층 설계

> 선형 해시체인만으로는 **"운영자가 DB를 통째로 재생성하는 공격"을 막지 못한다.**
> 체인을 처음부터 다시 계산하면 완벽하게 일관된 가짜 체인이 만들어지고, 검증해도 통과한다.
> **루트 해시는 운영자가 닿을 수 없는 곳에 게시되어야만 의미가 있다.**

```
[계층 1] 로그 엔트리 — prev_hash 선형 체인
   entry_hash(n) = SHA256(prev_hash || canonical_json(entry_n))
   → 단일 엔트리 변조 즉시 탐지, 쓰기 비용 O(1)

[계층 2] 배치 — Merkle 루트
   batch_root = MerkleRoot(entry_hash[i..j])
   → 개별 엔트리의 O(log n) 포함 증명(inclusion proof)

[계층 3] 앵커링 — RFC 3161 TSA에 batch_root 타임스탬프   ← 핵심 차별화
   FreeTSA: https://freetsa.org/tsr  (무료)
```

| 위협 | 방어 계층 |
|---|---|
| 단일 로그 행 수정/삭제 | 계층 1 (체인 불일치) |
| 특정 엔트리 존재 증명 요구 | 계층 2 (포함 증명, O(log n)) |
| **DB 전체 재생성 (내부자/운영자)** | **계층 3만이 막을 수 있음** |
| 과거 시점 위조 | 계층 3 — TSA의 `genTime` |

**구현 함정 — 하나라도 어기면 검증이 조용히 깨진다 (→ Part III-D §5)**

1. **정규화 필수** — `json.dumps(obj, sort_keys=True, separators=(",", ":"), ensure_ascii=False)`
2. **도메인 분리** — 리프 `sha256(b"\x00" + data)`, 노드 `sha256(b"\x01" + l + r)` (RFC 6962). 안 하면 트리 구조를 속일 수 있다
3. **TSA 요청 간 15초 이상 지연** → 매 로그가 아니라 **배치** 앵커링이어야 하는 실무적 이유
4. **TSA 인증서·CA 체인(`tsa.crt`, `cacert.pem`)을 앵커와 함께 보관** — 10년 뒤 웹에서 못 구한다
5. **앵커를 로그 DB와 물리적으로 분리 보관** — 앵커를 잃으면 전부 무의미

**시연 각본 (가장 시각적으로 강한 데모)**
DB에서 `Alert` 한 행의 `risk_score`를 직접 UPDATE로 바꾼 뒤 검증을 돌려,
**어느 엔트리에서 체인이 깨지는지 화면에 보여준다.**

발표 문장: **"해시체인만으로는 내부자의 전체 재작성을 막을 수 없어서, 배치 루트를 외부 TSA에 앵커링했습니다."**

## 5-5. 게임화 저축 루프 — 가상 포인트 · "살 뻔했다" (2026-07-21 방향 전환)

### 문제 (사용자 정의)
감정·보상 심리로 **충동 소비**가 생기고 → 카드 결제일에야 규모를 **뒤늦게** 인지하고 → 소비 내역을 봐도
**뭘 줄여야 할지 모른다**("자잘한데 모으니 왜 많지?"). 근본 원인은 **"절약한 금액이 실제 자산 축적으로
연결됐다는 감각의 부재"** — 조회·조언만으로는 행동이 안 바뀐다(III-C §2 "조회만으로는 죽는다"와 정합).

### 통찰 & 전환
수동적 조언자 → **능동적 저축 게임**. 참은 지출(충동)을 **눈에 보이는 '아낀 돈'**으로 만들어 적금(더미)으로
쌓아, 절약이 자산으로 연결되는 감각을 만든다. 시장 승자의 이동 방향("조회 → 자동 실행", III-C §5)과 같은 축이다.

### 핵심 메커니즘 (Phase 1 — 구현 대상)
1. **월급 → 포인트**: 월 소득을 그 달의 사용 가능 포인트로 본다.
2. **지출 → 차감**: 실제 소비가 포인트를 깎는다(잔여 포인트 = 이번 달 쓸 수 있는 돈).
3. **"살 뻔했다"**: 충동을 참은 지출을 기록하면 포인트가 빠지되, 실제로 안 썼으므로 같은 금액이
   **'아낀 돈(적금에 넣을 수 있는 금액)'**으로 적립된다. → 절약이 **행동**이 된다.
4. **적금 이체(가상)**: 아낀 돈을 적금(더미)으로 옮기면 이자까지 붙어 **자산으로 축적**된다.

### Qapital 벤치마크 (Phase 2·3 — ✅ 구현, 2026-07-21)
미국 Qapital의 **사용자 정의 무제한 목표 버킷**을 도입하고, 절약을 **즉시 자산화**하는 방향으로 단순화했다.
- **목표 버킷(SavingsGoal)**: 사용자가 이모지+이름+목표금액으로 **무제한** 생성(사진 대신 이모지 — 로컬·더미라 파일저장 회피).
- **살 뻔했다 → 즉시 랜덤 목표 입금**: 참은 순간 그 돈이 **바로 랜덤 목표 버킷에 자동 입금**된다. 대기 풀(savedPot)·
  수동 이체 과정은 없앴다(사용자 요청). "랜덤"은 재현성(§4) 위해 **입금 순번 회전**(depositCount % 목표수)으로 결정론적 분산 —
  진짜 난수는 시드 기반 교체 가능.
  - **획득 프레이밍(참는 순간 진척 번역)**: 입금과 동시에 **어느 목표가 몇 %→몇 %로, 며칠 앞당겨졌는지**(`GoalGain`)를 계산해
    스냅샷에 실어("✨ 참았어요! ✈️ 여행 가기 0%→1% · 🗓 1일 앞당겨졌어요") 보여준다 — "돈을 안 썼다(손실)"가 아니라
    "내 목표를 조금 샀다(획득)"로 인식을 바꾸는 **이 서비스의 심장**. `build` 오버로드로 시그니처를 안 깨고 스냅샷에만 추가.
- **저축 규칙(길티플레저·지출절감·행동연동)은 제거**(사용자 요청) — 절약 경로를 '살 뻔했다' 하나로 단순화.
- **소비 vs 살 뻔한 것 구분**: '소비했어요'는 실제 Consumption(필요/불필요)을 만들고, '살 뻔했다'는 목표 입금(DEPOSIT)을 만든다.
- **선물상자 시각화**: 저축하면 채워지고(반짝임), 필요없는 소비면 금이 가고, **예산 초과면 크게 깨지며** 잔액 큰 목표에서
  초과분을 **강제차감**(WITHDRAWAL)한다. 저축은 예산을 넘겨 못 하지만(마이너스 불가), 소비는 초과가 가능하다.
- **치팅데이 쿠폰(Phase 3)**: 참은 저축 누적이 임계치(10만)를 넘을 때마다, **가장 많이 참은 분류**로 자유이용권을 제안한다.
  사용자가 **쓸지/말지 선택** — 안 쓰면 축하·쓰면 응원(전부 가상 상징 보상).
- **회계**: `pointsRemaining=예산−이번달소비−이번달입금`, `goalBalance=ΣDEPOSIT→g−ΣWITHDRAWAL←g`. 저장 이벤트는
  DEPOSIT·WITHDRAWAL 둘뿐. 전부 가상 — 입금·강제차감은 화면상 이동일 뿐 실 송금·결제 아님(규제 안전 유지).

### 고민 목록 — 폴센트 응용 (✅ 구현, 2026-07-21)
가격 때문에 살까 말까 고민하는 상품을 담아두고, **결국 안 샀다**는 걸 확인해 그 금액을 아낀 돈으로 적립한다(안 산 걸 칭찬).
- **유입(사용자가 URL/스샷을 건넨다 — 화면 훔쳐보기 아님)**:
  - **URL**: 서버가 상품명·가격·이미지를 읽는다 — **OpenGraph 메타**뿐 아니라 페이지에 **임베드된 JSON 상태**
    (`__NEXT_DATA__`·`window.__PRELOADED_STATE__` 등)·**JSON-LD `offers`**에서도(요즘 SPA는 가격을 그쪽에 실음).
    할인가 키(`salePrice`→`finalPrice`→…→`price`)를 먼저 봐 정가·배송비 오탐을 줄인다. 임의 URL 페치라 **SSRF 방어**(내부망·로컬 차단).
    안티봇 사이트(쿠팡 등)는 페치가 막히거나 가격이 없을 수 있어 스크린샷/수동으로 폴백.
  - **스크린샷**: **Gemini 비전**으로 상품명·가격·카테고리 추출(키 없거나 실패 시 조용히 폴백 → 수동 입력).
  - **수동**: 항상 되는 최종 폴백.
- **조회는 추출만 하고 저장하지 않는다** — 화면에 채워 사용자가 확인·보정한 뒤 담는다(add).
- **"안 샀어요" → 즉시 아낀 돈으로 적립**({@code PointService.deposit} 재사용, 랜덤 목표 직입). 구체적 상품 회피는 이번 달
  예산과 무관한 실제 절약이라 **예산 캡을 걸지 않는다**(추상적 '살 뻔했다'는 예산 캡을 거는 것과 대비).
- **규제·법**: 실쇼핑몰(쿠팡 등) 직접 스크래핑은 **ToS·법적 리스크**로 하지 않는다 — 공개 메타가 없으면 못 채우고 수동 입력.
  실상품 데이터를 넣지 않는 원칙(설계원칙 5)과 정합. 전부 가상이며 실제 구매·결제가 아니다.
- **가격 API 조사 (2026-07-21, 정직 기록)**: 폴센트는 URL→상품ID→**쿠팡 파트너스(제휴) API**로 가격을 가져온다 —
  이는 제휴 가입+커미션 수익 모델이라 원칙5(비중개)와 충돌해 **채택 안 함**. 쿠팡 WING **판매자 API**
  (`/seller_api/.../vendor-items/{vendorItemId}/inventories`)는 **판매자 본인 상품만**(HMAC 인증·벤더 등록) 조회 가능해
  **임의 상품 URL 조회 불가**. 그래서 우리는 키·제휴 없는 **범용 임베드-JSON 파싱**만 강화했다.

### 목표 마일스톤 (✅ 구현, 2026-07-21)
목표를 **단계(마일스톤)**로 쪼갠다 — 예: 도쿄여행 = ✈️비행기표·🏨호텔·🎫교통패스. 목표 잔액이 마일스톤의
**누적 비용**을 넘으면 순서대로 하나씩 '획득'되고, 획득 순간 축하한다(저축이 추상적 숫자가 아니라 눈에 보이는 단계로 채워짐).
- **획득 로직**: `acquiredCount` = 누적 비용 ≤ 잔액인 단계 수(순수 함수, 테스트). 현재 진행 단계는 진행률·남은 금액 표시.
- **"가는 날 N일 단축"**: `fundedDays = round(잔액/목표 × 목표기한일)`(`SavingsGoal.deadlineDays`, 기본 90). 잔액이 커버한 기한일수.
- **마일스톤은 사용자 정의**(이모지+이름+금액 추가/삭제). 데모 목표엔 예시 마일스톤을 시드한다.
- **축하는 프론트에서 스냅샷 비교**로 처리(백엔드 `build` 시그니처 불변) — 직전 대비 새로 획득한 단계를 감지해 표시.

### FDS 재프레이밍 (✅ 구현, 2026-07-21 · 2026-07-21 톤 공감형 순화)
원래 통계적 FDS(감사로그의 발표 주인공)를 게임 루프의 **행동 신호**로 다시 엮었다.
> **톤 원칙(비전 재점검)**: 신호 **탐지 로직은 그대로** 두되, 문구는 **지적·통제가 아니라 공감·응원**으로 낸다. 자책을
> 유발하지 않고 '다음 한 번'의 획득 기회로 돌린다("소비건전성지수가 떨어지고 있어요" 같은 손실 프레이밍은 걷어냈다).
- **실제 소비건전성지수 표시**: 포인트 화면에 `ScoreService`의 지수(0~100·등급)를 실어 흐름을 보여준다
  (지수는 전체 계획소비 비율 기반이라 완만하게 움직인다 — 즉각 신호는 아래 streak/경고가 담당).
- **미계획 소비 연속(streak)**: **사용자가 앱에서 실제 기록한 소비(USER_INPUT)** 끝에서 이어지는 '필요없는 소비' 횟수.
  3회 이상이면 "요즘 마음 가는 대로 쓴 소비가 몇 번 있었네요. 다음 한 번은 참아서 목표에 담아볼까요? 💪"(더미 시드가 아니라 '최근 내 행동').
- **다짐 어김**: '안 샀어요'로 담아둔(NOT_BOUGHT) 카테고리에서 **그 뒤** 또 미계획 소비하면 "'○○' 대신 비슷한 걸 골랐네요.
  괜찮아요 — 다음엔 그 돈을 목표에 담아봐요 🌱". 담은 시점(createdAt) 이후의 미계획 소비만 본다.
- **조합**: `PointService`가 기존 `ScoreService`·`AnalysisEngine`을 재사용(중복 계산 없음). streak은 순수 함수로 테스트.

### 통장 비교 (정보성) — 절감액을 담을 실제 통장 (✅ 구현, 2026-07-21)
비전의 "절감액 기반 금융상품 **정보** 비교"를 구현한다. 목표에 모은 돈을 실제로 어디에 담을지, **실 예·적금 금리**를 비교로 보여준다.
- **자격 제한 제외**: 불러온 목록에서 상품명에 `간부·청년·장병·미소·청약`(가입 자격이 제한돼 일반 사용자가 담기 어려운
  통장)이 들어가면 제외하고, 남은 것을 **기본금리 내림차순**으로 정렬. 제외 키워드·조건은 전부 `application.yml`(원칙 4).
- **데이터 출처 & 견고성**: 네이버페이 저축상품 목록 API(비공식, `reference/test.py` 방식)를 서버가 offset 페이징으로 조회.
  공식 API가 아니라 차단·스키마 변경에 취약 → **실패 시 현실적 더미로 자동 폴백**, 성공은 **TTL 캐시**(매 요청마다 외부 안 때림).
  순서는 결정론적이라 재현 가능(§4). 순수 `filterAndRank`·`parseRate`는 단위 테스트로 고정(외부 실호출 없음).
- **규제(원칙 5 개정, §4)**: 금융위·금감원 유권해석(2022.6.15, 금융소비자정책과) — 목적이 **단순 정보제공**이고 **판매 목적이
  아니며** 판매사와 **제휴·광고 계약이 없고 가입 편의를 제공하지 않으면** 금소법 '금융상품판매대리·중개업'에 해당하지 않는다.
  그래서 이 화면은 **가입 버튼·제휴 링크 없이** 정보만 보여주고 "가입은 각 금융사에서"를 명시한다. **제휴·가입유도를 붙이는
  순간 다시 중개업 리스크**가 되므로 경계를 지킨다.

### 목표별 저축 계획 & 통장 추천 (✅ 구현, 2026-07-21)
"이 목표는 **이런 소비를 줄이면** 총 얼마를 **몇 개월** 모으면 된다"를 사용자가 세우고, 그 기간에 맞는 실 적금을 목표마다 붙인다.
- **계획(줄일 소비 선택)**: 목표 카드에서 내 **습관 소비**(미계획 소비의 카테고리별 월평균 = `cutOptions`)를 골라 끈다.
  **월 절약액 = 고른 카테고리 월평균 합**, **달성 개월수 = ⌈목표금액 / 월 절약액⌉**. 고른 카테고리는 `SavingsGoal.planCutCategories`(CSV)로 저장.
- **목표별 실 API 추천**: 개월수를 적금 지원 버킷(**6·12·24**)으로 매핑해 그 기간으로 `SavingsCompareService`를 검색하고
  자격 제한 제외 후 **최고 기본금리**를 고른다. **목표 1·2·3에 서로 다른 통장**(이미 배정된 상품은 다음 후보로 — 중복 금지).
  계획이 없는 목표는 목표 기한(deadlineDays)을 기간으로 대체한다.
- **더미 추천 교체**: 기존 `맞춤 상품 추천`(더미 `FinancialProduct` + ScoreService 매칭)을 **이 목표별 실 추천으로 교체**했다.
  정보성(원칙 5)이라 가입 버튼·제휴는 없다. (구 매칭 로직 `RecommendService`는 코드로 남아 있으나 화면에선 미사용.)
- **재현성·안전**: 개월수·버킷·정렬은 결정론(§4). 추천은 스냅샷 build와 분리된 별도 엔드포인트(`GET /api/points/recommendations`)로
  불러와, 외부 API 지연이 게임 루프 액션을 막지 않는다. 순수 함수(`monthsToGoal`·`cutsMonthlySaving`·`nearestPeriodBucket`)만 테스트.

### 충동예산 절약통 — 수동 '살 뻔했다'를 대체 (✅ 구현, 2026-07-21 방향 전환)
'살 뻔했다'를 매번 누르는 능동 액션 대신, **참는 것이 시간에 따라 저절로 쌓이게** 만든다. 선물상자의 의미가
'목표 입금액'에서 **'안 쓰면 모이는 충동예산'**으로 바뀐다(저축 탭의 새 주인공, 기존 목표·통장 루프는 접이식 '참고'로 보존).
- **예산 = 충동 카테고리의 월 평균 지출**: 카드내역(=시드된 3~6개월 소비 이력)에서 '충동'으로 지정한 카테고리들의
  월평균 지출 합. "이걸 안 쓰면 매달 이만큼 모인다"의 그 금액.
- **자동 성장(방문 기반)**: 들어올 때마다 **하루 할당량(예산÷그달 일수)**을 **50→30→20%**로 드러낸다. 안 들어온 날의
  할당량은 다음 방문에 합산되고, **며칠 비우면 그만큼 더 크게** 커진다(누적). 예산이 상한.
- **균열**: **충동소비를 기록할 때만** 그 금액만큼 잔액이 깎인다(필요소비는 영향 없음).
- **월간 재검증**: 다음달 **카드내역 CSV(`날짜,카테고리코드,금액`)를 재업로드**하면, 지정한 충동 카테고리의 지출이
  기준선(그 달을 뺀 월평균) 대비 **실제로 줄었는지 월대월로 대조**해 "▼N% 줄었어요"로 보여준다.
- **재현성(§4)**: 성장은 시간 의존이지만 `Clock` 주입 + 저장 상태(마지막 방문일·오늘 드러낸 비율·잔액)만으로 결정론.
  성장 로직(`nextFraction`·`accrueDelta`)은 순수 함수로 테스트한다. 신규 `DataSource.CARD_UPLOAD`, 파기 대상에 절약통·업로드 카드내역 포함.

### 단계 로드맵 (남은 것)
- **Phase 4 커뮤니티**: 공동 목표로 함께 모으기, 서로 프로필(누가 며칠 남았는지) 열람.

### 기존 기능과의 관계
- **절약 시뮬레이터(what-if)는 포인트 루프로 흡수**한다. "이렇게 줄이면 얼마"라는 수동 계산이 "살 뻔했다"라는
  능동 행동으로 대체된다. 시뮬레이터의 계산(적금 미래가치·미계획 지출 후보 탐지)은 포인트 서비스가 내부 재사용한다.
- **FDS**는 부가로 남아, '안 사기로 한 것을 산' 이상 신호로 재활용된다(Phase 3).

### 규제 안전 (불변)
포인트는 **가상**, 저축·"이체"는 **가상 이동일 뿐 실제 송금·결제·계좌연동이 아니다**. 추천/매칭에 쓰는 상품은 **더미**다.
단, **통장 비교(정보성)**만은 실 금리 정보를 표시한다 — 판매·중개가 아니라 정보제공이며(무제휴·가입 편의 없음, "가입은
각 사로") 금융위 유권해석상 중개업이 아니기 때문(원칙 5 개정). 이 경계(제휴·가입유도 금지)를 넘지 않는 한 안전하다.
실거래·실상품이 없으므로 금소법 중개/이체 판단 대상이 아니다(D-04·설계원칙 유지). 화면·약관에 "가상 시뮬레이션"임을 명시한다.

## 6. API 명세 확정

```
GET  /api/products/recommend?userId=1     ← Top3 추천 (근거필드 + gatingRelaxed 포함)
GET  /api/report/monthly?userId=1&month=2026-07
GET  /api/alert/list?userId=1             ← FDS 경고 목록
GET  /api/score/{userId}                  ← 소비건전성지수
POST /api/consumption                     ← 실사용자 소비내역 입력 (source=USER_INPUT 고정)
GET  /api/audit/verify                    ← 감사로그 체인·Merkle·앵커 검증 결과
POST /api/dev/seed                        ← [개발/관리자 전용] DUMMY_SEED 삽입, 일반 접근 불가
```

**공통 규약**
- **모든 조회 응답(`recommend`/`report`/`alert`/`score`)에 `dataSourceMode`를 포함한다.**
- **모든 추천 응답에 "왜 이 순위인지" 근거 필드를 포함한다** — 선택 보강 지표(설명가능성)의 검증 대상이다 (→ D-10).
  `D19`가 요구하는 "조건 추출 → 속성 매칭 → Top 3 비교 추천"을 화면에서 설명하려면 어차피 필요하다.
- `POST /api/dev/seed`는 프로파일 분리로 운영 환경에서 비활성화한다.

| **RFP 필수** | 모의 상품 가입 의향 | `C13` `D20` | **필수 수집·제출** |
| **RFP 필수** | 상품 추천 클릭률 | `D20` | **필수 수집·제출** (GA/Amplitude 이벤트) |
| 선택 보강 | 재현성 — 같은 입력 → 같은 추천 순위 100% 재현 | 자체 | 여건 되면 구현 |
| 선택 보강 | 설명가능성 — 모든 추천에 근거 필드 존재 | 자체 | 여건 되면 구현 |
| 선택 보강 | 무결성 — 감사로그 변조 100% 탐지 | 자체 | 여건 되면 구현 |
| 선택 보강 | 탐지 성능 — 이상거래 시나리오 N건 중 탐지율 | 자체 | 여건 되면 구현 |

> **왜 RFP 지표를 우선하는가**: `B6`·`D20`·`D24`가 이들을 **명시적 제출물**로 지정한다. 빠지면 미제출 감점이다.
>
> **"통계적으로 무의미하다"는 반론이 해소된 이유**: RFP는 애초에 이 지표들을 통계 지표로 요구하지 않았다.
> `D24`가 "리포트 만족도 **정성** 피드백", `C11`이 "진정성 있는 고품질의 **정성** 피드백"이라고 명시한다.
> 따라서 **n을 밝히고 정성 자료로 보고하면 요건을 충족**한다.
> 단, 만족도·가입의향은 n에 취약하고 **클릭률·체류시간은 GA 이벤트라 n에 상대적으로 강건**하므로 구분해 보고한다.
>
> **선택 보강 지표의 성격**: 사용자 수가 0명이어도 측정되며, 채용 담당자가 실제로 보는 것에 가깝다.
> 다만 **필수 구현 대상이 아니다** — 일정에 여유가 있고 적절하다고 판단될 때만 부가적으로 넣는다.

> - 총대 모금의 실재성은 미검증 → **공동 모금 기능은 MVP 스코프에 넣지 않는다.**

>
> 페르소나가 뼈대 완성 후에 바뀔 수 있으므로, 지금 만드는 엔진에 특정 세그먼트를 못 박으면 나중에 통째로 고쳐야 한다.
> 따라서:
> 1. **카테고리를 하드코딩하지 않는다.** `Category`는 DB 데이터이고, `overspendingCategories`는
>    "비중 30% 초과"라는 **규칙**으로만 산출한다. "덕질", "굿즈" 같은 이름이 코드에 등장해서는 안 된다.
> 2. **임계치를 상수로 박지 않는다.** 과소비 30%, Modified Z 임계 3.5, `CONFIRMED` 30건/14일을
>    **설정값(config)으로 분리**한다. 페르소나가 바뀌면 값만 갈아끼우면 되도록 한다.
> 3. **FDS 룰 3개는 구조만 고정한다.** "심야 고액 / 신규 카테고리 급증 / 빈도 이탈"은
>    세그먼트와 무관한 일반 패턴이므로 그대로 두되, 시간대·배수 파라미터는 설정값으로 뺀다.
> 4. **더미 시드 생성기를 스크립트로 만든다.** 손으로 만든 CSV를 커밋하면 페르소나 교체 시 재작업이 크다.
>    분포 파라미터를 받아 생성하는 스크립트여야 세그먼트 교체 비용이 낮다.
>
> **FDS 임계치 캘리브레이션("100건 중 3~5건")은 페르소나 확정 후에 다시 돌린다.**
> 지금 시드로 잡은 계수는 잠정값이며, 엔진이 설정값을 읽도록 만들어 두면 재캘리브레이션이 값 교체로 끝난다.

### 11-1. 구현으로 드러난 설계 결함

**① 룰 ②(신규 카테고리 급증)가 영원히 발화하지 못하는 구조였다** — 가장 심각

§5 ①은 "카테고리 표본 10건 미달 시 산출하지 않는다"와 "경고 = z 플래그 AND 룰 1개 이상"을
동시에 규정한다. 그런데 **신규 카테고리는 정의상 표본이 없다.** 따라서 z를 못 내고,
z가 없으면 AND 조건이 성립하지 않아 룰 ②는 **논리적으로 발화 불가능**했다.

→ **수정**: 카테고리 표본이 부족하면 사용자 **전체 분포**로 대체해 z를 산출한다
(`AnalysisResult.BaselineSource.CATEGORY | GLOBAL`). 경고 문구도 "같은 카테고리의 평소 지출 대비" /
"회원님 전체 평소 지출 대비"로 구분해 근거를 밝힌다.

**② MAD = 0이면 Infinity가 되어 전건이 경고로 뜬다**

균일한 소비 패턴(모든 금액이 동일)에서 MAD가 0이 되고, `0.6745 × (x−median) / 0` → Infinity.
그러면 **모든 거래가 임계 3.5를 넘어 경고가 도배**된다 — Part III-D §7-2가 경고한 바로 그 실패다.

→ **수정**: MAD가 0이면 평균절대편차로 대체하고, 그마저 0이면 편차가 없다는 뜻이므로 0을 반환한다.
`StatsTest.modifiedZHandlesZeroMad`가 이를 고정한다.

**③ 재현성은 "정렬 고정"만으로는 달성되지 않는다**

엔진이 `LocalDateTime.now()`를 직접 부르면 **같은 입력이 시각에 따라 다른 출력**을 낸다.
FDS 윈도우가 현재 시각 기준이므로 재현성 검증(원칙 3)이 원천적으로 불가능해진다.

→ **수정**: `Clock`을 빈으로 주입하고 엔진은 `referenceTime`을 **파라미터로 받는다.**
테스트는 고정 Clock을 주입한다. 정렬 고정·TreeMap과 함께 세 가지가 다 있어야 재현성이 성립한다.

### 12-1. 감사로그 계층 3 — TSA 앵커링 **실동작 확인**

RFC 3161 구현(BouncyCastle `bcpkix-jdk18on` 1.85)으로 **FreeTSA에서 실제 서명을 받았다.**

```

   batch=1 앵커링 완료 (genTime=2026-07-20T15:37:22Z)
```

**★ 계층 3의 존재 이유를 실증했다** — 계층 1·2를 모두 통과하는 공격을 만들어 앵커만이 잡는 것을 확인:

| 단계 | 공격 행위 | 계층 1 | 계층 2 | 계층 3 |
|---|---|---|---|---|
| ① 단순 변조 | 페이로드 1건 수정 | **탐지** | — | — |
| ② 엔트리 삭제 | 로그 1건 delete | 통과 | **탐지** | — |
| ③ **완전 재생성** | 삭제 + Merkle 루트 정직하게 재계산 | 통과 | 통과 | **탐지** |

③의 검증 출력: `batch=1: TSA 앵커 검증 실패 — messageImprint 불일치 — 앵커된 루트가 현재 배치 루트와 다르다`

> 발표에서 이 세 단계를 순서대로 보여주면 "왜 해시체인만으로는 부족한가"가 말이 아니라 화면으로 증명된다.
> 시연: `./scripts/demo-tamper.sh break` → `regenerate`

**구현 중 발견한 함정 2건**
- `messageImprint`는 batch_root의 **raw 32바이트를 SHA-512**한 값이다. hex 문자열을 해시하면 `openssl ts -verify`와 어긋난다.
- **외부 문자열은 길이를 신뢰할 수 없다.** FreeTSA의 인증서 DN이 212자라 `tsa_name VARCHAR(200)`을 넘겨
  **앵커링은 성공했는데 저장이 통째로 실패**했다. 컬럼을 넓히고 코드에서도 방어적으로 자른다.

### 12-2. FDS 임계치 — **3.5가 임의값이 아님을 실증**

z>5.0      경고 0/62 = 0.00/100

                    warning: "표본 2명 — 통계적 유의성이 없다. '정성 피드백'으로만 해석할 것."
```

> **n에 강건한 지표(클릭률·체류시간)와 취약한 지표(만족도·가입의향)를 응답에서 분리**하고,
> 표본 30 미만이면 경고 문구를 자동으로 붙인다. n=2 평균을 통계인 척 제시하면 심사자에게 정확히 반박당한다.
>
> GA/Amplitude로도 보내지만 **서버에도 쌓는다** — 외부 도구는 키·네트워크에 의존하는데
> 발표 당일 막히면 RFP 필수 제출물이 통째로 사라진다.

### 12-5. 그 외

- **리포트 캐싱**: `Report` 엔티티를 실제로 쓴다. 단 **`ESTIMATED`는 캐시하지 않는다** —
  캐시하면 "N건 더 기록하면 정확해집니다"라고 안내해 놓고 갱신되지 않는 모순이 생긴다. 소비 입력 시 해당 월 캐시를 무효화한다.
- **Jackson 3**: Spring Boot 4는 `com.fasterxml.jackson`이 아니라 **`tools.jackson`**이다.
- **`@Lob` 함정**: H2에서 VARCHAR로 떨어져 값이 잘린다. `@Column(length = 1_000_000)`로 교체
  (H2는 CHARACTER VARYING, MySQL은 length>65535라 LONGTEXT로 매핑).

### 12-6. 적대적 코드 리뷰 — **확정 결함 7건 발견·수정**

6개 관점(정확성 / 재현성 / 감사·암호 / 개인정보·규제 / API·프론트 / 문서-코드 정합)으로 독립 검토한 뒤,
각 지적을 **두 개의 다른 렌즈(기술적 타당성 / 이 프로젝트에서의 중요도)로 반박 시도**해
둘 다 반박하지 못한 것만 확정했다. 90개 에이전트가 검토했고 대부분은 기각됐다.

| # | 심각도 | 결함 | 수정 |
|---|---|---|---|
| 1 | **치명** | **삭제·동의철회가 `Alert`·`Report` 잔재를 남긴다** | `PrivacyService`가 세 테이블을 모두 지우도록 수정 |
| 2 | 높음 | 변동성 **측정 불가**를 CV=0(=안정성 만점)으로 흘려보냄 | `volatilityMeasured` 플래그 도입, 해당 항 제외 후 가중치 정규화 |
| 3 | 높음 | 보유기간 파기가 `Consumption`만 지움 | 딸린 `Alert` 삭제 + `Report` 캐시 무효화 |
| 4 | 높음 | **운영 compose가 dev 시드 엔드포인트를 켜 둠** | 기본 `false`, `mysql` 프로파일에서도 강제 차단 |
| 5 | 높음 | 입력 폼 기본 일시가 UTC (KST 9시간 오차) | 로컬 벽시계 문자열 생성 함수로 교체 |
| 6 | 중간 | 리포트 캐시 키 불일치로 stale 응답 | 사용자 단위 전체 무효화(`invalidateAll`) |

**1번이 가장 뼈아프다.** `Alert`는 `amount`·`occurredAt`·`categoryCode`를 **자기 테이블에 복사해 갖고 있는데**,
내 `PrivacyFlowTest`는 `Consumption`만 확인하고 통과했다. 즉 **테스트가 초록불인 채로 처리방침을 위반**하고 있었다.
"동의 철회 → `GET /api/alert/list`" 두 번의 호출로 심사자가 1분 안에 재현할 수 있는 모순이었다.

> **교훈**: 개인정보 삭제는 "원본 테이블"이 아니라 **"그 정보가 복사된 모든 곳"**을 지워야 한다.
> 파생 테이블(경고·리포트 캐시)이 개인정보의 사본을 들고 있다는 사실을 놓치기 쉽다.

**2번**도 실질적이다. 관측 월수가 3개월 미만이면 CV를 측정할 수 없는데 0.0을 그대로 내보내
`stability = 1 - 0/0.6 = 1.0`, 즉 **안정성 만점 30점**을 공짜로 받았다.
같은 소비 패턴이라도 **기록을 적게 한 사용자가 더 높은 점수를 받는 역설**이 생긴다.
회귀 테스트 `ScoreFairnessTest.lessDataDoesNotOutscoreMoreData`가 이를 고정한다.
같은 이유로 `RecommendService`가 측정 불가를 '안정형'으로 단정하던 것도 '중립형'으로 바꿨다.

## 13. 마이데이터 전면 도입 (계획, 2026-07-21)

> **이 절은 향후 계획의 단일 기준이다.** 이후 마이데이터 관련 구현은 전부 이 절을 따르며, 방향이 바뀌면 이 절을 고치고 변경 이력에 남긴다.

### 13-0. 배경 — 방향 전환
사용자 판단: **① 마이데이터(더미) 서버 · ② 휴대폰 본인인증 · ③ 마이데이터로 카드사용내역을 불러오는 흐름 · ④ 디자인/화면 구성**을
우리 서비스에 이식한다. 즉 "더미"라고 부르던 데이터를 **우리가 직접 띄우는 마이데이터 서버**가 제공하고, 본인인증을 거쳐 그 서버에서 카드·소비내역을 당겨와 기존 분석 엔진에 태운다.

**우리 기존 리서치와 정합한다.** 개인은 실 마이데이터 사업자 허가·테스트베드에 접근할 수 없다(§III-B §4·§5: 자본금 5억·망분리,
테스트베드 참가기관 전용). 그러나 **마이데이터 표준 API 규격은 공개 열람 가능**하며, 우리는 이미
"*원본은 접근 불가이므로 공개 표준 스펙 모양으로 더미를 설계한다*"를 근거로 세워뒀다(§III-B §6-1). **우리가 자체 더미
은행/카드사 서버를 띄우는 방식이 바로 그 실현**이다. 실 신용정보 연동이 아니라, 표준 스펙 모양의 더미 제공자를 우리가 운영하는 것이다.

### 13-1. 채택한 마이데이터 시뮬레이션 방식 (핵심 기법)

개인은 실 마이데이터에 접근할 수 없으므로(§III-B §4·5), **공개된 마이데이터 표준 API 규격 모양의 더미**를 우리가 직접 운영한다(§III-B §6-1). 채택한 기법:

- **제공자 분리**: 마이데이터 제공자를 본체와 **별도 프로세스·별도 DB·HTTP 경계**로 띄운다 — 데이터 출처의 분리가 마이데이터의 본질이다.
- **CI 식별**: `CI = SHA-256(이름+주민번호앞7자리+전화번호)` 를 마이데이터 조회 키로 쓴다(실 인증값이 아닌 **가상 생성값**, §13-2).
- **결정론 더미**: 사용자 → 카드(3~6장, 카드 풀에서 선택) → 결제(수십건, 최근 N일, **7대분류×N중분류** 카테고리 트리, 가맹점)를 **시드·기준일 고정**으로 생성(규칙 3, `now()`·무시드 금지).
- **혜택 모델**: 카드 혜택을 **대분류별·전월실적 구간별** 할인율/월한도로 두고, 결제 1건의 받은 혜택을 실적구간 대조로 계산한다.
- **인증 보완**: 인증코드는 서버에 **TTL 저장 + 검증 엔드포인트**로 무결성을 확보한다(단순 발송·클라 비교만으로는 재현·검증 불가).

### 13-2. 우리 적용 원칙 & 정합/충돌

- **정합**: §III-B §6-1(공개 표준 스펙 더미)의 실현. 우리 4대 설계원칙 유지 — 판단은 코드가, `AnalysisResult` 재사용,
  재현성(`Clock`·시드·`TreeMap`), 카테고리 코드 비박제·임계치 `application.yml`.
- **D-26/PII — 해소(사용자 결정 2026-07-21)**: coolsms **실 발송은 최종 목표**지만, **현 단계는 "가상으로 인증되었다"는 가정 스텁**으로
  만들고 **실 인증은 후속**으로 붙인다. 핵심은 **우리 CI가 실 신용정보 인증(NICE평가정보 등)에서 받아오는 실 CI가 아니라, 가상으로
  생성한 값**이라는 점이다 — 인증 절차를 거쳐 **가상 CI를 받아 마이데이터(더미) 서버 조회 키로 넘기는 용도**일 뿐, 실 신용정보를
  다루지 않는다. 따라서 **D-26("실제 인증·개인정보 요구 없이 가상 시나리오")의 취지에 부합**한다(신원·금융데이터 전부 더미).
  - **현 단계(Phase 2)**: 실 SMS 없이 `AuthService`가 입력 신원으로 **가상 CI를 계산하고 '인증됨'으로 처리**한다(전화번호 실수집 없음).
  - **후속**: 실 coolsms 발송을 앞단에 붙일 때 전화번호(수신용)를 수집하게 되며, 그때 처리방침(`legal/privacy-policy.md`·`PrivacyService.policy()`)에
    반영한다. coolsms 키는 사용자가 직접 발급한다(공개된 타 저장소에 노출된 키는 타인 자격증명이므로 재사용 금지). **규칙이 충돌하면 규칙을 수정한다**(사용자 방침).

### 13-3. 목표 아키텍처 (우리 이식)

- **2서버**: `backend`(8080, 본체) + **`backend-mydata`(8082, 마이데이터 제공자)** — 별도 Maven 모듈·별도 프로세스·별도 DB.
  본체→마이데이터 호출은 **`RestClient`**(Spring 6/Boot 4 동기 클라이언트, WebFlux 불필요)로 한다.
  DB는 H2 파일(dev)/MySQL(prod), Redis는 향후(§13-8).
- **데이터 모델(이식)**:
  | 클러스터 | 엔티티 |
  |---|---|
  | 마이데이터 서버 | `MyDataUser`(id=CI, 이름, 주민번호, 전화) · `MyDataCard`(serial, 카드코드, 만료, 전월실적액) · `MyDataPayment`(일시, 대분류/중분류, 금액, 가맹점, 받은혜택) · 카탈로그 `CardProduct`·`CardCompany`·`Category1/2`·`CardBenefit`(실적구간·할인율·월한도) |
  | 본체 | `AppUser`에 CI/전화 연결 · `UserCardCompany`(연동 카드사) · `UserCard`(보유카드·전월/당월실적) · `UserPayment`(불러온 결제) |
- **기존 엔진 재사용**: 마이데이터에서 불러온 `UserPayment`를 **`Consumption(source=MYDATA)`로도 투영**해,
  소비건전성지수·절약 리포트·충동예산 절약통·FDS가 **재계산 없이 그대로** 쓴다. → "마이데이터로 우리가 희망한 결과를 그대로 구현".

### 13-4. 본인인증 (coolsms)

요청→코드→검증→발급 흐름을 두되, 단순 발송에 그치지 않고 **서버 측 코드 저장(만료 TTL)·검증 엔드포인트**를 둔다(재현성·무결성).
`CI = SHA-256(이름+주민6+성별세대코드+전화)`. 코드는 결정론 테스트를 위해 생성기를 분리(운영은 난수, 테스트는 주입).
환경변수 `SMS_KEY`·`SMS_SECRET_KEY`·`SENDER`는 **사용자가 직접 발급**한다 — 키 미설정 시 앱이 죽지 않도록 개발 폴백(코드 로깅)을 둔다(그 사실을 문서에 명시).

### 13-5. 더미데이터 생성 (Faker → 결정론)

우리 규칙 3(재현성)에 맞춰 **시드 고정 `Random` + `Clock`** 으로 생성한다. 규모가 다소 작아지더라도 **구조는 동일**(사용자→카드 3~6→결제 ~수십건, 7×N 카테고리 트리, 카드 풀, 실적구간 혜택 산식).
카테고리 트리·카드 카탈로그·혜택 규칙은 마이데이터 서버가 소유(코드/시드), 본체는 판단 로직에 카테고리를 박지 않는다.

### 13-6. 프론트 (✅ 구현, 2026-07-21)

## §2. 리서치가 반증한 것

### D-04. 규제 논거는 틀렸다 — 교체

**최초 기획서 §10**: "추천은 '투자자문'이 아니라 '정보 제공/비교' 수준으로 포지셔닝"

**반증**: 금융위 2021-09-07 유권해석은 **"비교·추천"을 중개 인정 사례로 명시**한다.
판단 기준은 "정보제공이냐"가 아니라 **"판매 목적이냐"**다.

- 이 해석 하나로 2021년 카카오페이·토스의 보험 비교·추천이 **전면 중단**
- 미등록 중개 = 금소법 제67조 **5년 이하 징역 또는 2억원 이하 벌금**
- 현재 플랫폼 보험 비교·추천은 정식 라이선스가 아니라 **혁신금융서비스(샌드박스) 특례 11개사**로만 운영
- 반복 등장하는 단서: *"정보제공이 추천, 가입 유도, 가입 시 수수료 수취 등과 결합되는 경우 자문이나 투자권유 행위로 볼 수 있으며…"*

**확정된 올바른 논거** → Part I §2 "규제 방어 논거" 박스. 상세 근거 → Part III-B.

### D-05. 확장 로드맵 2단계는 경로가 없다 — 재작성

**최초 기획서 §11**: "7~12주: 오픈뱅킹·마이데이터 테스트베드 연동"

**반증**: 오픈뱅킹 테스트베드는 2023-09-18부로 참가기관 전용 폐쇄형 전환(공인 IP 사전등록 필수, 개인 불가).
마이데이터 테스트베드는 허가·예비허가 사업자 한정. 마이데이터 허가는 자본금 5억 + 망분리·CCTV·회선이중화 + 법인격 필수.
무허가 영업 시 신용정보법 제50조 제2항 **5년 이하 징역 / 5천만원 이하 벌금**.

**교체**: 2단계를 **"공개 API로 실데이터 대체"**로 재작성 — OPENDART, 한국은행 ECOS, KRX Data Marketplace, 공공데이터포털.

정직하게 *"마이데이터 원본은 개인이 접근할 수 없으므로, 표준 API 규격을 읽고 그 스펙 모양으로 더미를 설계했다"*고 쓰는 편이 훨씬 강하다.

### D-06. "숫자는 코드, 문장만 AI"는 차별화가 아니다 — 프레이밍 변경

**반증**: 뱅크샐러드의 카드 추천이 이미 룰 기반 결정론 엔진이고, 그 **이유가 규제에 있다** —
알고리즘 사전 검증 의무 때문에 블랙박스 ML보다 설명 가능한 룰 엔진이 유리하다.

**이것은 나쁜 소식이 아니라 더 좋은 소식이다.** 프레이밍만 바꾼다 → Part I §4 원칙 1.

### D-07. 발표 주인공은 FDS + 감사로그

**최초 기획서의 모순**: §2는 추천을 "1순위 필수", §12는 FDS를 "발표 주인공"이라 부른다.
**차별화 요소를 5주차에, 가장 지친 상태로, 가장 잘리기 쉬운 자리에 배치한 셈이다.**

**시장 리서치가 이 모순을 해소한다 — 추천·가계부는 전부 레드오션이다** (상세 → Part III-C):
브로콜리 2020년 종료 / 뱅크샐러드 MAU 115만→70만 / 순수 PFM 흑자 회사 없음 /
마이데이터 10곳 라이선스 반납·통신3사 전원 철수 / 비은행 핀테크 D30 리텐션 4.5~5.8%

**반대로 FDS+감사로그는 리서치가 강하게 뒷받침한다**: ISMS-P 2.9.4 / 전자금융거래법 제22조 5년 보존 / 전자금융감독규정 중요원장 5년.

**확정**: FDS를 발표 주인공으로 유지하되, **개발 순서에서도 추천보다 먼저**에 놓는다.

### D-08. 감사로그 설계 보강 — 해시체인만으로는 부족

**문제**: 선형 해시체인은 **"운영자가 DB를 통째로 재생성하는 공격"을 막지 못한다.**
로그와 검증 근거를 같은 주체가 통제하면 아무것도 증명하지 못한다.

**확정 설계 (3계층)** → Part I §5-4. 상세 근거·구현 함정 → Part III-D §4~6.

**계층 3이 이 프로젝트의 최대 차별화 지점이다.** "제3자가 서명한 시각 증명"이 붙는 순간
"전자서명·TSA" 태그가 말이 아니라 코드가 된다.

## §3. 미결정 → 확정

### D-09. 페르소나 — **미결 (확정 시점을 뒤로 미룸, 2026-07-20)**

진짜 고객이 채용 담당자로 바뀌었으므로 **페르소나의 중요도는 최초 기획서가 가정한 것보다 낮다** —
색·카피 결정용으로만 쓰고 시간을 크게 쓰지 않는다.

**사용자 결정: 페르소나는 확정하지 않는다. 기술 검토를 마치고 기술적 기본 뼈대를 다 만든 뒤에
페르소나를 수정해 확정한다.**

- 최초 기획서는 "디자인 착수 전 하나로 확정"을 요구했으나, **진짜 고객이 채용 담당자인 이상 그 순서는 구속력이 없다.**
  엔진이 먼저 서야 세 기능이 서고(D-02), 페르소나는 그 위에 얹는 표현 계층에 가깝다.
- 잠정 작업 가정(팬덤·성인 한정)과 임시 시드 S1·S2 → Part I §8. **이는 결정이 아니라 엔진 개발용 재료다.**

> ⚠️ **이 결정이 만드는 기술 부채와 그 상환 방법**
> 페르소나가 바뀌면 (a) 더미 데이터 (b) FDS 임계치 캘리브레이션 (c) 카테고리 구성이 함께 흔들린다.
> 이를 값 교체로 흡수하려면 **엔진을 세그먼트 비의존적으로 설계해야 한다** — 카테고리 하드코딩 금지,
> 임계치 설정값 분리, 시드 생성기 스크립트화. 상세 → Part I §8 마지막 박스.
> **이 제약을 지키지 않으면 페르소나 확정 시점에 엔진을 다시 만들어야 한다.**

### D-10. 성공 지표 — **확정 (2026-07-20)**

**최초 판단**: 최초 기획서의 "추천 만족도·체류 시간·모의 가입 의향"은
*"인터뷰 1~2명 조건에서 통계적으로 무의미하다"*고 보아 기술 지표로 **대체**하려 했다.

**RFP 원문 확보 후 정정 — 이 판단은 과했다.**
- **RFP는 이 지표들을 통계 지표로 요구한 적이 없다.** `D24`가 "리포트 만족도 **정성** 피드백",
  `C11`이 "진정성 있는 고품질의 **정성** 피드백"이라고 명시한다. 통계적 엄밀성을 기준으로 공격한 것은 허수아비였다.
- 게다가 `B6`·`D20`·`D24`가 이들을 **명시적 제출물**로 지정한다. 대체 대상이 아니라 **필수 제출 요건**이다.

**확정 (사용자 결정)**: **RFP 지표를 우선(필수)으로 삼는다.**
기술 지표(재현성·설명가능성·무결성·탐지율)는 확정 결정이 아니라, **여건이 되고 적절하다고 판단될 때
부가적으로 구현하는 선택 항목**으로 둔다. 필수 구현 대상이 아니다. 상세 표 → Part I §7.

### D-11. FDS 임계치 — **확정 (2026-07-20)**

> 기존 상태: *"기획서 §7-4는 'z-score 임계치 초과 시 플래그'라고만 적혀 있고, 임계치의 근거가 비어 있다."*

**확정**: Modified Z-score `0.6745 × (log(amount) − median) / MAD`, 임계 **3.5**, 룰 AND 결합,
더미 100건 중 경고 3~5건 역산. 산식 전문·근거 → Part I §5 ①, Part III-D §7.

> 개선안이 제시한 단순 `N=2`(2σ)는 **채택하지 않는다** — 정상 소비의 약 5%가 경고로 뜨고,
> 데모에서 경고가 도배되면 발표 주인공 기능이 그 자리에서 무너진다.
>
> **결정 시점 원칙은 유지**: 최종 계수는 페르소나 확정 후 더미데이터를 설계할 때 함께 역산한다.

### D-12. `test.py` — 네이버페이 상품 조회 기능 **(신규 미결, 2026-07-20)**

**사용자 결정: 보류.** 네이버를 통해 금융정보를 확인하는 기능으로, 추후 금융상품 비교 단계에서 재검토한다.
**현재 파이프라인에는 넣지 않는다.**

투입 전 반드시 해소할 것:

1. **D-04의 규제 방어 논거가 "실재하지 않는 더미 상품을 쓰므로 영업이 아니다"에 서 있다.**
   실상품을 추천에 넣는 순간 이 논거가 무너지고 금소법 중개업 판단 대상이 된다 (제67조: 5년 이하 징역 또는 2억원 이하 벌금).

### §4. 디커플링 분석

기존 마이데이터 앱의 고객 가치 사슬:

리서치 확인 사실: [2] 조회는 **완전히 상품화**됐다(토스·카카오페이·네이버페이가 무료 제공, 브로콜리는 여기서만 놀다가 종료).

승자들은 [5]를 **"자동 실행"**으로 바꾸는 중이다.

**따라서 [2]에 들어가는 건 자살이다.**

- **후보 1: [3] 인사이트를 "세그먼트 특화"로** → 약함. 범용 앱이 카테고리만 추가하면 따라잡힌다. **방어선이 없다.**

  ```
  [1] 연동 → [2] 조회 → [3] 인사이트 → [★ 공동 모금·집행·증명 ★] → [4] 추천 → [5] 실행

#### 2-1. 금융위 2021-09-07 유권해석

**판단 기준은 "정보제공이냐"가 아니라 "판매 목적이냐"다.**
> 서비스의 목적이 정보제공 자체가 아니라 **판매를 목적**으로 하는 경우 일반적으로 **중개**로 본다.

③ 소비자가 플랫폼을 **계약주체로 오인**할 가능성

#### 2-2. 광고 vs 중개 구분표

| 구분 | 광고에 가까움 | **중개에 해당** |

→ https://www.criminalip.io/ko/knowledge-hub/blog/20175

**이 한 문장에 4개 조건이 AND로 결합돼 있다:**
```
(연령 세그먼트) × (단시간 윈도우) × (수취인 신규성) × (소액 × 다건)

**우리 FDS 룰 설계에 그대로 차용할 수 있는 구조다.** 룰 3개(심야 고액 / 신규 카테고리 급증 / 빈도 이탈)도
단일 조건이 아니라 **AND 결합**으로 만들면 오탐이 크게 준다.

#### 3-2. 기타

- **FDS 적용 범위는 '로그인 – 전자금융거래 – 로그아웃' 3단계 전부** (금감원 규정). 거래 시점만이 아니라 세션 전체
- 국내 FDS는 **"시나리오 기반 / AI 기반 / 블랙리스트 기반"**으로 분류되며 **대부분 2개 이상을 조합**
- 법적 근거: FDS 운영을 직접 규정한 법령은 없다. 「통신사기피해환급법」 시행령 제2조의4의
  "상시적으로 점검하는 전산시스템" 조항과 「전자금융거래법」 제3장이 근거로 원용된다

### §4. 감사로그 3계층 설계 ★★

설계 본문 → Part I §5-4. 근거:

**기획서 §6 설계의 결함**: (선택) `audit_log`는 prev_hash + payload_hash + timestamp의 선형 해시체인이다.
**선형 해시체인은 "운영자가 DB를 통째로 재생성하는 공격"을 막지 못한다.**
체인을 처음부터 다시 계산하면 **완벽하게 일관된 가짜 체인**이 만들어지고 검증해도 통과한다.

> **루트 해시는 운영자가 닿을 수 없는 곳에 게시되어야만 의미가 있다.**
> 같은 주체가 로그와 검증 근거를 모두 통제하면 해시체인은 아무것도 증명하지 못한다.

**왜 발표에서 강한가**: TSA는 **신뢰받는 제3자**로서 "이 해시값이 이 시각 이전에 존재했다"는 사실에 서명한다.
중요한 건 **TSA가 원본 데이터를 보지 않는다**는 점이다 — 해시만 전송하므로 로그 내용의 기밀성이 유지된다.

### §5. 구현 함정 (반드시 지킬 것)

#### 5-1. 정규화(canonicalization) 필수
**JSON 키 순서·공백·유니코드 정규화가 흔들리면 검증이 전부 실패한다.**
```python
def canon(obj: dict) -> bytes:
    return json.dumps(obj, sort_keys=True, separators=(",", ":"),
                      ensure_ascii=False).encode("utf-8")
```
RFC 8785(JCS)를 쓰거나, 직렬화 바이트 원본을 그대로 저장할 것.

#### 5-2. 도메인 분리 (second-preimage 방지) — RFC 6962 방식
```python
def leaf_hash(data):  return sha256(b"\x00" + data).digest()
def node_hash(l, r):  return sha256(b"\x01" + l + r).digest()
```
**안 하면 트리 구조를 속일 수 있다.**

#### 5-3. TSA 요청 간 15초 이상 지연
공개 TSA(Sectigo 등)는 스크립트 연속 호출 시 **각 요청 사이 15초 이상 지연**을 요구한다.
→ **이것이 "매 로그마다"가 아니라 "배치 앵커링"이어야 하는 실무적 이유다.**

#### 5-4. TSA 인증서·CA 체인을 앵커와 함께 보관
**10년 뒤 그 인증서를 웹에서 못 구한다.** `.tsr` 파일과 `tsa.crt`, `cacert.pem`을 같이 보관할 것.

#### 5-5. 앵커를 로그 DB와 물리적으로 분리 보관
앵커 자체를 잃어버리면 전부 무의미하다.

### §6. 검증 절차 (감사자 관점) — 시연 시나리오로 그대로 사용

```
1. 앵커 검증
   openssl ts -verify -in anchor_N.tsr -queryfile anchor_N.tsq \
        -CAfile cacert.pem -untrusted tsa.crt
   → genTime 추출, messageImprint == SHA512(batch_root_N) 확인

2. 배치 체인 검증
   batch_header_N.prev_batch_root == batch_root_{N-1}

3. Merkle 루트 재계산
   저장된 entry_hash 목록으로 batch_root_N 재계산 → 앵커된 값과 일치 확인

4. 엔트리 체인 검증
   SHA256(prev_hash || canonical_json(entry)) == entry_hash

5. 단일 엔트리 포함 증명
   sibling 경로만으로 batch_root 재구성 → 전체 로그 공개 없이 검증
```

#### 6-1. openssl 기본 워크플로
```bash
# 1) 타임스탬프 요청(TSQ) 생성
openssl ts -query -data merkle_root.bin -no_nonce -sha512 -cert -out root.tsq

# 2) TSA에 전송
curl -H "Content-Type: application/timestamp-query" \
     --data-binary '@root.tsq' https://freetsa.org/tsr > root.tsr

# 3) 검증
openssl ts -verify -in root.tsr -queryfile root.tsq \
     -CAfile cacert.pem -untrusted tsa.crt

# 4) 내용 확인
openssl ts -reply -in root.tsr -text
```

`[사실]` FreeTSA 현행 인증서는 ECC P-384(secp384r1), 유효기간 2040년 2월까지. 배포 파일: `tsa.crt`, `cacert.pem`.

**관련 표준**
- RFC 3161 — Time-Stamp Protocol: https://datatracker.ietf.org/doc/html/rfc3161
- RFC 5816 — ESSCertIDv2 Update (SHA-1 문제 해결. **RFC 5816 준수 TSA를 쓸 것**)
- RFC 6962 — Certificate Transparency (도메인 분리 근거)

**국내 상용 TSA** `[참고]`: 코스콤 SignKorea, 한국정보인증(KICA), 한국전자인증(CrossCert).
`[추정]` 대부분 계약 기반 유료이며 공개 무료 엔드포인트를 제공하지 않는다. 이 프로젝트는 FreeTSA로 충분하다.

### §7. FDS 임계치를 어떻게 정할 것인가 — **D-11로 확정됨**

#### 7-1. 왜 문제인가
`[사실]` 사기 탐지에서 **오탐(false positive) 비용이 미탐(false negative) 비용보다 크다**는 것이 업계 통계의 일관된 방향이다.
정상 거래를 잘못 차단하면 고객 이탈이 발생하고, 그 손실이 실제 사기 손실을 웃돈다.

`[주의]` 관련해 인용되는 구체적 배수(예: "오거절이 사기의 9배")는 **대부분 사기방지 벤더가 발행한 자료**이며
방법론이 공개되지 않은 경우가 많다. **방향성만 취하고 수치는 인용하지 말 것.**

#### 7-2. 실무적 문제
**그냥 2σ로 잡으면 정규분포 가정 하에 정상 소비의 약 5%가 경고로 뜬다.** 소비 변동이 큰 세그먼트라면 훨씬 더 뜬다.
→ **데모에서 경고가 도배되면 "이거 그냥 다 잡는 거 아니냐"는 질문을 받는다. 발표 주인공 기능이 그 자리에서 무너진다.**

#### 7-3. 채택한 접근 (→ Part I §5 ①)
- **금액은 로그정규 분포에 가깝다** → raw z-score는 부적절. `log(amount)`에 z-score를 걸거나,
  이상치에 강건한 **Modified Z-score** 사용: `0.6745 × (x − median) / MAD`, **임계 3.5**
- **고정 시간대 룰보다 사용자별 시간대 프로파일이 낫다** — 24시간은 원형 변수이므로 단순 비교 불가
- **단일 조건보다 AND 결합** — §3-1의 금감원 시나리오 구조 참고
- **임계치는 "최적화"가 아니라 "제약 충족"으로 정한다** — "더미데이터 100건 중 경고가 3~5건만 뜨도록" 역산

#### 7-4. 결정 시점
**세그먼트(페르소나) 확정 후, 더미데이터를 설계할 때 같이 정한다.**
더미데이터의 분포를 내가 만들기 때문에, **임계치와 데이터를 함께 설계해야 데모가 성립한다.**

### §8. 이 프로젝트가 구현할 것 / 안 할 것

| 구현 | 이유 |
|---|---|
| ✅ 룰 기반 탐지 (심야 고액 / 신규 카테고리 급증 / 빈도 이탈) | **AND 결합 구조로** |
| ✅ 통계 탐지 (Modified Z-score, 엔진의 median·MAD 재사용) | 공유 엔진 원칙 |
| ✅ 해시체인 + Merkle + TSA 앵커링 | 차별화 핵심 |
| ✅ 변조 탐지 시연 | 가장 강한 데모 |
| ❌ ML 모델 | 5주 일정 + 설명가능성 요구. 룰이 오히려 정답 (→ III-C §6) |
| ❌ 실시간 스트리밍 (Kafka/Flink) | 더미 거래 수백 건에 불필요 |
| ❌ Redis 벨로시티 카운터 | 동일 |
| ❌ 룰 엔진 프레임워크 (Drools/CEL) | 룰 3~5개에 프레임워크는 과잉 |
