# 최종 서비스 배포 실행 계획서 (launch plan)

> **문서 위상**: 결정의 단일 기준은 `reference/finntech_things.md`(마스터)다. 이 문서는 마스터 §13(마이데이터 도입)·§13-8(호스팅)·§13-11(대량 생성 준비)을 **실행 순서로 풀어낸 작업 계획서**로, 마스터와 어긋나면 마스터가 우선하고 이 문서를 고친다.
> 작성 2026-07-21. 계획이 실행되면 각 절의 상태를 갱신하고 말미 변경 이력에 남긴다.

## 0. 한 장 요약

- **현재**: 로컬 2서버(본체 8080 + 마이데이터 8082) + Vite 프론트가 전 기능 동작. 마이데이터 시드 12명(결정론), 온보딩→본인인증(가상 스텁)→연동→분석 전 과정 실측 완료. 테스트 72개 정의(기본 실행 71 통과, TSA live 1건은 `-Dtsa.live=true` 조건부). Phase 1~4 완료 — 단 KWCAG는 재설계 화면(온보딩·내 카드·차트) 한정, 잔여 4탭은 Phase 12(W6-4).
- **최종 목표**: 페르소나 기반 마이데이터 약 1,100만 건(Faker 생성·하루 활동/동선 개연성·**이상거래 자연 발생**·**낭비/필수 라벨**·온라인 채널·**상품(품목·가격)**·위치 포함)을 담은 마이데이터 서버와, 실 SMS 본인인증을 갖춘 본체를 **EC2+RDS(MySQL)·Docker·nginx·HTTPS**로 외부 공개 배포한다. 소비를 **낭비 vs 필수로 분류하는 해석가능 ML(EBM)이 규칙 FDS를 대체**하고, 미래 결제가 실시간으로 붙되 **쓸수록 낭비가 줄어드는 하강 곡선**("서비스 효과")으로 나타난다.
- **가는 길**: 8개 워크스트림(W1 페르소나·대량생성 / W2 실시간 전환 / W3 실 SMS / W4 운영 DB / W5 배포 / W6 품질 / W7 보안·규제 / **W8 낭비/필수 ML**). **순서 원칙(사용자 확정): 대량 데이터 완성 → ML 학습·통합 → 그다음 공개 배포**(배포 메커니즘은 로컬 총연습으로 먼저 위험 제거) — 상세는 §5 로드맵.

## 1. 현재 상황 진단 (2026-07-21 기준)

> 5개 영역(본체·마이데이터·프론트·인프라·데이터/규제)을 코드·실측 기준으로 전수 조사한 결과다. 근거는 `파일:줄` 형태로 남긴다.

### 1-1. 시스템 전경 (현재 동작 형상)

```
[브라우저] ── 5173 (Vite dev / nginx)

    ▼
[backend 8080 본체] ──RestClient──▶ [backend-mydata 8082 제공자]
  H2 ./data/finntech                  H2 ./data/mydata
  엔진·FDS·게임화·감사로그·개인정보     사용자12·카드56·결제2,165(실측)
```

### 1-3. 결론 — 무엇이 남았나 (한 문장)

**기능은 끝났고, "운영"이 안 됐다.** 남은 것은 (a) 페르소나 대량 데이터(이상거래 포함), (b) 실 SMS, (c) 운영 DB·컨테이너·배포, (d) 그리고 배포하는 순간 치명상이 되는 보안·정합 결함들의 해소다.

### 1-4. 공백 전수 목록 (배포 관점, G-번호로 워크스트림에 매핑)

**A. 배포하면 즉시 깨지는 것 (블로커)**

- **G-1 배포 산출물에 마이데이터 서버가 없다**: `docker-compose.yml`에 backend-mydata 서비스·`MYDATA_BASE_URL` 주입·`finntech_mydata` DB 생성이 전부 없고, `backend-mydata/Dockerfile`도 없다. compose로 올리면 온보딩(인증→연동)이 통째로 실패한다. → W5
- **G-2 dev-up.sh가 8082를 안 띄운다**: 스크립트 전문에 mydata 참조 0건. README_PROJECT.md:23과 마스터 §13-8("8080+8082 동시 기동")이 실제와 불일치 — 심사자가 README대로 실행하면 첫 화면에서 실패. → W5(선행 작업)
- **G-3 프론트 번들에 `http://localhost:8080` 하드코딩**: `api.ts:1`의 폴백이 dist에 박혀 있고(grep 실측) Dockerfile에 `VITE_API_BASE` ARG가 없다. 원격 배포 시 방문자 브라우저가 자기 로컬 8080을 호출 → 전 화면 통신 실패. nginx의 `/api` 동일오리진 프록시는 현 빌드에선 죽은 경로. → W5
- **G-4 CORS가 `localhost:5173` 하드코딩**(`AppConfig.java:30-32`) — 운영 도메인 설정화 안 됨. → W5·W7

**B. 배포하면 사고가 되는 것 (보안)**

- **G-5 무인증 IDOR 구조(최대 결함)**: Spring Security·세션·토큰 전무. 전 API가 클라이언트 제공 `userId`를 신뢰 — 타인 리포트 열람, 소비 조작, `DELETE /api/users/{id}/data`(전체 파기), `/api/privacy/purge`·`/api/audit/anchor` 같은 관리 작업까지 무인증. → W7
- **G-6 8082가 무인증으로 주민번호 13자리를 반환**: `UserView`가 socialNumber·phoneNumber를 그대로 담고(`MyDataDtos.java:14`), 무인증 curl로 전 거래+신원 수신을 실측 확인. `/ci/{ci}`로 존재 탐침도 가능. CORS는 서버-투-서버를 못 막는다. → W7·W5(내부망 격리)
- **G-7 H2 콘솔 활성+`sa`/빈 비밀번호+`AUTO_SERVER=TRUE`**: 포트 노출 시 원격 SQL 셸. compose의 DB 비밀번호 평문(`rootpw`), `.env` 체계 부재. → W4·W7

**C. 데이터·엔진 (대량 생성 전 선행)**

- **G-8 마이데이터 경로에 이상거래가 0건**: 시드가 08~21시·3천~5만원만 생성(`MydataSeedGenerator.java:156-159`) — FDS 시연이 본체 DUMMY_SEED에만 의존. 페르소나 생성 스펙에 이상 주입(σ-종속 발화 배수 — W1-2)이 반드시 필요. → W1
- **G-9 대량 조회 병목 후보**: `findCards`가 카드별 루프(N+1)+혜택 lazy 로딩+기간 하한·페이지 없음 — 페르소나 사용자당 수천 건이면 응답 수 MB. 본체 `analyze()`도 매 호출 전량 재계산. → W1·W4
- **G-10 대량 적재 경로 미비**: 내장 시드는 건별 `save`·단일 트랜잭션 — 11M 건 불가. JDBC 배치·외부 파이프라인 직적재 필요. `generation.enabled=true`면 내장 시드가 **카탈로그조차 안 만드는데** 외부 파이프라인의 카탈로그 소유 명세가 §13-11에 없다. → W1
- **G-11 페르소나 확정 후 재캘리브레이션 미실시**: z 3.5는 현 시드 분포 기준 실증값 — 새 분포에서 `/api/dev/calibrate` 재스윕 후 임계치 교체가 남은 절차(도구는 완비). → W1·W6

**D. 정합·품질**

- **G-12 법무 정본이 코드보다 뒤처짐**: 약관 제8조 "모든 금융상품은 더미"가 실 금리 정보성 비교와 불일치. 방침은 "4개 항목 수집·본인인증 없음"인데 실제는 CI 저장+마이데이터 카드·결제 적재. "고지한 약속=코드" 서사의 역방향 위반. → W7
- **G-13 MYDATA 동의·보유기간 정합 공백**: `linkCardCompanies`는 CI만 확인(동의 게이트는 프론트 약관 화면 의존), 보유기간 자동 파기는 USER_INPUT만 대상(`PrivacyService.java:125-126`) — MYDATA 보유 정책이 코드에 없다. → W7
- **G-14 backend-mydata 테스트 0건**: 커트오프 경계·replace 결정론·CI 산식 본체 일치 전부 무테스트. 프론트 테스트도 0건. → W6
- **G-15 운영 스키마 관리 부재**: 양쪽 다 `ddl-auto: update` 전 프로파일 공용, 마이그레이션 도구 없음. → W4
- **G-16 운영성 기타 — 처분표** (매핑 화살표가 본문과 1:1이 되도록 항목별 처분 명기):
  - **(해소)** CI/CD → W5-6 / TLS → W5-3 / 모니터링·actuator(마이데이터) → W5-5·W5-2 / TSA 앵커 실행 절차 → W5-5 관리자 런북(내부 실행으로 nginx 타임아웃 회피; 비동기화·15초 대기 하향은 후순위) / **localStorage 게이트와 서버 상태 불일치 → W2-2 ⑨**(서버 상태 교차 확인 — W1 replace 후 빈 화면 방지).
  - **(의도적 수용 — 이번 계획 범위 밖, 사유 명기)** 분석지표 인메모리(코드 주석의 전제대로 "수집 후 즉시 내보내는" 운영 절차로 커버, 릴리스 전 백업 조회) / 요청 로깅·트레이스 ID(데모 규모에선 컨테이너 로그로 충분) / 라우터 부재(단일 데모 사용자·탭 UI 특성상 수용 — 한계로 문서화) / USER_ID=1 하드코딩(공용 데모 계정 구조의 의도된 단순화 — W5-7 부트스트랩이 id=1 생성을 검증).

### 1-6. 강점 (지킬 것)

- 결정론·설정 외부화·공유 `AnalysisResult`·PII 리플렉션 테스트·감사 3계층 — **원칙이 코드로 강제되는 구조**는 이미 완성. 모든 후속 작업은 이 구조를 훼손하지 않는 방향으로만 한다(§3).
- 커트오프+`/renewal` 증분, replace 모드, generation 스캐폴드, persona 컬럼 — **대량 생성의 착륙 지점이 이미 준비**돼 있어 W1은 "만드는" 일이지 "고치는" 일이 아니다.

## 2. 최종 목표 정의 (Definition of Done)

배포가 "끝났다"고 선언하려면 아래 전부가 참이어야 한다.

1. **외부 접근**: 공개 URL(HTTPS)에서 온보딩 → 본인인증 → 카드사 선택 → 마이데이터 연동 → 홈/내 카드/내 소비/혜택·저축/더보기 전 탭이 동작한다.
2. **데이터**: 마이데이터 서버가 페르소나 기반 대량 데이터(목표 약 1,100만 건, 과거+미래 결제, **결제별 품목·가격·위치·낭비/필수 라벨 포함**)를 서빙하며, `mydata.now=system`으로 미래 결제가 시간 경과에 따라 등장한다. 데이터는 하루 일정·동선을 모사한 개연성 있는 소비이며 학습/검증/테스트/서비스 셋으로 분리돼 있다.
3. **낭비/필수 ML 판정 시연**: 해석가능 ML(낭비 vs 필수 분류)이 실제 소비를 분류하고 "왜 낭비인지"를 설명하며, 시간이 지날수록 낭비 판정이 줄어드는 "서비스 효과"가 드러난다. 규칙 FDS는 비교 baseline으로 남고, 사용자 피드백으로 개인별 재분류가 반영된다.
4. **본인인증**: coolsms 실 SMS 발송(사용자 발급 키) 또는 명시적 스텁 모드 중 하나로 설정되며, 어느 쪽인지 문서·화면에 고지된다.
5. **격리**: 마이데이터 서버(8082)는 외부에서 직접 접근 불가(내부 네트워크 전용), 본체만 공개.
6. **재현성 유지**: 엔진은 여전히 `Clock`·`referenceTime` 주입으로 동작하고, 대량 생성은 시드 고정으로 재현 가능하다.
7. **규제·개인정보**: 더미 상품 고지·정보성 비교 문구("가입은 각 사로")·마이데이터 시뮬레이션 고지가 화면에 있고, `legal/` 정본과 `PrivacyService.policy()`가 정합하며, AppUser에 실 PII가 저장되지 않는다(테스트로 강제).
8. **품질 게이트**: 백엔드 테스트 전체 통과(본체 + 마이데이터 모듈), 프론트 `tsc`+빌드 통과, 릴리스 회귀 체크리스트 완주.
9. **운영 최소선**: 헬스체크 엔드포인트, 컨테이너 자동 재시작, DB 백업 1회 이상 검증, 롤백 절차 문서화.

## 3. 변하지 않는 제약 (마스터 §4 재확인)

계획 전체에 걸리는 가드레일. 어떤 워크스트림도 이를 깨는 방식으로 구현하지 않는다.

- 판단은 **설명가능한 모델(규칙 또는 해석가능 ML)**이, 표현은 AI가(집계만 전송) — 마스터 §4 원칙 1 **개정 예정**(W8/Phase 8 착수 시 확정). 낭비/필수 판정을 해석가능 ML로 대체하되(W8) "왜 낭비인지"를 계수·특징중요도로 설명한다. black-box 금지. **그 전까지 마스터 §4 원칙 1(판단은 코드가)이 유효**하며, 본 계획은 그 개정을 요구한다.
- 하나의 `AnalysisResult`를 세 서비스가 재사용(임계치 재계산 금지).
- 재현성: 시드 고정 `Random`, `Clock` 주입, `now()` 직접 호출 금지, `TreeMap`.
- 카테고리 이름 코드 비박제, 임계치는 전부 `application.yml`.
- 추천·판매 흐름 금융상품은 전부 더미. 정보성 비교만 실 금리(무판매목적·무제휴·가입편의 없음).
- AppUser에 전화번호 등 실 PII 저장 금지(CI만) — `PrivacyFlowTest`가 강제.
- 팀 저장소의 `README.md`(클래스룸 원본)는 절대 수정하지 않는다.

## 4. 워크스트림 상세 설계

> 각 워크스트림은 "무엇을(설계) → 어떻게(절차) → 됐는지(검증 기준)"로 닫힌다. 현황 근거는 §1을 참조하고 여기서는 반복하지 않는다.

---

### W1. 페르소나 정의 + 하루 활동·동선 기반 마이데이터 대량 생성 (낭비/필수 라벨·상품·위치)

> **전제(변경 불가)**: 아직 데이터는 추가하지 않는다. §13-11의 "준비만 된 상태"를 실행으로 옮기는 **향후 실행 계획**이며, 실행 전까지 어떤 값도 바꾸지 않는다. 실행의 첫 관문은 **사용자의 페르소나 확정**이다.

#### W1-0. 생성 패러다임 (개연성 있는 하루 · 낭비/필수 라벨 · 데이터 분리)

**전역 설정 변경**: "위치 상세 미포함(가맹점명만)"을 **"포함"으로 변경** — 결제에 주소·좌표를 싣는다.

**① 하루 활동·동선 시뮬레이션 (끼워맞추기 금지)**: 소비를 카테고리 분포에서 독립 추출하지 않는다. 페르소나마다 **하루 일정**(기상·출근·점심·오후·저녁·주말 시퀀스)과 **동선**을 모사하고, 소비는 그 흐름의 **자연스러운 부산물**로 발생시킨다. 동선은 **공개 상권/POI 데이터(주소·좌표·업종)**와 **페르소나별 활동지역(집·직장 등)**에 앵커링한다(예: 출근길 집근처 편의점 → 점심 직장근처 식당 → 저녁 배달/외식 → 주말 활동지역 쇼핑). 효과: ㉠ **특정 분야 집중이 자연 방지**되고 ㉡ **이론 검증용 억지 주입이 사라진다** — 이상거래(심야 고액·치팅데이)도 "실제 있을법한 일상"에서 발생한다.

**② 거래 스키마 확장** (향후 `MyDataPayment`에 컬럼 + Flyway 마이그레이션): 기존 필드 + **품목명·가격** · **채널(ONLINE/OFFLINE)** · **위치(주소·좌표·POI)** · **낭비/필수 정답 라벨**.

**③ 낭비/필수 라벨 (다요인 잠재 — 순환 방지)** — **★ 재량 ≠ 낭비(사용자 확정 2026-07-22)**: 각 거래에 낭비 vs 필수 정답을 부여하되, **재량성(discretionary)과 낭비성(waste)은 다른 축**이다. 재량성은 "생존필수인가?"(무대 판정)일 뿐이고, **낭비는 '충동·과다·후회성'에서만** 나온다.
  - **필수 무대**(생존·생활필수: 식료품·마트·편의점·약·공과금·교통·통신·주유) → 거의 낭비 아님(`p_waste≈0.02`).
  - **재량 무대**(외식·쇼핑·배달·구독·취미·여가) → `impulse = Σ(심야·계획안됨·평소대비과다·최근빈도급증·구독누수·배달과다·치팅)`, **본인 취미의 비과다 지출이면 `× hobbyProtection(0.2)`로 보호** → `p_waste = clamp(impulse × 페르소나 충동성 × improvementCurve(t))` → 베르누이 draw.
  - **재량성 점수는 무대 판정·금액 분포에만** 쓰고 **p_waste에 직접 넣지 않는다**(재량이라는 이유만으로 낭비 아님). 상품 재량성은 오프라인 웹/LLM 1회 산출·캐시(예: "쌀 10kg"→필수 무대, "한정판 스니커즈"→재량 무대이나 낭비 여부는 충동·과다로 결정). 단일 규칙 금지(ML 되학습 방지) — 다요인+노이즈 유지.
- **정의(사용자 확정)**: **필수(비낭비)** = 생존·생활필수·정기·계획 **+ 본인 취미 지출 + 계획·가치 있는 재량**. **낭비** = **충동·과다·후회성**(계획 안 된 충동구매·평소 대비 과다·심야 폭소비·구독 누수·배달 과다·치팅). ⚠ 취미는 낭비 아님(단 과다·충동이면 낭비 가능). 조건은 `mydata.generation.label`에 인코딩.

**④ 상품 카탈로그**: 가맹점별 품목·가격 목록 + 재량성 점수. 가맹점·상품명은 브랜드 일반명만(외부 출처 표기 금지 — 규칙).

**⑤ 데이터 분리 (과적합·누수 방지)**: 생성 사용자를 **학습/검증/테스트/서비스(데모) 4셋으로 사용자 단위 disjoint 배분**(예 60/15/15/10, 확정 수치는 실행 시). **앱이 서빙·시연하는 데이터(서비스셋)는 ML 학습셋과 겹치지 않는다.** ML 평가(val/test)도 분리. → W1-3 규모 산정을 이 파티션으로 재배분.

**⑥ 위치·개인정보 주의**: 합성 페르소나의 위치는 더미라 무해하다. 단 **향후 실사용자가 실 마이데이터(위치 포함)를 연동하면 위치이력은 실 PII** → 그 시점에 `legal/`·`PrivacyService.policy()` 갱신 트리거(W7과 연계).

이하 W1-1~W1-7은 위 패러다임 위에서 파라미터·규모·검증을 구체화한다. 낭비 판정 자체는 **W8(해석가능 ML)**이 담당하고, 여기서는 그 학습·평가에 쓸 **라벨 있는 현실적 데이터**를 만드는 것이 목표다.

- **σ_log의 정의(중요)**: 엔진은 **카테고리별** log 금액에 Modified Z를 걸므로, σ_log는 페르소나 전체의 소비 다양성이 아니라 **z 산정에 실제 걸리는 카테고리 내 산포**다. 페르소나 간 다양성은 categoryMix·카테고리별 중앙값·건수로 표현한다. 범위는 **0.20~0.30** — 하한은 본체 시드의 실증 산포(건당 ±30%, log-σ≈0.15~0.2)와의 정합, **상한 0.30은 배수 상단에서 역산**한 값이다(z>3.5 돌파에 필요한 배수 M≥e^{3.5σ}이므로, 상단 M≈3.0이 유효하려면 σ≤ln(3.0)/3.5≈0.314). 이 범위를 벗어나면 W1-2의 주입 배수 전제가 무너진다(σ=0.45면 M≥4.8배, σ=0.8이면 M≥16배 필요).

#### W1-1a. 온라인 결제 채널 (개정 4)

실제 카드/마이데이터에서 온라인 결제는 **카테고리를 가로지르는 성격**이다(배달=식비·쇼핑몰=쇼핑·스트리밍/게임=여가·구독·간편결제). 이를 1급 차원으로 반영한다.

- **용어 주의(정합)**: 대분류 **"온라인"**(categoryMix의 7대분류 중 하나)과 **채널 ONLINE**(카테고리를 가로지르는 결제 방식)은 **별개 축**이다 — 예: "식비-배달"은 대분류는 식비지만 채널은 ONLINE.
- **채널 속성**: 결제마다 `channel ∈ {ONLINE, OFFLINE}`. `MyDataPayment`에 채널 컬럼(운영 validate → Flyway 마이그레이션, W4-2·부록 A). 서빙 DTO 노출은 표시 개선(후순위).
- **배정 규칙(결정론)**: `P(ONLINE) = clamp(카테고리 기본 온라인성 × 페르소나 onlineRatio)`. 기본 온라인성은 `PersonaCatalogData` 상수(스트리밍·구독·배달·쇼핑몰 높음, 편의점·카페 낮음). `onlineRatio`는 **성향 계수**(실현 온라인 비중 자체가 아님 — 실현 비중은 categoryMix와 곱해져 계수보다 낮게 나옴, W1-7 검증 참조). 값은 W1-1 표(절약 0.30·균형 0.45·과소비 0.70·구독과다 0.75·외식 0.60; 높은 3종 0.60~0.75). 서브시드 `Random`으로 재현 고정.
- **가맹점 강화**: 카탈로그에 온라인 가맹점(배달앱·종합쇼핑몰·오픈마켓·스트리밍·구독·간편결제) 충분히 포함(브랜드 일반명, 외부 출처 표기 금지).
- 채널은 W8 ML의 **입력 특징**이자 하강 곡선(W1-1b)의 주 감소 대상(온라인 충동구매). **판정 자체는 ML(W8)**이 하며, 유지되는 규칙-FDS baseline에서 채널은 새 룰이 아니다.

#### W1-1b. 미래 데이터 하강 곡선 — "서비스 효과" (개정 2)

이 서비스는 쓸수록 충동·낭비 소비가 **줄어야** 하므로, 생성 데이터가 그 서사를 담는다. 사용자 확정 궤적: 과거 낭비 많음 → 미래 감소(비단조).

- **과거 180일 = 안정 기준선**: 정상 소비 + 낭비 spike를 정상 비율로. ML 학습 라벨·규칙-FDS baseline 캘리브레이션이 여기서 측정된다.
- **미래 90일 = 하강 궤적(비단조)**: `ImprovementCurve`가 낭비(미계획/충동) 발생률을 낮춘다 — 점감 → **1개월차 최저** → 1.5개월차 살짝 반등 → **2개월차부터 랜덤 유지(수치는 가변, 가끔 낭비로 걸림)**. **무조건 감소 아님** — 변동적으로 늘어나는 날·**치팅데이(갑자기 폭증)** 포함. 페르소나별 개선 속도 차등(절약형 빠름, 과소비형 완만). **필수소비는 전 구간 일정.**
- **효과**: 커트오프가 "지금"을 지나며 최근월 낭비가 줄어 → **ML의 낭비 판정 감소·소비건강점수 상승·절약통 성장**이 시간과 함께 자연히 나타난다(게임화 §5-5 서사와 맞물림).
- **결정론**: 곡선은 **사전 각본**(같은 시드 → 같은 궤적). 앱 사용에 실시간 반응하지 않는 정적 곡선(재현성).
- **누수 방지(W8과)**: 이 시간 궤적은 **데이터 구성**에만 반영되고, "서비스 시작 후 개월 수"는 ML 특징으로 쓰지 않는다(W8-1). 개선은 데이터에서 자연 드러나고 모델은 개별 거래만 본다.
- **baseline 캘리브레이션 정합(주의)**: 규칙-FDS baseline의 z 3.5 → 3~5/100 캘리브레이션은 **과거 기준선 분포에서만** 측정. 하강 곡선은 시간 오버레이라 임계치 산정에 개입하지 않는다(W1-7 검증에서 분리 확인).

#### W1-2. 낭비 이상소비 — 자연 발생 + 규칙-FDS baseline 정합 (G-8 해소)

> **역할 재정의(개편)**: 주 판정은 **ML(W8)**이 담당한다. 이 절은 두 역할만 갖는다 — ① 낭비 spike(심야 충동구매·치팅데이·배달 폭주)가 **하루 활동 모델에서 자연 발생**하도록 하는 **현실성 가드레일**(W1-0 ① — 억지 주입·끼워맞추기 아님, 일상의 극단으로 등장), ② 유지되는 **규칙-FDS baseline**(W8 비교용)이 의미를 갖도록 하는 **통계적 정합**(σ-종속 배수). 즉 아래 수치는 "z를 맞추려 주입"이 아니라 "자연 발생하는 낭비 spike의 크기가 baseline에서 탐지 가능한 현실 범위에 있도록"의 가드레일이다.

규칙-FDS baseline 발화 조건은 **z>3.5 플래그 AND 룰 ≥1**(`AlertService.java:40-49`). 자연 발생한 낭비 spike의 금액이 그 범위에 들도록 σ-종속 배수로 가드한다(baseline이 완벽·무의미해지지 않게). 동시에 이 spike들은 전부 **낭비 라벨**(W8 학습 정답)로 기록된다.

| 룰(`Enums.java:19`) | 시나리오 | 주입 사양 |
|---|---|---|
| `NIGHT_HIGH_AMOUNT` (A) | 심야 고액 일탈(저녁 모임 과지출의 극단) | 시각 00~05시, 발화군 금액 = baseline 중앙값 × **M ∈ [max(2.0, e^{3.5σ_cat}×1.05), e^{3.5σ_cat}×1.35]** — σ-종속 배수(σ=0.20→약 2.1~2.7배, σ=0.30→약 3.0~3.9배)로 룰의 ×2.0 조건과 z 돌파를 **동시에** 보장. 발화 경계(e^{3.5σ}) 아래 1.5배~경계×0.95 구간은 **비발화 대조군**(라벨 expected_alert=false) — 캘리브레이션 커브의 경계를 만든다 |

**배수 체계의 근거(캘리브레이션 실증, 마스터 §12-2)**: 6~10배 주입 시 임계 2.0~5.5 전 구간 결과 동일(10.81건/100건) — 임계치가 구속조건이 아니게 된다. 경계 강도(1.5~3.0배) 재시드에서 z>3.5가 3.23건/100건으로 목표(3~5) 적중 — 이 실증은 본체 시드의 산포(log-σ≈0.15~0.2) 위에서 성립한 것이므로, 페르소나 생성에서는 배수를 고정 구간이 아니라 **σ-종속**으로 잡아 같은 원리를 이식한다. 수리: 로그정규에서 MAD_log≈0.6745σ이므로 Modified Z의 0.6745 계수가 상쇄되어 **z≈ln(M)/σ_cat, 발화 경계 M*=e^{3.5σ_cat}**(σ=0.20→2.02, 0.25→2.40, 0.30→2.86). 발화군은 경계×1.05 이상, 대조군은 경계×0.95 이하로 잡아 발화/비발화가 임계 양쪽에 걸리게 한다 — 임계치가 실제 판별력을 갖는 구조. **생성 전 필수 선행 작업: 페르소나×카테고리별 (σ_cat, M, 기대 z) 검산표를 작성**해 ① A·C의 발화 배수가 임계 양쪽에 걸리는지 ② NIGHT 룰의 "중앙값×2.0" 금액 조건과 z 돌파의 동시 충족 ③ B는 σ_cat이 아닌 **사용자 전역 σ_global**로 검산 ④ 구독 지배 카테고리의 실효 MAD를 확인하고, **미달 페르소나는 배수 또는 σ를 W1-1 표에서 먼저 수정한다**(검산표가 표에 우선 — 드라이런이 아니라 스펙 단계에서 잡는다).

- 검산: (35+55+90+60+70)/5=62건 × 20,000명 × 9개월 = **11.16M ≈ 1,100만**. 현행 `target-count` 11,000,000 대비 +1.45%이므로 실행 시 **`target-count: 11160000`으로 조정**해 W1-7 기준 1의 검증 창(11.0M~11.2M)과 기계 검사가 어긋나지 않게 한다.
- **미래 결제 배분**: 9개월 중 미래 3개월 ≈ **372만 건**이 기준일 이후 90일에 배치 → 커트오프가 시점을 지나며 순차 등장(실시간 시뮬레이션, W2).
- **데이터 분리 배분(W1-0⑤·W8)**: 20,000명을 **사용자 단위로 학습/검증/테스트/서비스(데모) = 60/15/15/10**(예)로 disjoint 배분 → 학습 12,000명·검증 3,000·테스트 3,000·서비스(데모) 2,000명. **데모 사용자(앱이 서빙)는 ML 학습셋과 겹치지 않는다**(누수·과적합 방지). 데모 신원(고정 풀)은 서비스셋에 둔다.
- 카드: 사용자당 3~6장(현행 설정 유지), 월건수를 주력카드 60/25/15% 가중 배분. `prev_month_amount`는 현행 난수 대신 **직전월 실제 합**으로 계산해 실적구간 혜택과 정합(개선점).
- 저장량 추정: 결제 ~250B × 11M ≈ 2.8GB + 인덱스 ≈ 총 4GB 내외.

#### W1-4. 생성기 구현 위치와 결정론

| 대안 | 판정 | 근거 |
|---|---|---|
| **① backend-mydata 내 프로파일 러너(전용 프로파일) + Faker** | **채택** | 엔티티·`Catalog`·`Ci` 재사용, `generation.enabled` 훅 완비(내장 시드가 스스로 비켜남). 값 생성은 Faker(이름·가맹점·금액류)에 위임 |
| ② 별도 모듈/스크립트(CSV→LOAD DATA) | 기각 | 스키마·카탈로그·CI 로직 중복 정의, 결정론 이원화 |

- **값 생성은 Faker(Java `datafaker`)로, 시드를 주입해 결정론을 유지**한다. `new Faker(new Random(userSeed))`처럼 사용자 서브시드로 초기화하면 같은 시드 → 같은 값이므로 규칙 3(재현성)을 지킨다 — "무시드·`now()` 금지"는 시드 주입으로 충족되며, Faker를 배제할 이유가 아니다(마스터 §13-5 "Faker → 결정론"과 정합). Faker는 이름·가맹점 후보·금액 노이즈 등 **값 채우기**만 담당하고, 시각·미래 결제·PK·카테고리 배정은 여전히 `reference-date` 오프셋·`Catalog` 데이터·결정론 규칙이 소유한다(판단·구조는 코드가, 값은 Faker).
- 신설 패키지 `com.finntech.mydata.generation`: `PersonaProfile` / `PersonaCatalogData`(5종 상수) / `PersonaGenerationRunner`(`@Profile("generation")`+`enabled=true` 조건, 완료 후 컨텍스트 종료) / `GenerationJdbcWriter`(배치 적재) / `GenerationLabel`(엔티티). 의존성 `net.datafaker:datafaker`를 `backend-mydata/pom.xml`에 추가(생성 프로파일 전용).
- 실행: `SPRING_PROFILES_ACTIVE=mysql,generation` **1회성 잡**(서빙 기동과 분리). 카탈로그는 파이프라인이 "없으면 생성"을 소유한다(§13-11의 명세 공백 보완 — 현행 코드에서 generation 모드는 카탈로그조차 만들지 않으므로).
- **마스터 개정 선행**: 이 선택(모듈 내 Faker 기반 결정론 러너)은 마스터 §13-11의 "외부 Faker 파이프라인이 적재" 서술과 소유 주체가 다르다 — **착수 전 마스터 §13-11을 "backend-mydata 내 generation 프로파일의 Faker 기반 결정론 러너(시드 주입·1회성 잡)"로 개정**한 뒤 실행한다(개정 항목: 소유 주체·카탈로그 생성 책임·`personas[]` 키 처분, 규칙 1). §13-5(Faker→결정론)와는 그대로 정합.
- 설정: **기존 키 재사용** — `enabled`·`future-window-days`·`target-count`(11,160,000으로 조정)·`personas[]`(라벨 열거 검증용으로 유지, **파라미터 진실 소스는 `PersonaCatalogData`**임을 주석 명기) + **신설 5종** — `seed: 20260721`, `users-per-persona: 4000`, `past-window-days: 180`, `batch-size: 10000`, `demo-users-per-persona: 3`.
- **결제 PK 스펙(단일 확정)**: 결정론 고정폭 문자열 `g{페르소나인덱스}-{사용자6}-{순번5}`(40자 제한 준수). 사용자 블록 오름차순 생성·삽입으로 클러스터드 인덱스 **append 지역성**을 확보한다(시간 정렬이 아니라 블록 정렬 — W1-5·W4-3의 삽입 지역성 요구를 이 스킴 하나로 충족). W1-2 라벨 재현·W1-7 체크섬도 이 스킴 기준.
- **결정론**: 마스터 시드 → `userSeed = mix64(masterSeed, personaIndex, userIndex)` 서브시드 → 사용자마다 **`Random`과 그 `Random`으로 초기화한 `Faker` 한 쌍**(병렬 생성·부분 재생성에도 결과 불변 — 현행 단일 Random 순차 소비는 2만 명 병렬화 불가). 이름·가맹점 후보는 Faker의 시드 결정론 출력, CI는 기존 `Ci.of` 재사용. 시각·미래 결제 전부 `reference-date` 오프셋 — `now()` 금지.

#### W1-5. 성능 설계

- **JPA 우회**: 건별 `save`는 12명용. 1,100만 건은 `JdbcTemplate.batchUpdate`(배치 10,000행, 10만 행 청크 커밋), 사용자 단위 생성→즉시 flush.
- **H2 불가·MySQL 전용**: 11M 행이면 MVStore 파일 수 GB 팽창·컴팩션 정지·손상 시 전체 유실 → 대량 생성은 **운영 `mysql` 프로파일에서만**. 개발 H2는 12명 시드 용도로 유지.
- **MySQL 적재**: JDBC URL `rewriteBatchedStatements=true`, 적재 세션 한정 `unique_checks=0`·`foreign_key_checks=0`. 결제 PK는 W1-4 확정 스킴(`g{p}-{사용자6}-{순번5}`) — 블록 오름차순 삽입으로 클러스터드 인덱스 지역성 확보.
- **인덱스**: 조회는 카드별 date range뿐이므로 기존 `idx_mydata_payment_card_date` 하나로 충분 — 인덱스 있는 채 적재 감당 가능. 라벨 테이블은 PK만.
- **소요 추정**(리허설로 실측 확정): 적재 2만~4만 행/초 가정 시 INSERT 5~10분, 잡 전체 15~40분. 수용 기준 60분.

#### W1-6. replace 교체 런북과 롤백

1. **드라이런**: 1/100 스케일(`users-per-persona:40`)로 전 절차 리허설+속도 실측.
2. **백업**: `mysqldump --single-transaction finntech_mydata > mydata-YYYYMMDD.sql`.
3. 서빙 정지 → `mysql,generation` 프로파일 잡 기동(내장 시드 자동 회피).
4. replace 의미론: **라벨→payment→card→user 순 삭제, 카탈로그 보존** → 데모 신원(고정 풀, persona 라벨 포함) → 사용자→카드→결제(과거180+미래90)→라벨 순 생성. 대량 상태에서의 재교체는 `deleteAllInBatch` 대신 **해당 세션에서 `SET FOREIGN_KEY_CHECKS=0` → TRUNCATE 라벨→payment→card→user → `SET FOREIGN_KEY_CHECKS=1`**(MySQL은 FK 제약이 걸린 테이블의 TRUNCATE를 참조 테이블이 비어 있어도 거부하므로 순서만으로는 실패한다).
5. W1-7 검증 통과 후 `enabled=false` 복귀, 통상 기동(`mode=keep`).
6. **본체 정합**: 기존 연동 계정은 재인증·재연동(전체 동기화가 기존 MYDATA 데이터를 지우고 재적재). CI가 바뀌므로 필수. 방문자 브라우저의 localStorage 게이트는 **W2-2 ⑨(서버 상태 교차 확인)**가 온보딩으로 회귀시킨다 — 이것이 없으면 replace 직후 공개 데모가 빈 화면이 된다.
7. **재캘리브레이션(G-11)**: `/api/dev/calibrate` 스윕 → `modified-z-threshold` 확정("스윕 역산" 서사 유지). 확정 임계치의 **근거(새 분포 기준 스윕 결과)는 마스터 §12-2·D-11에 갱신**한다 — 상수 값의 근거는 마스터 소관(CLAUDE.md 규칙 1의 단서)이며 tech_log는 구조 변경분만. 수정 보고(규칙 2) 동반.
8. **운영 지속 조건**: 미래 창은 기준일+90일에 소진된다 — Phase 7(대량 데이터) 완료 시 **소진 예정일을 기록**하고, 시연이 그 이후까지 계속되면 같은 러너를 reference-date만 이동해 재실행(시드 유지 → 과거 구간 결정론 재현, W1-7 체크섬으로 확인) 후 재연동한다. 이때 본체의 `finntech.mydata.reference-date`·sync 바닥값 키(`floor-days` 기준일)도 함께 이동한다. 시연 종료가 그 이전이면 생략.

**롤백**: 1차 = 백업 복원. 2차(백업 유실) = **결정론 복원** — 단, 대량 데이터 위에서 내장 replace의 `deleteAllInBatch`를 타면 안 된다(W4-3이 금지한 undo 로그 폭발과 동일 작업). **① 먼저 해당 세션에서 `SET FOREIGN_KEY_CHECKS=0` → TRUNCATE 라벨→payment→card→user → `SET FOREIGN_KEY_CHECKS=1` ② 그 다음** `generation.enabled=false`+`mode=replace`+seed 20260721로 1회 기동하면 기존 12명이 비트 단위 원상 재생성된다(같은 시드+기준일→같은 데이터). 재현성(규칙 3)이 백업 없는 롤백을 가능케 하는 안전판. 본체는 재연동 1회.

### W2. 실시간 연동 시뮬레이션 운영 전환

> 현행: 커트오프 3모드·`/renewal`·`findCardsSince` 클라이언트까지 구현됐으나 **호출자가 0곳**이고, `lastRenewalTime`·연동 카드사의 저장 자리가 본체에 없다. W2는 "반쯤 놓인 다리의 본체 쪽 절반"을 완성한다.

#### W2-1. reference→system 전환 절차

`mydata.now`는 생성자에서 1회 읽으므로 **재기동이 전환의 전부**다. 파일 무수정 — 환경변수로 덮어쓴다:

```bash
pkill -f backend-mydata-0.0.1-SNAPSHOT.jar
MYDATA_NOW=system nohup java -jar target/backend-mydata-*.jar > /tmp/finntech-mydata.log 2>&1 &
```

- 데이터는 재생성되지 않는다(`mode: keep`) — 전환은 순수하게 조회 커트오프만 바꾼다. 데이터 불변·시점만 이동이 규칙 3 관점의 핵심.
- 현 12명 시드는 미래 결제가 없으므로 system 모드는 기준일이 지나면 "전부 보이고 더는 안 늘어나는" 상태 — **지속되는 실시간감은 W1의 미래 90일 적재(372만 건)로 완성**된다.
- `dev-up.sh` 확장 시 `MYDATA_NOW` 통과 지원(기본 reference).

#### W2-2. 증분을 당기는 시점 — 온디맨드+수동(A+C), 스케줄러 기각

| 대안 | 판정 |
|---|---|
| A. 프론트 진입 시 온디맨드 sync | **주 경로 채택** — 부하가 사용량 비례, 볼 때만 최신이면 충분, 테스트에서 호출 시점 완전 통제(재현성) |
| B. 백엔드 `@Scheduled` 폴링 | 기각 — 대량 사용자로 가면 상시 배치, 백그라운드 쓰기가 테스트 결정론과 마찰, 데모 통제 상실. 대량 국면에서 재검토 |
| C. 수동 새로고침 버튼 | **보조 채택** — 데모에서 등장 순간을 명시 연출 |

- 신규 `POST /api/mydata/sync?userId=` — 프론트는 ① 앱 로드 ② '내 카드' 탭 진입 시 호출. 서버측 쿨다운(`finntech.mydata.sync.min-interval-seconds: 60`, 원칙대로 yml)으로 중복 no-op. `MyCardPanel`에 새로고침 버튼+"+N건" 표시.

**저장 자리 신설 — `UserCardCompany` 엔티티** (마스터 §13-3 데이터 모델에 이미 예약된 자리):
`id, userId, companyId, companyName, linkedAt, lastRenewalTime(not null)` + `(userId, companyId)` 유니크. AppUser 컬럼 1개 대안은 기각 — 카드사별 증분 시각이 뭉개져 부분 실패 시 누락·중복이 생기고, AppUser는 PII 최소화 대상이라 연동 메타를 불리지 않는다. 운영은 validate이므로 신설은 **Flyway `V2__user_card_company.sql`을 동반**하고, 배포는 **마이그레이션 선적용 → 앱 교체** 순서다(W4-2·W5-4의 스키마 호환 제약).

**증분 적재 로직(신규 `renew(userId)`)**:
1. CI 없으면 no-op. `UserCardCompany`를 companyId 오름차순 순회(정렬 고정).
2. 회사별 `findCardsSince(companyId, ci, lastRenewalTime)` 호출.
3. 수신 결제를 (paymentDate, paymentId) 오름차순 재정렬 후 적재(결정론적 삽입 순서).
4. **멱등 키 = `UserPayment.paymentId`(@Id)**: 존재하면 스킵, 신규만 UserPayment 저장+`Consumption(MYDATA)` 투영(전체 동기화와 동일 경로 — 카테고리 온디맨드 생성). UserPayment 존재 여부가 이중 투영 방지의 유일한 게이트임을 주석으로 명시.
5. 카드 실적(`currentPerformance`)은 증분 가산이 아니라 **DB 재계산**(당월 sum) — 몇 번 돌려도 같은 값.
6. **`lastRenewalTime := max(수신 paymentDate, 기존값)`** — 벽시계 미사용. 두 서버 시계가 어긋나도, 시간여행 중에도 공백·중복 없음. 빈 응답이면 안 움직여 멱등.
7. 초기값: `linkCardCompanies` 완료 시 upsert + 그 시점 적재분의 max(paymentDate). **적재분 0건 경계**: 결제가 0건인 카드사는 고정 바닥값 — **본체 신설 키 `finntech.mydata.sync.floor-days: 180`**(본체 `finntech.mydata.reference-date` − floor-days의 자정; backend-mydata generation의 past-window-days=180과 값 정합 유지 주석)으로 upsert한다. lastRenewalTime은 not null(8082 `/renewal`의 필수 파라미터이기도 함)이고, 바닥값이 데이터 시작보다 일러도 다음 renew가 전 구간을 조회할 뿐 paymentId 멱등(④) 덕에 안전하다. 벽시계 대입은 금지(⑥ 원칙 훼손+커트오프 이후 등장할 과거분 영구 누락). 재연동 벌크삭제 목록과 `PrivacyService.eraseUserData` 삭제 목록에 `UserCardCompany` 추가(파기권 정합).
8. 응답 `{newPayments, newCardsTouched, skippedByCooldown}`.
9. **온보딩 게이트 서버 교차 확인(G-16 해소)**: 프론트 진입 시 localStorage 플래그 단독이 아니라 **`myCards` 빈 응답 기준**으로 교차 확인(현 API로 즉시 가능 — CI 유무는 API에 노출 수단이 없어 채택하지 않는다) — 서버가 미연동이면 플래그를 무시하고 온보딩으로 회귀한다(W1-6 6단계의 replace 후 빈 화면 방지).

#### W2-3. 재현성 × 실시간의 긴장 해소 — "엔진은 주입, 서빙만 실시간"

**분석 엔진과 판정 경로는 한 줄도 바꾸지 않는다.** 실시간성은 세 곳에만 존재한다:

실시간성이 존재하는 세 곳 = **제공자 커트오프 · 동기화 renew · 서빙 기준월**. 엔진·투영은 불변임을 표에 함께 보인다:

| 계층 | 처리 |
|---|---|
| 엔진 `analyze(userId, referenceTime)` | **불변.** 호출부는 이미 주입 Clock 경유 — 테스트는 고정 Clock으로 동일 입력→동일 출력 |
| 제공자 커트오프 `cutoff()` | 유일한 시점 권위. 데이터 불변, now만 이동. ISO 고정 시 완전 결정론 |
| 동기화 `renew()` (신규) | 벽시계 미사용 — lastRenewalTime은 수신 데이터에서 유도. 쿨다운 판정만 주입 Clock |
| 서빙 기준월 | 신규 스위치 `finntech.mydata.time-mode: reference(기본)\|system`. reference면 현행 그대로(기존 테스트 불변), system이면 `YearMonth.from(LocalDate.now(clock))` — 주입 Clock 경유만 |
| 투영 | 증분도 같은 `Consumption(MYDATA)` 경로 → 세 서비스는 다음 analyze 때 하나의 `AnalysisResult`로 자동 반영(원칙 2). FDS 반영은 기존 `/api/alert/rescan` 재사용(판단 로직 신설 없음) |

#### W2-4. 데모 시나리오 — ISO 고정 "시간여행"

```bash
# T0. '오전'으로 고정 기동 → 온보딩·연동 (오전 커트오프라 오후 결제가 안 보임)
MYDATA_NOW=2026-07-21T09:00:00 java -jar target/backend-mydata-*.jar &
# T1. '저녁'으로 재기동 (데이터 재생성 없음 — keep 모드 로그로 증명)
pkill -f 'backend-mydata.*jar'   # 기존 인스턴스 종료 없이는 8082 포트 충돌로 기동 실패 (글롭 인용 필수 — zsh)
MYDATA_NOW=2026-07-21T22:00:00 java -jar target/backend-mydata-*.jar &
# 기동 대기: until curl -fsS localhost:8082/actuator/health; do sleep 1; done (T0에도 동일)
# T2. '내 카드' 새로고침 → POST /api/mydata/sync → "+N건", 실적 게이지·홈 요약 갱신
# T3. FDS 반영: POST /api/alert/rescan → 새 소비가 하나의 AnalysisResult로 흘러듦
```

내레이션: "서버를 재기동했지만 DB는 그대로다 — 바뀐 것은 '지금이 몇 시인가' 하나. 낮 동안의 결제가 증분으로 흘러들어온다." 마지막에 `MYDATA_NOW=system`이면 같은 메커니즘이 벽시계를 따라 도는 운영 모드다.

#### W2-5. 검증 기준

- **제공자**: ① 단조성 `P(t1) ⊆ P(t2)`, 차집합 결제일이 전부 (t1,t2] ② 같은 ISO 반복 조회 응답 동일 ③ `findCards(t2)` = `findCards(t1)` ∪ `findCardsSince(t1→t2)`, 교집합 공백.
- **본체**: ④ 보존식 — 전체 재연동 == 초기 link+증분 누적(건수·sum·투영 3중 일치) ⑤ 같은 sync 2회 → 2회차 newPayments=0 ⑥ UserPayment 수 == Consumption(MYDATA) 수 ⑦ 카드사 2곳 중 한쪽만 갱신돼도 다른 쪽 lastRenewalTime 무오염 ⑧ 쿨다운 내 재호출 부작용 0 ⑨ 파기 후 UserCardCompany 0건.
- **재현성**: ⑩ 기존 테스트 전부 통과, `engine/` diff 0 ⑪ 고정 Clock+ISO 조합 통합 테스트 반복 동일 ⑫ 신규 코드에 `now()` 직접 호출 0건(grep).
- **UI**: ⑬ T0→T2 시나리오 실측.

완료 시 tech_log "실시간 시뮬레이션(커트오프/증분/시점 3계층)" 갱신 + 마스터 §13-11 상태 정정(규칙 1).

### W3. coolsms 실 SMS 본인인증 전환

#### W3-1. 흐름 — 2단계(요청→검증) 전환, 스텁과 공존

```
[SMS 모드] request(신원) → 세션 생성(코드·신원·TTL, 서버 메모리) → SMS 발송(또는 로그 폴백)
           confirm(코드) → 만료/시도 검사 → 일치 → 세션에 봉인된 신원으로 CI 계산
           → AppUser.ci 연결 → checkCi → VerifyResult → 세션 즉시 파기
[스텁 모드] 기존 POST /api/mydata/verify = verifyAssumed 그대로 (기본값·현행 유지)
```

핵심: **confirm 시 CI는 request 때 세션에 봉인한 신원으로 계산** — "코드를 수신한 그 전화번호"가 CI에 들어간다는 무결성 성립(다른 번호 재제출 우회 차단).

| 엔드포인트 | 역할 |
|---|---|
| `GET /api/mydata/verify/mode` | `{mode: STUB\|SMS}` — 프론트가 코드입력 단계 노출 결정 |
| `POST /api/mydata/verify` (기존) | STUB 전용. SMS 모드에서는 409(우회 차단) — 숨은 롱프레스 스킵이 SMS 모드에서 자동 봉인되는 의도된 안전 동작 |
| `POST /api/mydata/verify/request` | 형식검증(`social7` `^\d{7}$`, `phone` `^01[016789]\d{7,8}$`) → `{expiresInSeconds, resendAfterSeconds, delivery: SMS\|LOG_FALLBACK}` |
| `POST /api/mydata/verify/confirm` | 성공: 기존 `VerifyResult` / 실패: 400+`{remainingAttempts}` |

backend-mydata는 무변경 — 인증은 전부 8080 소관.

**코드 저장소**: **인메모리 채택** — 전화번호가 디스크에 닿지 않아 방침 정합 최상, 코드 수명 3분이라 재시작 소실 무해, 현 단계 단일 인스턴스(Redis는 마스터가 "향후"로 명시). DB 저장은 기각(전화번호 영속 = 방침 정면충돌). `VerificationStore` 인터페이스로 추상화(후일 Redis 이음새), 만료 판정은 주입 `Clock`, 지연 삭제+`@Scheduled` 스윕 이중 정리. 세션 키는 userId(활성 세션 1개, 재요청 시 교체).

#### W3-2. coolsms 연동·설정

- 의존성 `net.nurigo:sdk`(버전·전이 의존성 무충돌은 구현 시 확증 — 충돌 시 REST API v4를 기존 `RestClient`로 직접 호출·HMAC 서명, 신규 의존 0).
- 환경변수 `SMS_KEY`/`SMS_SECRET_KEY`/`SENDER` — **사용자 직접 발급**, 코드·저장소·yml 기본값에 절대 금지(타 공개 저장소 노출 키 재사용 금지). 키 하나라도 비면 `LogFallbackSender`: 앱은 죽지 않고 코드를 로그로 출력(기동 WARN 1회, 응답 `delivery: LOG_FALLBACK`, 로그 전화번호 마스킹 `010****2005`).
- `SmsSender` 인터페이스 → `CoolsmsSender`/`LogFallbackSender`(Gemini `hasKey` 선례와 동일 분기).

```yaml
finntech:
  auth:
    sms:
      enabled: false                # false=가상 스텁(기본), true=실 SMS 2단계
      api-key: ${SMS_KEY:}
      api-secret: ${SMS_SECRET_KEY:}
      sender: ${SENDER:}
      code-length: 6
      code-ttl-seconds: 180
      resend-cooldown-seconds: 60
      max-attempts: 5
      max-sends-per-phone-per-day: 5
      cleanup-cron: "0 */5 * * * *"
```

수치 전부 설정값(원칙 4). `application-test.yml`은 `enabled: false` 유지 → 기존 테스트 무수정.

#### W3-3. 코드 생성기 분리(재현성)와 정책 수치

- `VerificationCodeGenerator` 인터페이스 — 운영 `SecureRandomCodeGenerator`(보안 난수가 맞고, 재현성 원칙의 적용점은 "주입 가능한 이음새"), 테스트는 `FixedCodeGenerator` 빈 오버라이드. 만료·쿨다운은 `Clock.fixed`로 결정론 검증.
- 정책: 6자리/TTL 180초(시도 5회 제한과 결합해 무차별 대입 차단), 재발송 쿨다운 60초, 세션당 시도 5회 초과 시 파기, 전화번호당 일일 5건(과금 보호), 5분 주기 스윕.

#### W3-4. 개인정보 정합 — 전화번호의 수명과 문서 동시 갱신

**전화번호는 인증 세션 동안만, 서버 메모리에서만.** 어느 경로로든 세션과 함께 소멸, DB에는 CI만. **어떤 엔티티에도 phone 필드를 추가하지 않는다** → `PrivacyFlowTest` 리플렉션 가드와 충돌 없음. 감사로그에는 이벤트(`VERIFY_CODE_SENT`/`VERIFY_CONFIRMED`)와 userId·결과만 — 전화번호는 마스킹조차 넣지 않는다(해시체인은 영구 잔재).

`PrivacyFlowTest` 비충돌 논증: 조항 수 7개 유지(기존 조항 본문 수정), 금지어 문구 유지, `policy()`는 `sms.enabled` 주입으로 모드별 분기 — 스텁 모드(테스트) 문안은 현행 그대로라 기존 테스트 무수정 통과.

**한 커밋으로 묶을 갱신 8건**: ① 방침 §2 수집항목(일시 이용·메모리 3분·DB 무저장·즉시 파기) ② 방침 §4 **SMS 발송 위탁 예외**(수탁자·항목·범위) ③ 방침 §5·6 보유기간 ④ `PrivacyService.policy()` 정합 ⑤ 약관 제4조 **및 제8조 2항**+`terms()` — 제4조는 "연동 시에만 SMS 인증"으로, 제8조 2항의 "본인인증 등 라이선스 기능 미제공" 문구는 "가상/SMS 인증은 마이데이터 시뮬레이션 연동 확인 목적이며 금융 라이선스 본인확인 서비스가 아님"으로 개정 ⑥ CLAUDE.md 문구 ⑦ 마스터 §13-2·13-4·13-9·변경이력+tech_log(규칙 1) ⑧ `AuthService`·`AppUser.ci` javadoc.

#### W3-5. 검증 기준

- **스텁 회귀(자동)**: 기존 전 테스트 무수정 통과. 신규 `AuthSmsFlowTest`(고정 생성기+Clock, enabled=true+키 미설정): 정상 경로 CI=세션 신원 / 다른 번호 재제출 우회 불가 / 오코드 5회→파기 / TTL 경과→거부 / 쿨다운 거부 / 키 미설정→LOG_FALLBACK 무예외 / SMS 모드 단발 verify 409 / **인증 후 전 테이블 전화번호 문자열 부재**. 신규 `PrivacySmsPolicyTest`: SMS 모드 문안 검증.
- **실발송 수동(키 보유자)**: 실 단말 수신(180초 내)·쿨다운·오코드/만료 거부·콘솔 발송건수 확인·종료 후 DB 덤프에 전화번호 부재. 실 신원은 시드에 없으므로 `existsInMyData=false`가 정상 — 전체 온보딩 확인은 dev 한정 수동 INSERT 절차 별도 문서화(실 전화번호를 시드 코드에 커밋하지 않기 위함).

구현 순서: 설정+`AuthConfig` → `service/auth/` 8클래스 → `AuthService`·컨트롤러 확장 → 프론트(mode 분기 코드입력 단계) → 테스트 2종 → 문서 8건 → 수동 시나리오.

### W4. 운영 DB 전환(H2→MySQL)과 대량 데이터 대비

> 대량 데이터(약 1,100만 건) 관련 항목은 전부 **W1 실행 시점의 향후 계획**이며, 그 전까지 어떤 데이터도 추가하지 않는다. 두 모듈의 MySQL 프로파일·드라이버·DB 이름 분리(`finntech`/`finntech_mydata`)는 **이미 준비돼 있다** — 남은 일은 인프라에서 두 번째 DB를 실제로 만들고 스키마 관리를 도입하는 것.

#### W4-1. 두 DB 분리 구성

마이데이터 제공자는 개념상 별도 기관 — 프로세스가 분리됐듯 데이터도 물리 분리한다. MySQL 컨테이너는 **1개** 유지(포트폴리오 규모에서 2개는 낭비), 두 번째 DB는 init 스크립트로 만든다.

- `deploy/mysql-init/01-create-mydata-db.sql`: `CREATE DATABASE IF NOT EXISTS finntech_mydata CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;` + `finntech` 계정 GRANT. (init은 볼륨 최초 생성 시에만 실행 — 기존 `mysql-data` 볼륨이 있으면 `down -v` 후 재기동 필요.)
- **RDS 계층**: 단일 인스턴스 + 스키마 2개. 논리 분리를 **계정 권한으로 표현** — `finntech_app`(finntech.*만) / `mydata_app`(finntech_mydata.*만) 두 계정으로 "본체가 마이데이터 DB를 직접 읽는" 실수를 DB가 차단(§13 구조를 권한 계층이 강제). RDS 3306은 EC2 SG에서만 인바운드. 파라미터 그룹(신설): `character_set_server=utf8mb4`, `collation_server=utf8mb4_unicode_ci`, `time_zone=Asia/Seoul`.

#### W4-2. 스키마 관리 — `ddl-auto: update`의 운영 위험과 Flyway (G-15)

**왜 운영에서 update가 금지인가**: ① 컬럼명 변경=새 컬럼 추가로만 반영(고아 데이터), 타입 축소·삭제·인덱스 삭제는 아예 미반영 ② **이 저장소가 이미 겪은 enum 사고**(네이티브 ENUM에 MYDATA 추가 시 update가 ALTER를 못 만들어 깨짐 → `prefer_native_enum_types: false` 도입) ③ 1,100만 건 적재 후 임의 시점 ALTER가 수 분짜리 온라인 DDL ④ `audit_log` 해시체인 테이블이 도구에 의해 암묵 변경되는 것 자체가 감사 스토리와 배치.

| 방안 | 판정 |
|---|---|
| A. 운영도 update | 기각(위 위험 전부) |
| B. validate + 수동 DDL 스크립트 | 차선(이력·순서가 사람 손) |
| **C. Flyway** | **권장** — Boot 자동 통합, SQL 그대로, 버전·체크섬 이력, 대량 적재용 인덱스 조작도 마이그레이션으로 기록 |
| D. Liquibase | 기각(이 규모에 과함) |

- **운영(mysql): `ddl-auto: validate` + Flyway.** 개발(h2)·테스트(create-drop)는 현행 유지(재현성과 무관한 편의 영역).
- 실용 판단: §13-11에서 인덱스 재구성·페르소나 교체가 예정돼 있고 "재현성"이 이 프로젝트의 불변 원칙이므로 **스키마의 재현성도 Flyway로 확보하는 것이 일관적**. 도입 반나절 이하.
- **V1 베이스라인**: 로컬 MySQL에 update로 1회 기동 → `mysqldump --no-data`로 두 DB의 V1 작성 → `src/main/resources/db/migration/` 배치 → mysql 프로파일 `flyway.enabled: true`+`ddl-auto: validate`. 주의: dump에서 `@Column(length=1_000_000)`이 **LONGTEXT**로 나오는지 확인(H2=VARCHAR, MySQL=LONGTEXT; validate가 이 매핑 차이를 기동 시점에 잡는 안전망).

#### W4-3. 대량 대비 인덱스·스토리지 (향후)

- **필요한 복합 인덱스는 이미 있다**: `findByCardUpTo/Between`은 "동치(card_id)+범위(payment_date)+같은 컬럼 정렬"이라는 교과서 형태이고 `idx_mydata_payment_card_date`가 정확히 커버. MySQL 8은 `ORDER BY ... DESC`를 Backward index scan으로 처리(filesort 없음). **추가 인덱스 불요** — EXPLAIN으로 확인만.
- **문자열 PK 완화(스키마 변경 없이)**: 현행 PK는 VARCHAR(40)이라 무작위 채번이면 세컨더리 인덱스 비대+페이지 분할이 문제 — 대량 생성은 **W1-4 확정 스킴(`g{p}-{사용자6}-{순번5}`, 블록 오름차순)**으로 발급해 컬럼 타입 그대로 삽입 순서만 단조로 만들어 페이지 분할을 제거한다. (BINARY(16) 전환은 효과 크나 스키마·코드 변경이므로 후순위.)
- **적재 절차(향후)**: 인덱스 DROP → 정렬 CSV `LOAD DATA` 또는 JDBC 배치 → `ADD INDEX`(온라인, 일괄 정렬 생성이 삽입 중 유지보다 수 배 빠름).
- **replace 주의(향후)**: 대량 상태에서 `deleteAllInBatch` 3연쇄는 undo 로그 폭발 → 해당 세션에서 `SET FOREIGN_KEY_CHECKS=0` → **TRUNCATE**(라벨→payment→card→user) → `SET FOREIGN_KEY_CHECKS=1`로 대체(W1-6과 동일 문구 — MySQL은 FK 제약 존재만으로 TRUNCATE를 거부).
- **응답 페이로드 함정(G-9)**: `findCards`는 커트오프 이전 전체를 실어 나른다 → 최초 연동 후 `/renewal` 증분(W2)만 쓰고, 전체 조회에 조회 하한(최근 N개월) 파라미터 추가를 대량 실행 시점의 선행 작업으로. 임계·기간값은 yml.

#### W4-4. utf8mb4·타임존·커넥션 풀

- **utf8mb4**: 서버·RDS 파라미터·양 DB·init 콜레이션을 `utf8mb4_unicode_ci`로 통일(MySQL 8.4 기본 `utf8mb4_0900_ai_ci`와 섞이면 "Illegal mix of collations"). 카테고리·가맹점·페르소나 라벨이 전부 한글이라 필수. 검증에 한글 왕복 저장 포함.
- **타임존**: 엔티티는 `LocalDateTime`↔`DATETIME`(존 변환 없음)이지만, `mydata.now=system`은 **JVM TZ로 커트오프를 계산**하므로 컨테이너/EC2가 UTC면 9시간 어긋나 "미래 결제 등장" 시연이 틀어진다. 통일: 컨테이너 `TZ: Asia/Seoul`, JDBC `serverTimezone=Asia/Seoul`(이미 있음), RDS `time_zone=Asia/Seoul`, JVM `-Duser.timezone=Asia/Seoul`. 엔진 재현성(Clock 주입)은 그대로 — 인프라 계층 정합일 뿐.
- **HikariCP**: 명시 없음(기본 최대 10). 앱 2개 합산 20, **db.t4g.small(W5-3 확정 사이징)의 max_connections 수백** 대비 여유 → **기본값 유지가 정답**. 다만 운영 프로파일에 의도 문서화: `maximum-pool-size: 10`, `connection-timeout: 3000`(장애 조기 노출), `max-lifetime: 570000`. 대량 적재는 앱 풀 밖 별도 세션.

#### W4-5. enum 매핑 — `prefer_native_enum_types=false`

Hibernate 7이 `@Enumerated(STRING)`을 네이티브 ENUM DDL로 생성하려 해 상수 추가 시 update가 깨진 사고를 겪었다. `false`면 VARCHAR 매핑 → 값 추가가 자바 코드 변경만으로 끝난다(DB 값 제약 상실은 판단이 코드에 있으므로 허용 트레이드; 원하면 Flyway로 CHECK 추가). 운영 전환 체크 2건: ① 이 설정이 mysql 프로파일에도 적용 → V1 dump가 VARCHAR인지 확인(validate가 잡음) ② `@Enumerated` 사용처 8곳의 `length` 여유 확인(`Consumption.source` length=20, 최장 CARD_UPLOAD 11자). backend-mydata는 현재 enum 엔티티가 없으나 방어적으로 동일 설정 추가.

#### W4-6. 전환 절차 (로컬 검증 → RDS)

0. backend-mydata Dockerfile 확인/신설(W5), 드라이버는 양 pom에 이미 있음.
1. compose 보강(W4-1의 init.sql·mydata 서비스·`MYDATA_BASE_URL` — W5와 공유). 기존 볼륨 있으면 `down -v`.
2. 로컬 MySQL 프로파일 검증: `docker compose up --build` → W4-8 체크리스트. 시드 12명 결정론·keep 재기동 무중복 확인.
3. 스키마 베이스라인: `mysqldump --no-data`로 V1 → Flyway 도입 → validate 전환 → validate 기동 성공 확인. **첫 전환의 함정**: 이미 테이블이 있는 DB에 Flyway를 켜면 "Found non-empty schema without schema history table"로 기동 실패한다 — (a) 전환 1회 한정 `spring.flyway.baseline-on-migrate: true`+`baseline-version: 1` 적용 후 제거, 또는 **(b) 권장: `down -v`로 빈 스키마에서 V1부터 적용+시드 재생성**(결정론이라 무손실). 운영 yml에 baseline-on-migrate를 상시로 켜두지 않는다(빈 스키마 오판 방지).
4. RDS 생성·연결: MySQL 8.4, 파라미터 그룹, 스키마 2개+계정 2개, EC2 SG만 인바운드. **코드 변경 없이 환경변수만으로 전환**(현 yml 설계 취지).
5. **데이터 이행 안 함**: 현 H2는 전부 결정론 더미 → 시드 **재생성**이 가장 안전한 "이행"(H2→MySQL dump 변환 불요). 감사로그는 **최초 전환 시 MySQL에서 새 체인으로 시작**하고 시연 전 `TSA_ENABLED=true`로 앵커를 새로 만든다 — H2의 기존 체인을 옮길 수단은 없고 필요도 없다. W4-7의 "스냅샷 복원"은 전환 **이후** 운영 중 복구에만 적용된다.

#### W4-7. 백업·복구 최소 정책

| 대상 | 정책 |
|---|---|
| RDS 자동 백업 | 보존 7일, 새벽(KST) 윈도, PITR |
| 수동 스냅샷 | 시연(리허설·최종 데모) 직전 각 1회 |
| 복구 우선순위 | 더미 영역은 **시드 재생성**(결정론), 단 `audit_log`·`AuditBatch`(TSA 앵커)는 **스냅샷 복원** — 재생성하면 체인·앵커가 달라짐(`demo-tamper.sh regenerate`가 보여주는 논리와 동일) |

#### W4-8. 검증 기준

1. 양 모듈 `./mvnw test` 통과(H2 인메모리). MySQL 대상 자동화(Testcontainers)는 선택.
2. h2 프로파일 기동 성공 / mysql 프로파일 두 앱 `validate` **기동 성공**(스키마·엔티티 정합의 기계적 증명), actuator/health UP.
3. 기능 스모크(MySQL): dev-up 시드→분석·알림 정상, 마이데이터 전체·증분 조회 정상, 한글 왕복 무손상, reference에서 동일 요청 2회 완전 일치.
4. 인덱스 실증: EXPLAIN에서 `key=idx_mydata_payment_card_date`, `type=range`, `Backward index scan`, filesort 없음.
5. 응답시간: 현 시드 전체 연동 조회 p95<300ms, 증분<100ms. **대량 이후(향후)**: 목표치는 **W6-3(S1~S5)이 단일 정본** — 미달 시 W4-3의 조회 하한 도입을 발동한다(이 절에 별도 수치를 두지 않는다).
6. 타임존: system 전환 후 커트오프가 KST 기준 동작 확인 후 reference 복원.
7. **백업 복원 리허설(DoD 9의 "검증")**: 수동 스냅샷에서 임시 RDS 인스턴스로 복원 → validate 기동 + `GET /api/audit/verify` 무결 확인 1회 후 인스턴스 폐기 — 백업은 복원해 보기 전까지는 백업이 아니다.

### W5. 컨테이너·배포 아키텍처 (최종 서비스 배포)

> D-03("배포 실패가 시연 실패가 되면 안 된다")에 따라 **로컬 총연습 → EC2 이전** 2단계. 실측상 **새로 쓸 것은 `backend-mydata/Dockerfile` 하나**이고 나머지는 보강·신설이다.

#### W5-1. 컨테이너 구성 완성 (G-1·G-3·G-4 해소)

**설계 원칙**: ① 외부 공개는 프론트 nginx 하나(`/`→SPA, `/api/`→backend:8080, 이미 nginx.conf에 구현). Actuator는 `/api` 밖 경로라 이 프록시로 외부 도달 불가 → 내부 헬스체크 전용 자동 보장. ② **8082는 어디에도 발행 안 함**(ports 없음+internal 망+SG 무규칙 3중 차단). ③ 본체→마이데이터는 서비스명 DNS(`MYDATA_BASE_URL=http://backend-mydata:8082`).

`docker-compose.prod.yml`(신규; 기존 compose는 로컬 준비물로 유지) 핵심:
- `mysql`: `internal` 망만, ports 미발행, init 스크립트로 `finntech_mydata` 생성.
- `backend-mydata`: **현재 없는 서비스 신규 추가**, `mysql` 프로파일, ports 없음, healthcheck(actuator 추가 후).
- `backend`: `internal+edge` 망, ports 없음, **`MYDATA_BASE_URL=http://backend-mydata:8082`(G-1 필수 해소)**, `FINNTECH_DEV_SEED_ENABLED=false` 하드코딩, `GEMINI_API_KEY` 미설정 시 템플릿 폴백.
- `frontend`: build arg **`VITE_API_BASE=""`(빈 문자열→상대경로→동일 오리진, G-3 해소)**, `127.0.0.1:5173:80` 바인딩(호스트 nginx만 접근).
- `internal: {internal: true}` 망 → mysql·backend-mydata는 아웃바운드조차 없음. backend는 `edge`로 Gemini·TSA·금리조회 아웃바운드 유지. (복잡도 부담 시 단일 망+포트 미발행만으로도 요구 충족 — 차선.)
- json-file 로깅 `max-size:10m, max-file:5`를 4서비스 전부에.
- **로컬/RDS 겸용**: `mysql` 서비스에 `profiles: ["local-db"]` — 로컬 총연습은 `--profile local-db`로 기동, EC2는 프로파일 없이 기동하고 `DB_HOST`=RDS 엔드포인트를 `.env`로 주입(불필요한 mysql 컨테이너가 EC2 메모리를 잠식하지 않게). backend의 `depends_on: mysql`은 로컬 오버라이드 파일로 분리. **prod compose에는 고정 `container_name`을 쓰지 않는다** — blue-green 2벌 기동 시 이름 충돌로 green이 못 뜬다.
- 기존 로컬 `docker-compose.yml`에도 최소 2건 반영: backend-mydata 서비스 추가(로컬은 8082 발행 허용 — 디버깅), backend에 `MYDATA_BASE_URL`. (규칙 1·2 대상.)

#### W5-2. Dockerfile 작성·보강

| 파일 | 조치 |
|---|---|
| `backend-mydata/Dockerfile` | **신규** — 본체와 동일 멀티스테이지(temurin 17-jdk 빌드→17-jre 실행), curl 설치·비루트(uid 1001)·`-XX:MaxRAMPercentage=60`, EXPOSE 8082 |
| `backend/Dockerfile` | 보강 — curl 설치(healthcheck용, JRE에 없음)·비루트·메모리 |
| `frontend/Dockerfile` | 보강 — `ARG VITE_API_BASE=""`+`ENV` (api.ts의 localhost 폴백이 운영 번들에 박히는 것 차단; 빈 문자열은 nullish 아니라 상대경로) |
| `.dockerignore` ×3 | 신규 — backend/backend-mydata: `target/`,`data/`,`*.mv.db` / frontend: `node_modules/`,`dist/`(빌드 컨텍스트 축소, 로컬 H2 파일 유입 방지) |
| backend-mydata Actuator | 코드 변경 1건 — pom에 `spring-boot-starter-actuator`, yml `management.endpoints.web.exposure.include: health`. 준비 전 임시 헬스체크는 `curl .../bank/mydata/card-company` |

#### W5-3. EC2 + RDS 배치

```
[브라우저] ─443(HTTPS)→ [EC2 호스트 nginx(TLS 종단, certbot)] ─→ 127.0.0.1:5173
                          └ Docker: frontend ─/api/→ backend(8080) ─RestClient→ backend-mydata(8082)
                                    (edge)         (internal+edge)      (internal, 완전 비공개)
                                                          └─3306→ [RDS MySQL: finntech / finntech_mydata]
```

- **사이징**: 현행(시드 12명) EC2 **t3.medium(2vCPU/4GB)+스왑2GB** + RDS **db.t4g.small**, gp3 20GB(JVM 2개+nginx+Docker; t3.small은 EC2 내 build 시 OOM 위험). §13-11 대량 실행 시 RDS **db.t4g.medium**로 스케일업, gp3 30~50GB(스토리지 오토스케일만 미리 켬).
- **8082 차단 3중**: compose ports 미발행 + `internal: true` 망 + SG 8082 인바운드 부재. 8080·3306·5173도 외부 규칙 없음(5173은 127.0.0.1 바인딩이라 커널 수준 차단).

#### W7-2. 8082 격리 + 본체↔마이데이터 최소 인증 (G-6)

- **격리(필수, DoD)**: compose에 8082 `ports` 없음(내부 망 전용), EC2 SG 8082 미개방, nginx는 프론트+8080 프록시만(`/bank/**`는 외부에 존재하지 않게 됨). 8082 CORS는 운영에서 제거/무력화(브라우저가 8082를 직접 호출하는 코드가 없음). 운영에서 양 모듈 `h2.console.enabled=false` 명시.
- **공유 시크릿 헤더 — 채택(경량)**: 격리는 설정 실수 1건이면 무너지는 단층 방어이고, "제공자-사업자 간 인증"은 실 마이데이터 구조의 핵심이라 서사에도 부합, 비용은 필터 1개+헤더 1줄. 환경변수 `MYDATA_SHARED_SECRET` → 본체 `RestClient`에 `.defaultHeader("X-MyData-Token", secret)`, 마이데이터 `OncePerRequestFilter`가 `/bank/**` 불일치 시 401(**actuator 등 `/bank/**` 밖 경로는 제외** — 컨테이너 헬스체크 유지; 필터 도입 시점에 W5-2의 임시 card-company 헬스체크는 actuator/health로 교체 완료돼 있어야 한다). **미설정 시 dev(h2) 통과·운영(mysql) 기동 실패(fail-fast)** — "운영 기본값은 안전해야 한다" 관행. `dev-up.sh`는 로컬 고정 토큰 주입. 완료 시 tech_log 갱신.
- **데이터 최소화(2차 방어 — 리스크 대장 R2의 "마스킹")**: 본체가 `CardView.user`의 socialNumber·phoneNumber를 실제로 소비하지 않음을 확인한 뒤 제공자 `UserView`에서 두 필드를 **제거**(불가피하면 뒤 6자리 마스킹) — 격리가 뚫려도 주민번호 13자리가 평문으로 나가지 않게 한다. 응답 계약 변경이므로 W6-2 와이어 스냅샷 JSON에 반영.

#### W7-3. CORS 운영 전환·rate limit·시크릿 (G-4·G-7)

- **CORS**: 권장은 **동일 오리진**(nginx가 프론트 서빙+`/api` 프록시 → 교차 출처 소멸, CORS 불필요). 그래도 `AppConfig`의 하드코딩은 프로퍼티화(`finntech.cors.allowed-origins`, 운영은 env 주입). 프론트는 동일 오리진 시 `VITE_API_BASE=''`.
- **rate limit(nginx 계층만; 코드 라이브러리는 데모 규모에 과설계로 기각)**: rate는 `limit_req_zone`(http 컨텍스트)에서 정의하고 location에서는 zone 참조만 가능하다 — http 블록 `limit_req_zone $binary_remote_addr zone=verify:10m rate=5r/m;` + verify location `limit_req zone=verify burst=3 nodelay; limit_req_status 429;`(**W3 실 SMS 도입 순간 발송 비용·폭탄 방어선**; `limit_req_status` 없으면 기본 503이라 W7-6의 429 검증이 통과 불가). 전역 완만한 limit(`zone=api rate=60r/m`)도 같은 2단 구조로 외부 호출 유발(anchor·통장비교)을 커버. `/api/dev/**`·`/h2-console`·`/actuator/**` 외부 차단.
- **시크릿**: 목록 표 유지(`GEMINI_API_KEY`/`DB_PASSWORD`·`MYSQL_ROOT_PASSWORD`/`MYDATA_SHARED_SECRET`/htpasswd/(W3 이후)SMS 키). `.gitignore`에 `.env`·`.env.*` 추가, `.env.example`만 커밋. 운영 프로파일에서 `${DB_PASSWORD:finntech}` 기본값 제거(미설정 시 기동 실패), compose 하드코딩을 `${...}`로. gitleaks 1잡+최초 전체 이력 스캔 0건 기록.

### W8. 낭비/필수 해석가능 ML — 규칙 FDS 대체 (신설)

> 사용자 결정: 규칙기반 FDS를 **해석가능 ML**로 대체해 각 소비를 **낭비 vs 필수**로 분류한다. 마스터 §4 원칙 1을 *"판단은 설명가능한 모델(규칙 또는 해석가능 ML)이"* 로 개정하는 근거 절. 학습 데이터는 W1이 만든 **라벨 있는 데이터**(하루 활동 기반·데이터 분리).

#### W8-0. 위상·불변식
- **라벨 의미(W1-0 ③ 정합)**: 낭비 = **충동·과다·후회성**이지 재량 자체가 아니다(재량≠낭비). 본인 취미는 보호. 재량성 점수는 특징으로 쓸 수 있으나 라벨의 근거는 충동·과다 신호(심야·평소대비·빈도·누수).
- ML = **주 판정**(production judgment). 규칙 FDS(Modified Z, §12-2 실증)는 **비교 baseline으로 문서 보존**(삭제 X).
- **다운스트림 불변(원칙 2)**: 낭비 판정 → 절약통·게임화·소비건강점수·리포트(기존 경로 재사용, 판정 소스만 규칙→ML).
- **설명가능성(원칙 1)**: "왜 낭비인지"를 계수·특징기여로 제시. black-box 금지.

#### W8-1. 특징 (feature engineering) — 누수 금지
- 금액: log 금액, **개인 평소 대비**(카테고리 중앙값 대비 배수·z류).
- 시간: 시각·요일 **주기 인코딩**(sin/cos)·심야 플래그.
- 채널·계획여부(가능 시)·**상품 재량성 점수**(W1 웹/LLM 산출)·가맹점·업종·**위치 유형**(집근처·직장근처·기타).
- 행동: 최근 빈도(RFM류)·같은 카테고리 최근 지출.
- **금지 특징**: "서비스 시작 후 개월 수"·미래정보 등 **누수(leakage) 특징 금지** — 개선 추세는 데이터 구성에서 자연 발생, 모델은 개별 거래만 본다.

| GBM(LightGBM/XGBoost)+SHAP | 중(전역 불투명, SHAP로 국소 설명) | 최상 | PMML/ONNX 가능 | **정확도 비교 벤치마크**(상한 재고) |
| 로지스틱 회귀(L1/L2) | 최상(계수=로그오즈 기여) | 보통~양호 | 최상(PMML 또는 수식 직접) | 단순 baseline·후퇴 옵션(§6) |

- **프로덕션 = EBM**(사용자 결정): 특징별 학습 곡선·쌍상호작용을 갖는 **글래스박스**라 우리 라벨(다요인 잠재+상호작용, W1-0③)의 구조에 가장 부합하면서 "왜 낭비"를 특징 기여 합으로 제시(원칙 1). 신용카드 사기탐지 선례로 도메인 정합.
- **비교 = GBM+SHAP**: 정확도 상한을 재어 "글래스박스로도 정확도 격차가 작음"을 실증(포트폴리오 강점). 로지스틱은 최소 baseline/후퇴 옵션.

#### W8-3. 학습 파이프라인 (Python / scikit-learn)
- **데이터 분리(W1-0⑤)**: 사용자 단위 disjoint 학습/검증/테스트 + 서비스셋 격리(앱은 학습 데이터로 데모 안 함). 사용자 그룹·시간 누수 차단.
- **인간 라벨 검수(요구 8)**: 학습셋 서브셋(수천~수만)을 사람이 전수 확인·수정(오분류 제거) → CSV/간이 UI 검수 도구. 검수 완료 라벨로 학습.
- **불균형**: 낭비/필수 비율이 시간곡선에 따라 변함 → `class_weight` 또는 임계 조정, **PR-AUC** 중심 평가(불균형 적합).
- **확률 캘리브레이션**: Platt/isotonic로 낭비 확률 보정.
- **평가**: 정밀도·재현율·F1·PR-AUC·혼동행렬, **페르소나별·시간대별 분해**. 사용자 피드백 루프 고려해 **정밀도(오탐 최소)** 우선.
- 교차검증 + 최종 test셋 1회 평가(오버피팅 방지).

#### W8-4. Java 통합 (Python 학습 → Java 추론)
- **EBM(프로덕션)**: interpretML은 Python 중심이라 PMML/jpmml 지원이 약하다. EBM은 결국 **덧셈 모델**(특징별 구간→기여값 테이블 + 쌍상호작용 테이블 + 절편 → 시그모이드)이므로, 그 테이블을 내보내 **작은 Java 스코어러 `SpendingClassifier`를 직접 구현**한다(룩업+합+시그모이드 — 단순·결정론). 비교용 로지스틱·GBM은 **PMML**(sklearn2pmml→jpmml-evaluator) 또는 ONNX(onnxruntime-java)로 배포.
- 본체의 판정 경로(현 `AlertService`의 규칙 FDS)를 ML 추론으로 교체. 규칙 FDS는 baseline으로 남겨 비교 API·문서에 보존.
- 재현성: 모델 산출물(EBM 테이블·PMML)은 버전·시드 고정, 배포는 정적 아티팩트(추론 결정론). Java 스코어러 == Python EBM 예측 일치(W8-6).

#### W8-5. 운영 개인화 층 (사용자 피드백 재분류 — 요구 10)
- 실사용자 조회 시 ML이 **1차 분류**(낭비/필수 + 확률 + 설명).
- 사용자가 오분류 지적·제외 → 그 **카테고리/가맹점/상품을 그 사용자에 한해 재분류·확정** → **개인 override 저장**(예: `UserSpendingOverride`).
- 이후 **실시간 유입 결제**는 override 우선 → ML → 최종. "통념상 낭비지만 본인엔 필수"인 분야 보호.
- override는 개인정보·파기 흐름에 포함(동의 철회·삭제 시 함께 처리).

#### W8-6. 검증 기준
- 학습/검증/테스트/서비스셋이 사용자 단위로 **교집합 0** 확인.
- 인간 검수 후 라벨로 학습, test셋 PR-AUC·정밀도·재현율이 목표치(1차 실측 후 확정) 충족.
- **Java 스코어러 == Python EBM 예측**(동일 입력, 수치 오차 허용) — EBM 테이블/PMML 왕복 일치.
- 규칙 FDS baseline과 **동일 데이터 성능 비교표** 산출(대체 정당화).
- 개인화: override 후 같은 유형 실시간 결제가 사용자 기준으로 분류됨. 파기 시 override 삭제.
- 원칙 1: 임의 낭비 판정에 "왜"(상위 기여 특징) 제시 가능.

8. **KWCAG 잔여 4탭 전수 점검(W6-4)** → 치명 위반(명도대비·키보드 함정)만 수정.

### 부록 E. 생성→ML 구현 순서 (Phase 7·8 상세 절차)

> 사용자 확정 원칙: **"페르소나 마지막" = "인터넷 검색 등 먼저 할 수 있는 단순 작업을 끝낸 뒤"** 이며, ML 학습에 지장을 주면서까지 강제로 맨 끝에 두라는 뜻이 아니다. 따라서 인터넷 데이터·기반을 먼저 확보하고, 페르소나 확정 → 실제 데이터 생성 → **그 실제 데이터로 ML 제작·학습**(방법론상 깨끗). **더미 데이터로 ML을 먼저 만드는 우회는 채택하지 않는다**(드라이런은 더미가 아니라 실 페르소나·소규모 1/100).

**A. 인터넷 데이터·기반 (먼저 — 페르소나 불필요)**
- A1 공개 상권/POI(주소·좌표·업종) 확보·정제 → 적재.
- A2 카테고리·업종 매핑 + 일반 상품/가격 참조(인터넷) → 상품 카탈로그 씨앗.
- A3 MySQL·Flyway + 스키마 확장 마이그레이션(품목·가격·채널·위치·라벨 — 부록 A Phase 7의 V2~V5).
- A4 datafaker·결정론 시드·JDBC 배치 기반.
- A5 상품 카탈로그 + 웹/LLM 재량성 점수 enrichment(카탈로그 1회, 페르소나 무관).
- (병행) 생성기·라벨러·ML 스크립트 **골격 코드**를 미리 작성 가능 — 단 검증·학습은 실제 데이터로(C·D).

**B. 페르소나 확정 (프렙 후)** — 사용자 페르소나 정의 확정 → 파라미터·하루활동·활동지역 입력(W1-0·W1-1).

**C. 실제 데이터 생성 (= Phase 7 본체)**
- C1 **1/100 드라이런(실 페르소나·소규모)** → 스키마·분포·재현성·데이터분리 교집합 0 검증(W1-7).
- C2 11M 본 생성(replace) → 검증.
- C3 학습셋 **인간 라벨 검수·수정**(CSV/간이 도구, W8-3).

**D. ML (= Phase 8, 실제 데이터로)**
- D1 특징 엔지니어링(누수 금지) → **EBM 학습 + GBM 비교 + 평가**(PR-AUC·정밀도·캘리브레이션, W8-1~3).
- D2 EBM 테이블 내보내기 + Java 스코어러(**Java==Python 일치**, W8-4).
- D3 판정 경로 교체(AlertService→ML, 규칙 FDS baseline 보존) + 개인화 override 층(W8-5).
- D4 재캘리브레이션(baseline)·최종 검증(W8-6)·문서 정합(마스터 §4·§12·§13-11).
